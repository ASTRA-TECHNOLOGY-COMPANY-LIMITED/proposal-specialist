# 제안서 섹션 작성

$ARGUMENTS 에 지정된 목차 파일을 기반으로 제안서 섹션을 작성하라.

## 인자 파싱 및 목차 파일 확보

$ARGUMENTS 형식: `<목차파일경로> [섹션번호...]`

- `목차파일경로`만 있으면 전체 섹션 작성 여부를 사용자에게 확인한다
- `목차파일경로 1 3 5` 처럼 번호를 지정하면 해당 섹션만 작성한다

다음 순서로 목차 파일을 확보한다:

1. **$ARGUMENTS에서 확인**: 첫 번째 인자가 있으면 목차 파일 경로로 사용한다
2. **자동 탐색**: 인자가 없으면 `data/output/` 디렉토리에서 Glob Tool로 `**/목차.md` 파일을 찾는다
   - 1개 발견되면 자동 사용한다
   - 여러 개 발견되면 목록을 보여주고 사용자에게 선택을 요청한다
3. **사용자에게 요청**: 목차 파일을 찾지 못하면 대화형으로 요청한다:
   "목차 파일을 찾지 못했습니다. 목차 파일 경로를 입력해주세요. (예: data/output/사업명/목차.md)
   목차가 아직 없으면 먼저 생성해주세요: `/proposal-specialist:generate-toc <RFP경로> <시드경로>`"
   - 사용자가 경로를 입력하면 해당 파일로 계속 진행한다

## 실행 절차

### Step 1: 목차 파일 및 공통 설정 읽기

Read Tool로 목차 파일을 읽는다. 목차 파일의 YAML 프론트매터에서 추출할 항목:

- `rfp_path` — RFP 파일 경로
- `seed_path` — 시드 파일 경로
- `사업명` — 사업명
- `사업명_경로` — 파일시스템 안전 경로명
- `발주처` — 발주 기관명
- `평가방식` — 평가 비율
- `사업예산`, `사업기간`

**rfp_path 확보** (진행 중 누락 시 대화형 요청):
- 프론트매터에 rfp_path가 있으면 해당 경로를 사용한다
- 없거나 해당 경로의 파일이 존재하지 않으면 `data/business/` 디렉토리에서 Glob Tool로 자동 탐색한다
- 자동 탐색에서도 찾지 못하면 사용자에게 요청한다:
  "RFP(제안요청서) 파일 경로를 입력해주세요. 섹션 작성에 필요한 요구사항 상세 내용을 참조합니다."
  - 사용자가 경로를 입력하면 해당 파일로 계속 진행한다

**seed_path 확보** (진행 중 누락 시 대화형 요청):
- 프론트매터에 seed_path가 있으면 해당 경로를 사용한다
- 없거나 해당 경로의 파일이 존재하지 않으면 `data/seed/` 디렉토리에서 Glob Tool로 자동 탐색한다
- 자동 탐색에서도 찾지 못하면 사용자에게 요청한다:
  "기업 시드 문서 경로를 입력해주세요. 섹션별 기업 강점 인용에 사용됩니다."
  - 사용자가 경로를 입력하면 해당 파일로 계속 진행한다

목차 본문에서 추출할 항목:
- 섹션 목록 (번호, 장 제목, 절 제목)
- 각 섹션의 메타데이터 (배점, 핵심메시지, 강조강점, 요구사항 번호, 도표, 도안, 페이지예산)
- 세부항목 목록

출력 디렉토리: `data/output/{사업명_경로}/sections/`

`_common/` 디렉토리 확인:
- `data/output/{사업명_경로}/_common/common-config.json` 파일 존재 여부를 Glob Tool로 확인한다
- **존재하면**: Read Tool로 읽어 장별 startPage와 총페이지 정보를 가져온다
- **없으면**: 사용자에게 안내한다:
  "공통 리소스가 아직 생성되지 않았습니다. 먼저 실행해주세요:
  `/proposal-specialist:generate-common {목차파일경로}`"
  그리고 작업을 중단한다.

### Step 2: 원본 문서 읽기

RFP와 시드 문서를 Read Tool로 다시 읽어 전체 컨텍스트를 확보한다.

**RFP 읽기:**
- 사업개요, 시스템 현황 섹션 (모든 섹션에서 참조할 기본 정보)
- 요구사항 목록 전체 (섹션별 관련 요구사항을 찾기 위해)

**시드 읽기:**
- 전체 시드 문서 (각 섹션에서 관련 강점을 인용하기 위해)

### Step 3: 작성 범위 확인

섹션 목록을 보여주고 사용자에게 확인한다:

```
## 작성 대상 섹션

1. **I. 일반현황 > 1.1 제안사 일반현황** — 도표 4개, 도안 1개 | 4p
   - 핵심메시지: {메시지}
2. **I. 일반현황 > 1.2 조직 및 인원현황** — 도표 2개, 도안 1개 | 3p
   - 핵심메시지: {메시지}
...

총 {N}개 섹션, 예상 {P}페이지 (A4 세로 꽉 채움), 도표 {T}개 + 도안 {D}개 예정입니다.
전체 작성하시겠습니까? (Y/N, 또는 섹션 번호를 쉼표로 입력)
```

### Step 4: 섹션별 작성 (반복)

각 섹션에 대해 다음을 순차적으로 수행한다:

#### 4-1. 관련 RFP 요구사항 조회

목차의 `요구사항` 필드에 나열된 요구사항 번호를 기준으로,
해당 요구사항의 상세 내용을 RFP에서 Read Tool로 조회한다.

섹션-요구사항 매핑 가이드:
- 기술방안 관련 섹션 → SFR (기능 요구사항) 상세 내용
- 성능/품질 섹션 → PER (성능), QUR (품질) 요구사항
- 인터페이스 섹션 → INR (인터페이스) 요구사항
- 보안 섹션 → SER (보안) 요구사항
- 데이터 섹션 → DAR (데이터) 요구사항
- 수행체계 섹션 → PMR (프로젝트관리) 요구사항
- 교육/하자보수 섹션 → PSR (프로젝트지원) 요구사항

#### 4-2. section-writer 에이전트 호출

이 섹션의 startPage를 계산한다:

```
이 섹션의 startPage = common-config.json의 해당 장 startPage + 이 장 내 이전 섹션들의 페이지예산 합
```

section-writer 에이전트에게 다음 컨텍스트를 전달한다:

```
**섹션 정보:**
- 섹션 번호: {N}
- 장 번호: {장번호 (I, II, III...)}
- 장 제목: {장제목}
- 절 제목: {절제목}
- 세부항목: {세부항목 목록}
- 배점: {배점}
- 핵심메시지: {핵심메시지}
- 강조강점: {강조강점}
- 페이지예산: {N페이지} (A4 세로 기준, 빈 여백 없이 꽉 채움)
- 이 장의 첫 번째 섹션 여부: {Y/N} (Y이면 간지 페이지 추가)

**관련 RFP 요구사항:**
{해당 요구사항 상세 내용}

**기업 시드 데이터 (관련 부분):**
{해당 섹션에 필요한 시드 데이터 발췌}

**도표 요청 (인라인 HTML):**
- {도표1: 유형 + 설명}
- {도표2: 유형 + 설명}
모든 도표는 섹션 HTML 내에 인라인으로 작성한다.

**도안 요청 (인라인 HTML/CSS):**
- {도안1: 유형 + 설명}
- {도안2: 유형 + 설명}
모든 도안은 HTML/CSS/JS로 섹션 HTML 내에 인라인으로 구현한다.
image_text2img 등 이미지 생성 도구는 사용하지 않는다.

**PAGE_CONFIG:**
- badge: "{장번호}"
- title: "{장제목}"
- project: "{사업명}"
- section: "{절제목}"
- startPage: {계산된 시작 페이지}
- totalPages: {common-config.json의 총페이지}

**공유 스타일 경로 (HTML에서 참조):**
- CSS: `<link rel="stylesheet" href="../_common/page-frame.css">`
- JS: `<script src="../_common/page-frame.js"></script>`

**레이아웃 규칙:**
- 마크다운 테이블 금지 — 모든 표는 HTML <table>로 인라인
- 이미지 파일 참조 금지 — 모든 도안은 HTML/CSS/JS로 인라인 구현
- 별도 HTML 파일 분리 금지 — 도표와 도안 모두 섹션 HTML 내에 직접 작성
- 매 페이지 최소 1개 도표/도안 포함
- page-frame.js가 오버플로 시 자동으로 페이지 분할

**사업 기본 정보:**
- 사업명: {사업명}
- 발주처: {발주처}
- 사업예산: {예산}
- 사업기간: {기간}
```

#### 4-3. 파일 저장

각 섹션 완료 후 Write Tool로 즉시 저장한다:

`data/output/{사업명_경로}/sections/{번호:02d}_{절제목}.html`

파일명에서 공백은 `_`로 변환하고 특수문자는 제거한다.

**저장 시 확인사항:**
- `<!DOCTYPE html>` 선언이 있는지 확인
- `PAGE_CONFIG` 변수가 올바른 값으로 설정되어 있는지 확인
- `../_common/page-frame.css` 링크가 있는지 확인
- `../_common/page-frame.js` 스크립트가 있는지 확인
- 마크다운 테이블 문법(`| | |`)이 본문에 없는지 확인
- `<img>` 태그로 AI 생성 이미지를 참조하지 않는지 확인 (도안은 HTML/CSS로 인라인 작성)

### Step 5: 진행 상황 보고

각 섹션 완료 시 즉시 보고한다:

```
[{완료}/{전체}] {장}.{절} 작성 완료 ({분량}페이지, 도표 {M}개 + 도안 {K}개) → sections/{파일명}
```

### Step 6: 최종 요약

모든 섹션 완료 후 종합 보고한다:

```
## 제안서 작성 완료

### 작성 결과

1. **I-1. 제안사 일반현황** → `sections/01_제안사일반현황.html` — 4p | 도표 3개, 도안 1개
2. **I-2. 조직 및 인원현황** → `sections/02_조직및인원현황.html` — 3p | 도표 2개, 도안 1개
...

- **총 섹션 수**: {N}개
- **총 분량**: 약 {P}페이지 (A4 세로 기준, 페이지 꽉 채움)
- **인라인 도표**: {M}개 (표, 차트, 비교표)
- **인라인 도안**: {K}개 (아키텍처도, 개념도, 프로세스 플로우)
- **저장 위치**: `data/output/{사업명_경로}/`

### 파일 구조

    data/output/{사업명_경로}/
    ├── 목차.md
    ├── sections/
    │   ├── 01_제안사일반현황.html    ← 완전한 HTML (도표+도안 인라인 포함)
    │   ├── 02_조직및인원현황.html
    │   └── ...
    └── _common/
        ├── page-frame.css    ← 공유 스타일
        ├── page-frame.js     ← 헤더/푸터/페이지번호/자동분할
        └── common-config.json ← 장별 시작 페이지 설정

### 검토 가이드

1. 각 HTML 파일을 브라우저(Chrome)에서 열어 헤더/푸터/페이지번호가 정상 표시되는지 확인하세요
2. 도표와 도안이 올바르게 렌더링되는지 확인하세요 (HTML 소스를 직접 수정 가능)
3. `<!-- 작성 메모 -->` HTML 주석을 확인하여 보완 사항을 검토하세요
4. `{확인 필요}` 표시된 항목은 실제 데이터로 교체하세요
5. 각 섹션이 A4 페이지를 꽉 채우고 있는지 확인하세요 (여백 최소화)
6. 최종 제안서 편집 시 작성 메모 주석을 삭제하세요

### 재작성

특정 섹션을 재작성하려면:
`/proposal-specialist:generate-section data/output/{사업명_경로}/목차.md {섹션번호}`
```
