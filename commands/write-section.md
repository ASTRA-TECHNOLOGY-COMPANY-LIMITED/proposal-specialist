# 제안서 섹션 작성

$ARGUMENTS 에 지정된 목차 파일을 기반으로 제안서 섹션을 작성하라.

## 인자 파싱

$ARGUMENTS 형식: `<목차파일경로> [섹션번호...]`

- `목차파일경로`만 있으면 전체 섹션 작성 여부를 사용자에게 확인한다
- `목차파일경로 1 3 5` 처럼 번호를 지정하면 해당 섹션만 작성한다
- 인자가 없으면 `data/output/` 디렉토리에서 Glob Tool로 `목차.md` 파일을 찾는다

## 실행 절차

### Step 1: 목차 파일 읽기

Read Tool로 목차 파일을 읽는다. 목차 파일의 YAML 프론트매터에서 추출할 항목:

- `rfp_path` — RFP 파일 경로
- `seed_path` — 시드 파일 경로
- `사업명` — 사업명
- `사업명_경로` — 파일시스템 안전 경로명
- `평가방식` — 평가 비율
- `사업예산`, `사업기간`

rfp_path나 seed_path가 프론트매터에 없으면 사용자에게 직접 요청한다:
"RFP 파일 경로를 입력해주세요."

목차 본문에서 추출할 항목:
- 섹션 목록 (번호, 장 제목, 절 제목)
- 각 섹션의 메타데이터 (배점, 핵심메시지, 강조강점, 요구사항 번호, 이미지, 페이지예산)
- 세부항목 목록

출력 디렉토리: `data/output/{사업명_경로}/sections/`
이미지 디렉토리: `data/output/{사업명_경로}/images/`
HTML 디렉토리: `data/output/{사업명_경로}/html/`

### Step 2: 원본 문서 읽기

RFP와 시드 문서를 Read Tool로 다시 읽어 전체 컨텍스트를 확보한다.

**RFP 읽기:**
- 사업개요, 시스템 현황 섹션 (모든 섹션에서 참조할 기본 정보)
- 요구사항 목록 전체 (섹션별 관련 요구사항을 찾기 위해)

**시드 읽기:**
- 전체 시드 문서 (각 섹션에서 관련 강점을 인용하기 위해)

### Step 3: 작성 범위 확인

섹션 목록을 보여주고 사용자에게 확인한다:

```
## 작성 대상 섹션

| # | 장 | 절 | 핵심 메시지 | 도표(HTML) | 이미지(AI) | 분량 |
|---|---|---|---|---|---|---|
| 1 | I. 일반현황 | 1.1 제안사 일반현황 | {메시지} | 4개 | 1개 | 4p |
| 2 | I. 일반현황 | 1.2 조직 및 인원현황 | {메시지} | 2개 | 1개 | 3p |
...

총 {N}개 섹션, 예상 {P}페이지 (A4 세로 꽉 채움), 도표 {T}개 + 이미지 {I}개 생성 예정입니다.
전체 작성하시겠습니까? (Y/N, 또는 섹션 번호를 쉼표로 입력)
```

### Step 4: 섹션별 작성 (반복)

각 섹션에 대해 다음을 순차적으로 수행한다:

#### 4-1. 관련 RFP 요구사항 조회

목차의 `요구사항` 필드에 나열된 요구사항 번호를 기준으로,
해당 요구사항의 상세 내용을 RFP에서 Read Tool로 조회한다.

섹션-요구사항 매핑 가이드:
- 기술방안 관련 섹션 → SFR (기능 요구사항) 상세 내용
- 성능/품질 섹션 → PER (성능), QUR (품질) 요구사항
- 인터페이스 섹션 → INR (인터페이스) 요구사항
- 보안 섹션 → SER (보안) 요구사항
- 데이터 섹션 → DAR (데이터) 요구사항
- 수행체계 섹션 → PMR (프로젝트관리) 요구사항
- 교육/하자보수 섹션 → PSR (프로젝트지원) 요구사항

#### 4-2. section-writer 에이전트 호출

section-writer 에이전트에게 다음 컨텍스트를 전달한다:

```
**섹션 정보:**
- 섹션 번호: {N}
- 장 제목: {장제목}
- 절 제목: {절제목}
- 세부항목: {세부항목 목록}
- 배점: {배점}
- 핵심메시지: {핵심메시지}
- 강조강점: {강조강점}
- 페이지예산: {N페이지} (A4 세로 기준, 빈 여백 없이 꽉 채움)

**관련 RFP 요구사항:**
{해당 요구사항 상세 내용}

**기업 시드 데이터 (관련 부분):**
{해당 섹션에 필요한 시드 데이터 발췌}

**도표 요청 (HTML → Chrome MCP → 이미지):**
- {도표1: 유형 + 설명}
- {도표2: 유형 + 설명}
HTML 저장 경로: data/output/{사업명_경로}/html/
이미지 저장 경로: data/output/{사업명_경로}/images/

**이미지 요청 (AI 이미지 생성):**
- {이미지1 설명}
- {이미지2 설명}
이미지 저장 경로: data/output/{사업명_경로}/images/

**레이아웃 규칙:**
- 마크다운 테이블 금지 — 모든 표는 HTML 도표 이미지로 대체
- 각 절은 새 페이지에서 시작 (page-break-before: always)
- 매 페이지 최소 1개 도표/이미지 포함
- 페이지 하단 여백 최소화

**사업 기본 정보:**
- 사업명: {사업명}
- 발주처: {발주처}
- 사업예산: {예산}
- 사업기간: {기간}
```

#### 4-3. HTML 도표 제작 및 이미지 변환

section-writer가 작성한 내용에 포함된 모든 표, 차트, 프로세스 플로우 등을 HTML로 제작하고
Chrome MCP를 통해 이미지로 변환한다.

**HTML → 이미지 변환 절차:**

1. section-writer가 도표 HTML 코드를 제공하면, Write Tool로 HTML 파일을 저장한다
   경로: `data/output/{사업명_경로}/html/{섹션번호:02d}_{도표명}.html`

2. Chrome MCP 도구를 사용하여 HTML을 렌더링하고 스크린샷을 캡처한다:
   ```
   a. navigate_page: file:///절대경로/{사업명_경로}/html/{파일명}.html
   b. take_screenshot: data/output/{사업명_경로}/images/{섹션번호:02d}_{도표명}.png
   ```

3. 마크다운 섹션 파일에서 상대 경로로 이미지를 참조한다:
   `![표 N. 설명](../images/{섹션번호:02d}_{도표명}.png)`

**이미지 생성 도구를 사용하는 경우:**

개념도, 아키텍처도, 인포그래픽 등 비정형 시각자료는 image_text2img 도구로 생성한다:
1. 섹션 내용에 맞는 구체적 프롬프트를 작성한다
2. 저장 경로: `data/output/{사업명_경로}/images/{섹션번호:02d}_{이미지명}.png`
3. 마크다운에서 상대 경로로 참조: `![그림 N. 설명](../images/{파일명})`

#### 4-4. 파일 저장

각 섹션 완료 후 Write Tool로 즉시 저장한다:

`data/output/{사업명_경로}/sections/{번호:02d}_{절제목}.md`

파일명에서 공백은 `_`로 변환하고 특수문자는 제거한다.

**저장 시 확인사항:**
- 마크다운 테이블 문법(`| | |`)이 본문에 없는지 확인 — 모든 표는 이미지로 대체
- 각 절 끝에 `<div style="page-break-before: always;"></div>`가 있는지 확인
- 이미지 경로가 올바른 상대 경로(`../images/`)인지 확인

### Step 5: 진행 상황 보고

각 섹션 완료 시 즉시 보고한다:

```
[{완료}/{전체}] {장}.{절} 작성 완료 ({분량}페이지, 도표 {M}개 + 개념도 {K}개) → sections/{파일명}
```

### Step 6: 최종 요약

모든 섹션 완료 후 종합 보고한다:

```
## 제안서 작성 완료

### 작성 결과

| # | 섹션 | 파일 | 분량 | 도표(HTML) | 개념도(AI) |
|---|---|---|---|---|---|
| 1 | I-1. 제안사 일반현황 | sections/01_제안사일반현황.md | 4p | 3개 | 1개 |
| 2 | I-2. 조직 및 인원현황 | sections/02_조직및인원현황.md | 3p | 2개 | 1개 |
...

- **총 섹션 수**: {N}개
- **총 분량**: 약 {P}페이지 (A4 세로 기준, 페이지 꽉 채움)
- **HTML 도표**: {M}개 (표, 차트, 프로세스 플로우)
- **AI 생성 개념도**: {K}개 (아키텍처도, 인포그래픽)
- **저장 위치**: `data/output/{사업명_경로}/`

### 파일 구조

    data/output/{사업명_경로}/
    ├── 목차.md
    ├── sections/
    │   ├── 01_제안사일반현황.md
    │   ├── 02_조직및인원현황.md
    │   └── ...
    ├── html/              ← 도표 원본 HTML (수정/재생성 가능)
    │   ├── 01_기업개요표.html
    │   ├── 01_재무현황차트.html
    │   └── ...
    └── images/            ← HTML 스크린샷 + AI 생성 이미지
        ├── 01_기업개요표.png
        ├── 01_재무현황차트.png
        ├── 03_시스템아키텍처.png
        └── ...

### 검토 가이드

1. 각 섹션의 `> 작성 메모` 블록을 확인하여 보완 사항을 검토하세요
2. `{확인 필요}` 표시된 항목은 실제 데이터로 교체하세요
3. 도표 이미지가 올바르게 렌더링되었는지 확인하세요 (HTML 원본은 `html/` 폴더에서 수정 가능)
4. 이미지가 내용과 맞는지 확인하고, 필요시 재생성을 요청하세요
5. 각 섹션이 A4 페이지를 꽉 채우고 있는지 확인하세요 (여백 최소화)
6. 최종 제안서 편집 시 작성 메모 블록과 page-break div를 제거하세요

### 재작성

특정 섹션을 재작성하려면:
`/proposal-specialist:write-section data/output/{사업명_경로}/목차.md {섹션번호}`
```
