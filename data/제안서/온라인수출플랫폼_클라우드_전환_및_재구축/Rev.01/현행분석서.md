## 3. 현행 인프라 구성

### 3.1 핵심 확인사항: 현재 이미 민간클라우드 사용 중

> **중요**: 현행 시스템은 이미 민간클라우드 환경에서 운영 중입니다. 따라서 본 사업의 "클라우드 전환"은 최초 클라우드 마이그레이션이 아니라, **클라우드 네이티브 아키텍처로의 전환 및 재구축**을 의미합니다.

### 3.2 하드웨어(H/W) 현황

| 모델 | 구분 | 용도 | 수량 | 비고 |
|------|------|------|------|------|
| 민간클라우드 | WEB서버 | 운영, 개발 | 2식 | |
| 민간클라우드 | WAS서버 | 운영, 개발, 고비즈행정, 기타(API) | 7식 | |
| 민간클라우드 | DB서버 | 운영, 개발 | 2식 | |
| 민간클라우드 | NAS서버 | 운영, 개발 | 2식 | |

- 세부 수량은 정보시스템 최적화에 따라 변경(증가)될 수 있음
- **총 서버 수**: 13식 (WEB 2 + WAS 7 + DB 2 + NAS 2)

### 3.3 소프트웨어(S/W) 현황

| 구분 | 현행 S/W | 비고 |
|------|---------|------|
| 운영체제 | Linux | |
| 데이터베이스 | Maria(DB) | 발주사 협의 가능 |
| WEB/WAS | Nginx/Tomcat | 발주사 협의 가능 |
| 프레임워크 | 전자정부 표준프레임워크 5.0 | |
| JDK 버전 | Open JDK 17, 21, 23 | 발주사 협의 가능 |
| 프론트환경 | Nexacro N V24 | 행정(내부, 관리자)만 적용 |
| 개발언어 | JAVA, JS 등 | |

- SW는 제시된 목록 중 발주사와 협의하여 진행하며, 특별한 사유가 없는 경우 구축 시 최신버전 적용

### 3.4 주요 서버 구성 (목표 시스템 기준)

RFP에서 제시한 목표 시스템의 서버 구성표:

| 구분 | 민간클라우드 - 국문 | 민간클라우드 - 영문 | 민간클라우드 - 사업관리 및 관리자 | 중진공IDC - 통합정보시스템 |
|------|-------------|-------------|---------------------|-----------------|
| WEB | Nginx 2식 (국문/영문 공용) | ← | Nginx 1식 | Nginx 1식 |
| Front-end | - | - | **Nexacro 1식** | **Nexacro 1식** |
| WAS | Tomcat 2식 | Tomcat 2식 | Tomcat 1식 | Tomcat 1식 |
| DB | MariaDB 2식 HA 구성 (3개 사이트 공용) | ← | ← | Oracle (기존 활용) |

> **핵심 포인트**: **사업관리 및 관리자사이트**는 중진공 IDC가 아닌 **민간클라우드**에 배치된다. 다만, 중진공 업무망에서 클라우드 내부망으로 VPN 연결하여 접속하는 **하이브리드 클라우드** 구조이다. Nexacro가 적용되는 시스템은 2개(사업관리/관리자 + 통합정보시스템)이며, 각각 민간클라우드와 중진공IDC에 분리 배치된다.

- API서버, 기타서버는 발주사와 협의
- 시스템 구성도(HW/SW)는 **보안상 생략** (요청 시 열람 가능)

### 4.1 목표 시스템 아키텍처 (RFP 기반 분석)

RFP에서 시스템 구성도는 "보안상 생략"되어 있으나, 주요 서버 구성표와 사업범위 설명을 기반으로 목표 아키텍처를 분석합니다.

│  │  WEB: Nginx 2식 (국문/영문 공용) + Nginx 1식 (관리자)│        │
│  └───────┬───────────────────────────────┬──────────┘        │
│          │                               │                   │
│  ┌───────┴───────────────────┐  ┌────────┴────────────┐      │
│  │  국문사이트    영문사이트    │  │ 사업관리 및 관리자    │      │
│  │  Tomcat 2식   Tomcat 2식  │  │ Nexacro N V24 1식   │      │
│  │  (React/Vue)  (Next.js)   │  │ Tomcat 1식          │      │
│  └───────┬───────────────────┘  └────────┬────────────┘      │
│          │                               │                   │
│  ┌───────┴───────────────────────────────┴────────────┐      │
│  │  DB: MariaDB 2식 HA 구성 (3개 사이트 공용)           │      │
│  │  NAS: 운영/개발 2식                                  │      │

### 4.2 WAS 7식 구성 추정

WAS가 7식으로 상대적으로 많은 이유를 분석합니다:

| WAS 용도 | 추정 수량 | 근거 |
|---------|---------|------|
| 국문사이트 운영 | 1~2식 | 사용자 접속 서비스 |
| 영문사이트 운영 | 1~2식 | 해외바이어 접속 서비스 |
| 관리자/고비즈행정 | 1식 | 내부 관리 서비스 |
| API 서버 | 1식 | 외부 연계/데이터 서비스 |
| 개발 서버 | 1~2식 | 개발/테스트 환경 |

### 4.3 네트워크 구성 분석

| 구간 | 구성 | 비고 |
|------|------|------|
| 외부 → 국문/영문사이트 | 민간클라우드 DMZ 경유 | 공개 접근 (인터넷) |
| **중진공 업무망 → 사업관리/관리자사이트** | **중진공 업무망 → VPN → 민간클라우드 내부망** | **하이브리드 클라우드**: 관리자사이트는 민간클라우드에 배치, 업무망에서 VPN 접속 |
| 민간클라우드 → 중진공 IDC 통합정보시스템 | IPsec VPN | 통합정보시스템(Oracle) 연계, 민감정보 처리 |

### 4.4 DB 구성 현황

| DB | 위치 | 용도 | 비고 |
|----|------|------|------|
| MariaDB | 민간클라우드 | 고비즈코리아 국문/영문/관리자 사이트 | 현행 운영 중 |
| Oracle | 중진공 IDC | 통합정보시스템(수출) - 수출사업 접수/평가/정산, 민감정보 처리 | 기존 활용 (TO-BE에서도 유지) |

> **핵심 포인트**: 이미 **하이브리드 DB 환경**이 구성되어 있음 (MariaDB + Oracle). TO-BE에서는 MariaDB를 HA 구성으로 강화하고, Oracle은 기존 그대로 활용하는 방향.

### 4.5 TO-BE MariaDB HA 구성 상세

RFP에서 "MariaDB 2식 HA 구성"을 요구하고 있으며, 국문/영문/관리자 3개 사이트가 공용으로 사용합니다.

#### (1) HA 구성 방식 비교

| 방식 | 복제 유형 | 쓰기 노드 | 자동 Failover | 데이터 손실 | 적합성 |
|------|---------|---------|-------------|---------|------|
| **Master-Slave Replication** | 비동기 | Master만 | 수동 (또는 MHA/Orchestrator) | 비동기이므로 일부 유실 가능 | 단순 읽기분산 |
| **Galera Cluster** (권장) | 동기 (wsrep) | **모든 노드** | **자동** | **없음** | 공공시스템 HA에 최적 |
| **MaxScale + Replication** | 비동기 + 프록시 | Master만 | 자동 (MaxScale) | 일부 유실 가능 | 라이선스(BSL) 확인 필요 |

#### (2) 권장 구성: Galera Cluster 2노드 + Garbd

```
┌─────────────────────────────────────────────────────────────┐
│                      민간클라우드                              │
│                                                             │
│   ┌────────────┐     동기 복제 (wsrep)    ┌────────────┐    │
│   │   Node 1   │ ◀─────────────────────▶ │   Node 2   │    │
│   │  MariaDB   │                         │  MariaDB   │    │
│   │  Galera    │                         │  Galera    │    │
│   │ (읽기/쓰기) │                         │ (읽기/쓰기) │    │
│   └──────┬─────┘                         └──────┬─────┘    │
│          │        ┌────────────────┐            │          │
│          └────────┤    Garbd       ├────────────┘          │
│                   │ (투표 전용 데몬) │                        │
│                   │ Split-brain방지 │                        │
│                   └───────┬────────┘                        │
│                           │                                │
│                ┌──────────┴──────────┐                      │
│                │  HAProxy/ProxySQL   │                      │
│                │ (로드밸런싱+헬스체크)  │                      │
│                └──────────┬──────────┘                      │
│                           │                                │
│          ┌────────────────┼────────────────┐               │
│          ▼                ▼                ▼               │
│   Tomcat 2식(국문)  Tomcat 2식(영문)  Tomcat 1식(관리자)      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### (3) Garbd(Galera Arbitrator)가 필요한 이유

Galera Cluster는 과반수 투표(Quorum)로 정상 노드를 판별합니다. 2노드만 있으면 1대 장애 시 Split-brain 문제가 발생합니다.

| 구성 | 1대 장애 시 Quorum | 서비스 지속 |
|------|----------------|---------|
| 2노드 (Garbd 없음) | 1/2 = 과반 미달 | **서비스 중단** |
| 2노드 + Garbd | 2/3 = 과반 확보 | **서비스 지속** |

Garbd는 데이터를 저장하지 않는 경량 데몬으로, 투표에만 참여하여 리소스 부담이 거의 없습니다.

#### (4) 장애 시나리오별 동작

| 시나리오 | 동작 | 소요 시간 | 데이터 손실 |
|---------|------|---------|---------|
| **정상 운영** | 쓰기→한 노드 라우팅, 읽기→두 노드 분산 | - | - |
| **Node 1 장애** | Garbd+Node2 Quorum 확보, HAProxy 자동 라우팅 전환 | 1~5초 | 없음 (동기복제) |
| **Node 1 복구** | IST/SST로 자동 데이터 동기화 후 재합류 | 수분~수십분 | 없음 |

#### (5) DGuard 호환성 확인 필요

중진공이 DB 암호화에 DGuard 솔루션을 사용 중입니다. **Galera Cluster 환경에서 DGuard가 정상 동작하는지 반드시 확인이 필요합니다.** DGuard가 커널/파일시스템 레벨 암호화인 경우 호환 가능하나, DB 엔진 레벨 암호화인 경우 wsrep 복제와 충돌 가능성이 있습니다.

#### (6) 대안: 클라우드 Managed DB 활용 시

민간클라우드 제공사의 Managed DB 서비스를 활용할 경우:

| 클라우드 사업자 | Managed DB 서비스 | HA 방식 | 비고 |
|---------|---------|---------|------|
| NHN Cloud | NHN Cloud RDS for MariaDB | Master-Slave 자동 Failover | 관리 부담 최소 |
| KT Cloud | KT Cloud DB (MariaDB) | HA 옵션 제공 | 벤더 Lock-in 고려 |
| AWS | Amazon RDS for MariaDB | Multi-AZ 자동 Failover | 공공클라우드 인증 필요 |

> Managed DB 사용 시 Galera 직접 구성이 불필요하지만, **DGuard 같은 외부 보안 솔루션 설치가 제한**될 수 있어 발주처 확인이 필요합니다.

## 9. 현행시스템 한계점 분석

### 9.1 영역별 한계점

| 영역 | 한계점 | 영향 | 대응 방향 |
|------|--------|------|---------|
| **인프라** | HW/SW 노후화로 성능 저하 및 장애 빈도 증가 | 서비스 안정성 저하 | 클라우드 네이티브 전환(SFR-001) |
| **아키텍처** | 모놀리식 구조로 개별 서비스 독립 배포/확장 불가 | 기능 추가/변경 시 전체 시스템 영향 | 컨테이너 기반 배포(SFR-002) |
| **확장성** | 수동 확장, 부하 분산 한계 | 트래픽 폭증 시 서비스 장애 | 오토스케일링 정책(PER-001~004) |
| **UI/UX** | 구형 인터페이스, 모바일 대응 미흡 | 사용자 이탈 및 만족도 저하 | UI/UX 재설계(INR-001~009) |
| **데이터** | 수출 데이터 비표준화, 사일로 현상, 통계 수기관리 | 통계 정확도 저하 및 의사결정 지연 | 데이터 표준화(DAR-001~009) |
| **연계** | 타기관 연계 제한적, API 기반 통합 미흡 | 맞춤형 수출정보 제공 한계 | API 연계 강화(SFR-011, INR-012) |
| **배포** | 수동 배포, 장애 시 전체 서비스 영향 | 배포 지연, 서비스 중단 | CI/CD, Blue/Green 배포(SFR-002) |
| **글로벌 접근성** | 해외 접속 시 지연 | 해외바이어 사용성 저하 | CDN 서비스 적용(SFR-005) |
| **결제** | 결제 연동 미구축 | 직접 거래 불가 | 토스페이먼츠 연동(SFR-005) |
| **마이데이터** | 공공마이데이터 미연계 | 사업자 정보 수동 입력 | PDS 기반 마이데이터 연계(SFR-008) |

### 9.2 주요 Gap 분석 (AS-IS → TO-BE)

| 분류 | AS-IS | TO-BE | Gap |
|------|-------|-------|-----|
| **클라우드** | 민간클라우드 (단순 호스팅) | 클라우드 네이티브 (K8s/Docker, 오토스케일링) | 아키텍처 전면 재설계 |
| **배포** | 수동 배포 | CI/CD 파이프라인, Blue/Green/Rolling/Canary 배포 | 배포 자동화 체계 구축 |
| **DB** | MariaDB (단일 추정) + Oracle (IDC) | MariaDB 2식 HA (Galera Cluster 권장) + Oracle (기존) | Galera 2노드 + Garbd 구성, DGuard 호환성 확인, HAProxy 로드밸런싱 추가 |
| **영문사이트** | 기본 상품 조회 | SNS 로그인, 토스페이먼츠 결제, 카페24/아마존 연동, CDN | 대폭 기능 확장 |
| **관리자** | 레거시 UI | Nexacro N V24 적용 전면 재개발 | UI 프레임워크 전환 |
| **통합정보시스템** | 기존 Oracle 기반 | 재개발 + 고비즈코리아 연계 강화 | 서비스 재개발 |
| **API** | 제한적 연계 | API Gateway + 6개 외부기관 연계 | API 통합 관리 체계 |
| **데이터** | 비표준화, 수기 통계 | 데이터 표준화, 통계 시스템화, 마이그레이션 | 데이터 아키텍처 수립 |
| **보안** | 기본 보안 | 다층 보안, 시큐어코딩, 보안관제 | 보안 체계 강화 |
| **메시징** | SMS 기본 | 대량메일/문자/카카오톡/RCS 통합 | 메시징 통합 |

## 10. 요구사항 총괄

### 10.1 요구사항 분류별 현황

| 요구사항 분류 | 수량 | 설명 |
|---------|------|------|
| 기능 요구사항(SFR) | 13건 | 목표시스템이 반드시 수행해야 하는 기능 |
| 성능 요구사항(PER) | 4건 | 처리속도, 처리량, 가용성 등 성능 |
| 인터페이스 요구사항(INR) | 12건 | 시스템/사용자 인터페이스 |
| 데이터 요구사항(DAR) | 10건*  | 데이터 변환, 표준화, 품질 (* RFP 요구사항 총괄표에는 6건으로 표기되나, 요구사항 목록에서는 DAR-001~DAR-009 + DAR-010까지 10건) |
| 테스트 요구사항(TER) | 4건*  | 테스트 유형, 환경, 방법, 절차 (* RFP 총괄표에는 9건으로 표기되나 실제 목록은 TER-001~004 4건) |
| 보안 요구사항(SER) | 7건 | 정보 기밀성/무결성 보안 |
| 품질 요구사항(QUR) | 4건 | 품질 항목, 평가 대상 및 목표 |
| 제약사항(COR) | 6건 | 기술/표준/업무/법제도 제약조건 |
| 프로젝트 관리(PMR) | 4건 | 관리방법, 추진 단계별 수행방안 |
| 프로젝트 지원(PSR) | 3건 | 교육훈련, 서비스 안정화, 하자보수 |
| **합계** | **68건** | (RFP 기준) |

> **참고**: RFP 요구사항 총괄표의 수량과 실제 요구사항 목록의 수량 간에 일부 불일치가 있음. 데이터 요구사항은 총괄표에 6건으로 표기되었으나 실제로는 DAR-001~010(10건)이며, 테스트 요구사항은 총괄표에 9건으로 표기되었으나 실제로는 TER-001~004(4건). 미팅 시 확인 필요.
