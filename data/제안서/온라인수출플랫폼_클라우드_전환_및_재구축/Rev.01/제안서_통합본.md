## I. 일반현황

### 1.1 제안사 일반현황

> **핵심 메시지**: 고비즈코리아의 베트남 버전(Gobiz Vietnam)을 직접 구축한 수출 플랫폼 전문 기업. 설립 2년차 매출 7.2억, 전년비 7,500% 성장으로 기술력과 사업수행력을 검증받은 기업

---

#### 1.1.1 기업 개요 및 주요 연혁

**(1) 기업 개요**

주식회사 아스트라비전은 AI 기반 수출 플랫폼 전문 기업으로서, 중소벤처기업진흥공단이 발주한 "베트남 온라인 수출플랫폼 모델 전수 ODA 사업"을 성공적으로 수행한 실적을 보유하고 있습니다. 본 사업의 대상 시스템인 온라인수출플랫폼(고비즈코리아)의 베트남 버전을 직접 설계하고 구축한 경험을 바탕으로, 수출 플랫폼의 기술 아키텍처와 비즈니스 로직에 대한 깊은 이해를 갖추고 있습니다.

#### 2.1.1 추진배경 및 목적 이해

본 사업은 중소벤처기업진흥공단(온라인수출처)이 운영하는 **온라인수출플랫폼(고비즈코리아, GobizKorea)** 의 전면 재구축과 클라우드 전환을 목적으로 합니다. 퀸텟시스템즈(총괄)와 아스트라비전 컨소시엄은 본 사업의 추진배경과 목적을 다음과 같이 이해합니다.

**추진배경**

디지털 무역의 급격한 성장에 따라 국내 중소기업의 온라인 수출액이 증가하고, 온라인 수출에 대한 참여가 확대되고 있습니다. 그러나 기존 고비즈코리아의 HW/SW 노후화로 인하여 온라인 수출 지원 제약이 발생하고 있으며, 민간클라우드 기반의 원게이트 플랫폼 구축이 필요한 상황입니다.

본 컨소시엄은 **베트남 온라인 수출플랫폼 모델 전수 ODA 사업**(중소벤처기업진흥공단 발주, 2024~2026년, 31.2억원 규모)을 통해 Gobiz Vietnam 시스템을 구축한 경험을 보유하고 있습니다. 이를 통해 고비즈코리아의 업무 프로세스, 데이터 구조, 사용자 패턴을 실무적으로 파악하고 있으며, 본 사업의 성공적 수행을 위한 도메인 전문성을 갖추고 있습니다.

**사업 목적**

| 구분 | 목적 | 대응 요구사항 |
|------|------|------------|
| 플랫폼 현대화 | 고비즈코리아 4개 사이트(국문/영문/관리자/통합정보시스템) 전면 재구축 | SFR-004, SFR-005, SFR-009, SFR-012 |

본 시스템은 **민간클라우드와 중진공 IDC를 IPsec VPN으로 연결하는 하이브리드 클라우드** 아키텍처로 구성합니다. 공개 SW 중심으로 시스템 SW(OS, WEB, WAS) 환경을 구성하고, 시스템 SW의 버전 업그레이드, 소프트웨어 변경에도 서비스의 안정성이 보장되도록 구현합니다(SFR-001).

**시스템 구성 총괄표**

| 계층 | 구성요소 | 민간클라우드 (신규) | 중진공 IDC (기존) | 비고 |
|------|--------|-------------|-------------|------|
| **WEB** | Nginx | 2식(국문/영문) + 1식(관리자) | 1식(통합정보시스템) | 리버스 프록시, 로드밸런싱, 정적 파일 서빙 |
| **Front-end** | Nexacro N V24 | 1식(관리자) | 1식(통합정보) | 행정/관리자 사이트 전용(RFP 지정) |
| **WAS** | Tomcat | 2식(국문) + 2식(영문) + 1식(관리자) | 1식(통합정보) | 전자정부 표준프레임워크 5.0 기반 |
| **DB** | MariaDB / Oracle | MariaDB 2식 HA 구성 | Oracle(기존 활용) | 신규 MariaDB, 기존 Oracle 병행 |
| **NAS** | 스토리지 | 2식(운영/개발) | - | 파일 스토리지 |
| **OS** | Linux | 전 서버 | 기존 활용 | - |
| **JDK** | Open JDK | 17/21/23 | - | 발주사 협의 가능 |
| **개발언어** | Java, JS 등 | - | - | - |

##### (2) 개발 및 운영 환경 분리

애플리케이션의 개발을 위한 클러스터와 운영을 위한 클러스터의 구성을 분리합니다(SFR-001).

| 환경 | 네트워크 | 용도 | 접근 정책 |
|------|---------|------|---------|
| **개발 환경** | 고비즈코리아 내부망 전용 | 소스변경 사항 배포, 기능 테스트 | 개발자 IP 제한 접근 |
| **운영 환경 (내부)** | 고비즈코리아 내부망 | 관리자/통합정보시스템 서비스 | 중진공 업무망 접근 |
| **운영 환경 (외부)** | 고비즈코리아 DMZ망 | 국문/영문사이트 외부 서비스 | 공개 접근(WAF 적용) |

##### (3) 네트워크 구성(VPN/DMZ)

![네트워크 구성도(VPN/DMZ)](./images/05_network_architecture_vpn_dmz.png)

외부 시스템 연계 시 IPSec VPN, 전용선 등의 보안통신을 구성합니다(SFR-001).

| 네트워크 구간 | 보안 구성 | 용도 |
|---------|---------|------|
| **Public → DMZ** | SSL/TLS, WAF, IP 필터링 | 국문/영문사이트 외부 접속 |
| **DMZ → Private Cloud** | 방화벽, 접근제어 | WEB→WAS→DB 내부 통신 |
| **Private Cloud → 중진공 IDC** | IPsec VPN 터널 | 통합정보시스템, Oracle DB 연계 |
| **중진공 업무망 → 관리자** | VPN, 업무망 내부 | 관리자사이트 접근(SFR-001) |
| **영문사이트 → CDN** | HTTPS, 엣지 캐시 | 해외 바이어 이미지 캐쉬서버(SFR-005) |

> **컨소시엄 강점**: 아스트라비전은 APEX/FECT 플랫폼을 Nginx/Tomcat 기반으로 운영 중이며, 퀸텟시스템즈는 AWS ISV Partner로서 클라우드 네트워크 보안 설계 경험을 보유하고 있습니다.

### 2.4 표준프레임워크 적용

### 2.5 개발방법론

#### 2.5.1 방법론 개요

본 사업은 **CBD(Component Based Development) 기반 반복적/점진적 개발 방법론**을 적용합니다. CBD 방법론은 재사용 가능한 컴포넌트 단위로 시스템을 설계/개발하여 개발 생산성과 품질을 동시에 확보하는 방법론으로, 본 사업의 4개 사이트 재구축 및 하이브리드 클라우드 전환에 최적화된 접근 방식입니다(PMR-003, PMR-004).

**방법론 선정 근거**

| 선정 기준 | CBD 방법론 적합성 | 본 사업 적용 |
|---------|------------|---------|
| **재사용성** | 컴포넌트 단위 재사용으로 개발 효율 극대화 | 4개 사이트 공통 컴포넌트(회원관리, 권한관리, 게시판 등) 재활용 |
| **점진적 개발** | 반복 사이클을 통한 위험 조기 발견 | 4주 스프린트 단위 점진적 기능 구현/검증 |
| **아키텍처 중심** | 시스템 구조 우선 설계로 안정성 확보 | 하이브리드 클라우드 아키텍처 선행 설계 후 기능 구현 |
| **공공사업 적합성** | 정부 SW개발사업 방법론 가이드라인 준수 | 전자정부 표준프레임워크 5.0과 자연스러운 통합 |

> **컨소시엄 강점**: 아스트라비전은 CBD 방법론을 **FTA Korea(8년 연속)**, **FECT 플랫폼**, **KISTI 공공R&D 플랫폼** 등 다수 공공사업에서 적용한 경험을 보유하고 있습니다. 4주 스프린트 기반 반복적 개발 프로세스를 자체적으로 수립하여 운영 중이며, CI/CD 파이프라인(GitHub Actions)을 통한 배포 자동화를 실현하고 있습니다.

### 2.6 추진체계

본 사업은 **퀸텟시스템즈(총괄) + 아스트라비전** 공동수급 컨소시엄이 수행하며, 베트남 온라인 수출플랫폼 모델 전수 ODA 사업에서 검증된 동일한 협업 체계를 재현합니다. PM(임진규)을 중심으로 수출 도메인 전문가, AI/SW 기술 전문가, 엔터프라이즈 시스템 전문가가 3축 협업 체계를 구성하여 사업 목표를 달성합니다.

## III. 기술 및 기능

### 3.1 기능요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 13개 기능요구사항(SFR-001~SFR-013)에 대한 구현 방안을 상세히 기술합니다. 퀸텟시스템즈(총괄)와 아스트라비전 컨소시엄은 buyKOREA 실무 운영 경험(KOTRA AI무역지원센터 3개소), 영문 B2B 플랫폼 구축 경험(Gobiz Vietnam, FECT), 그리고 대용량 메시지 처리 역량(RabbitMQ 일 100만건+)을 기반으로 68개 요구사항을 빠짐없이 충족합니다.

---

#### 3.1.1 클라우드 네이티브 적용 (SFR-001)

**[요구사항 개요]**

SFR-001은 공개 SW 중심의 클라우드 네이티브 환경을 구성하고, IPsec VPN을 통한 하이브리드 클라우드를 적용하며, 개발/운영 환경 분리, 오토스케일링, CI/CD 파이프라인을 구현하는 요구사항입니다.

**[구현 방안]**

**(1) 공개 SW 기반 시스템 환경 구성**

본 사업에서는 Linux OS, Nginx(WEB), Tomcat(WAS), MariaDB(신규DB) 등 공개 SW 중심으로 시스템 환경을 구성합니다. 시스템 SW의 버전 업그레이드 및 소프트웨어 변경에도 서비스 안정성이 보장되도록 컨테이너 기반 배포 환경을 적용합니다.

| 구분 | 적용 기술 | 버전 | 비고 |
|------|---------|------|------|
| OS | Linux (Rocky/Ubuntu) | 최신 LTS | 공개 SW |
| WEB | Nginx | 최신 안정 버전 | 리버스 프록시, 로드밸런싱 |
| WAS | Tomcat | 10.x | 전자정부FW 5.0 호환 |
| DB(신규) | MariaDB | 10.11+ | 고비즈코리아 메인 DB |
| DB(기존) | Oracle | 기존 활용 | 통합정보시스템(수출) |
| JDK | Open JDK | 17/21/23 | 멀티 버전 지원 |
| Framework | 전자정부 표준프레임워크 | 5.0 | 필수 적용 |

> **컨소시엄 강점**: 아스트라비전은 전자정부 표준프레임워크 기반의 FTA Korea 시스템을 2011년부터 최초 개발하고 8년간 연속 운영한 경험을 보유하고 있습니다. ODA 사업(Gobiz Vietnam)에서도 Spring Boot + Docker + Nginx 기반의 클라우드 네이티브 아키텍처를 성공적으로 적용하였습니다.

**(2) IPsec VPN 기반 보안통신 구성**

외부 시스템 연계 시 IPsec VPN, 전용선 등의 보안통신을 구성합니다. 민간 클라우드와 중진공 IDC 간 VPN 연결을 통해 하이브리드 클라우드를 적용하며, 국정원 보안성검토에 따른 보안정책을 준수합니다.

- 중진공 업무망과 고비즈코리아 내부망 간 IPsec VPN 터널 구성
- 관리자사이트(Nexacro N V24)는 업무망에서 접속하여 행정처리 편의성 강화
- 국정원 보안성검토 결과에 따른 보안정책 적용

**(3) 개발 및 운영 환경 분리**

애플리케이션의 개발을 위한 클러스터와 운영을 위한 클러스터의 구성을 분리합니다.

| 환경 | 위치 | 용도 | 접근 범위 |
|------|------|------|---------|
| 개발 환경 | 고비즈코리아 내부망 | 소스 개발/테스트 | 개발팀 전용 |
| 운영 환경(국문/영문) | 고비즈코리아 DMZ | 대외 서비스 | 인터넷 공개 |
| 운영 환경(정보시스템) | 중진공 IDC | 내부 행정처리 | VPN 접속 |

**(4) 오토스케일링 정책**

업무서비스 부하에 따른 컨테이너 리소스 용량 및 오토스케일링 정책을 마련하여, 클러스터 자동 확장 시 서비스 중단을 최소화합니다.

- CPU/메모리 사용률 기반 수평적 오토스케일링(HPA) 적용
- 이벤트(수출박람회, 바이어매칭 행사 등) 기간 트래픽 급증 대비 사전 스케일 아웃
- 클러스터 노드 오토스케일링으로 인프라 자원 탄력적 운영

**(5) CI/CD 파이프라인**

지속적 통합과 배포를 지원하는 CI/CD 파이프라인 기능을 제공합니다.

- 파이프라인 실행: 자동(트리거), 배포자 수동, 스케줄러 기반 배치 형태 전체 지원
- 배포 이력관리 및 롤백 지원
- 코드 품질 검사, 보안 취약점 스캔, 자동 테스트를 파이프라인에 통합

> **컨소시엄 강점**: 아스트라비전은 GitHub Action 기반 CI/CD 파이프라인을 FECT, APEX 플랫폼에 실 적용하고 있으며, 퀸텟시스템즈는 CALS Platz의 빌드 파이프라인을 통해 배포 시간 40% 단축을 실증하였습니다.

#### 3.1.2 컨테이너 기반 배포 (SFR-002)

**[요구사항 개요]**

SFR-002는 컨테이너화된 애플리케이션의 자동화된 배포 구현, 컨테이너 레지스트리 제공, 무중단 배포(Blue/Green, Rolling, Canary) 지원을 요구합니다.

**[구현 방안]**

**(1) 컨테이너 이미지 관리**

각 업무서비스를 독립적으로 개발, 배포, 변경할 수 있도록 Docker 컨테이너 이미지로 구성합니다.

- 이미지 리파지토리 생성, 컨테이너별 이미지 지정 및 리소스 할당
- 로드밸런싱을 통한 서비스 공개
- 애플리케이션 파일 배포 시 실행환경의 자동 생성 및 구동 지원

**(2) 컨테이너 레지스트리**

컨테이너 이미지를 안전하게 저장 관리하기 위해 프라이빗 레지스트리를 제공합니다.

- HTTPS를 통해 암호화되어 전송
- 네임스페이스를 사용하여 컨테이너 레지스트리(Registry, 이미지 저장소) 정의
- 개발 환경 이미지와 운영에 배포되는 이미지의 레지스트리 분리
- 운영환경 이미지 전송 시 신뢰할 수 있는 HTTPS 프로토콜 사용

**(3) 무중단 배포 지원**

서비스 중단 없이 새 버전을 배포할 수 있도록 다양한 무중단 배포 방식을 제시합니다.

| 배포 방식 | 적용 대상 | 특징 |
|---------|---------|------|
| Blue/Green | 국문/영문 사이트 전체 배포 | 신/구 버전 동시 운영 후 전환, 즉시 롤백 가능 |
| Rolling | 일반 업데이트 | 순차적 교체, 서비스 연속성 보장 |

#### 3.1.3 API Gateway (SFR-003)

**[요구사항 개요]**

SFR-003은 API 호출을 위한 토큰 발급/인증, 엔드포인트별 접근정책, API 라우팅, 프로토콜 변환, 모니터링 기능을 요구합니다.

**[구현 방안]**

**(1) 토큰 기반 인증 및 인가**

API 호출을 위한 토큰 발급 및 인증 체계를 구축합니다. 엔드포인트별 API 호출 인증 및 인가, 접근 정책(특정 클라이언트의 API 호출 불허에 의한 접근제어 기능)을 구현합니다.

- JWT(JSON Web Token) 기반 인증/인가 처리
- API Key 발급 및 관리
- Rate Limiting, Throttling 정책 적용
- IP 기반 접근 제어

**(2) API 라우팅 및 프로토콜 변환**

동일 API를 클라이언트나 마이크로서비스에 따라 다른 엔드포인트를 통해 서비스를 제공하도록 API 라우팅을 지원합니다.

- 동일 API를 여러 클라이언트/마이크로서비스별로 엔드포인트 제공
- HTTP, REST, XML 웹 서비스 등 다양한 프로토콜 변환 처리
- 동기/비동기 API 호출 패턴 변환 기능 제공

**(3) 로깅 및 모니터링**

로깅, 인증 등 공통 기능을 중복 개발 또는 처리하지 않도록 요청과 응답의 표준화 및 공통 로직을 처리합니다.

- API 호출 패턴 분석, 접근 상태 분석
- 요청 IP/클라이언트 일시 등 API 호출에 대한 로깅 및 모니터링
- 비정상 호출 탐지 및 알림

> **컨소시엄 강점**: 아스트라비전은 FECT 플랫폼에서 REST API 기반 기관 연계(관세청 UNI-PASS, 대한상공회의소)를 구현한 경험이 있으며, RabbitMQ 기반 일 100만건 이상의 비동기 메시지 처리 아키텍처를 운영하고 있습니다.

- 수출협상진행, 상품검수관리, 수출계약체결, 상담관리, 통계관리, 상품등록관리, 교육과정관리, SMS관리, 매칭메이킹관리, 온라인전시관 관리

**(6) 개인정보 기능개발**

- 통화기록 등 보관

### 3.2 데이터요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 10개 데이터요구사항(DAR-001~DAR-010)에 대한 구현 방안을 기술합니다. 퀸텟시스템즈(총괄)와 아스트라비전 컨소시엄은 FTA Korea 데이터 설계/운영 8년, FECT 무역데이터 표준화, VLE 11개사 마이그레이션 경험을 바탕으로 수출 데이터 표준 아키텍처 수립과 안전한 마이그레이션을 통해 데이터 무결성을 보장합니다.

---

#### 3.2.1 데이터표준 수립/관리 (DAR-001, DAR-002)

**[요구사항 개요]**

DAR-001은 데이터 표준 개선/정비 및 상위표준 준수를, DAR-002는 시스템 구축기간 동안 데이터 표준 관리를 위한 방안(변경이력 포함)을 요구합니다.

**[구현 방안]**

**(1) 데이터 표준 수립 (DAR-001)**

시스템 고도화(기능 추가 등)에 따라 추가적으로 필요한 데이터 표준항목(단어, 용어, 도메인, 코드)을 정의하고, 기관 데이터표준 시스템(S-Meta) 데이터 표준 사전 및 모델등록시스템(DA#)에 추가(반영)합니다.

- DB 표준항목 제정 시 범정부 표준(공공기관의 데이터베이스 표준화 지침, 행정안전부 고시), 공통표준용어와 발주기관 또는 국내외 산업 표준을 준수
- 표준 사전에 준수 여부를 식별할 수 있도록 체계 구축

| 표준 영역 | 주요 활동 | 산출물 |
|---------|---------|------|
| 단어 표준 | 업무용어 수집, 표준단어 확정 | 데이터 표준 사전 |
| 용어 표준 | 표준용어 정의, 도메인 매핑 | 데이터 표준 사전 |
| 도메인 표준 | 데이터 타입/길이/제약조건 정의 | 도메인 정의서 |
| 코드 표준 | 코드 분류체계 수립 | 코드 표준 사전 |

**(2) 데이터 표준관리 방안 (DAR-002)**

시스템 구축기간 동안 데이터 표준 관리를 위한 방안(변경이력 포함)을 제시하고, 표준 사전과 데이터 항목의 일치여부를 주기적으로 점검합니다.

- 데이터 표준 변경 관리 프로세스 수립 (변경 요청 → 검토 → 승인 → 반영 → 이력관리)
- 주기적 표준 준수 점검(주 1회) 및 위반 사항 보고
- 표준 사전 버전 관리 및 변경 이력 추적

> **컨소시엄 강점**: 아스트라비전은 FTA Korea 시스템에서 8년간 원산지관리 데이터 표준(HS코드, FTA 협정정보, 원산지 결정기준 등)을 수립/관리한 경험이 있으며, FECT 플랫폼에서 수출입 데이터 표준화를 수행하였습니다.

---

#### 3.2.2 구조설계/검증 (DAR-003, DAR-004)

**[요구사항 개요]**

DAR-003은 데이터 구조 설계(모델링) 기준 제시와 주제영역 정의 및 개념/논리/물리 모델 설계를, DAR-004는 설계된 데이터 모델에 대한 다자간(설계자/개발자/발주기관/전문가) 검증을 요구합니다.

**[구현 방안]**

**(1) 데이터 구조 설계 원칙 (DAR-003)**

- 데이터 구조 설계(모델링) 기준을 제시하고 이를 준수하여 데이터 구조를 설계, 구현
- 데이터 주제영역 정의 및 개념/논리/물리 모델 설계
- 데이터 활용, 업무요건 변화, 시스템 변경 등으로 인한 DB의 구조적 변화가 최소화될 수 있도록 유연한 구조로 설계
- 업무규칙(BR) 처리를 위한 데이터 주제영역, 개념 데이터 모델, 논리 데이터 모델, 물리 데이터 모델이 설계
- 데이터 설계 표준(데이터 분류체계, 명명 규칙, DB Object 사용기준 등)을 정의하고 적용

| 주제영역 | 주요 엔티티(예시) |
|---------|-------------|
| 회원 | 개인회원, 기업회원, 해외바이어 |
| 상품 | 상품정보, 상품카테고리, 상품이미지 |
| 수출지원 | 사업신청, 평가, 선정, 정산 |
| 매칭 | 바이어매칭, 상담이력, 거래이력 |
| 통계 | 수출실적, 마케팅통계, 매칭통계 |
| 시스템 | 권한, 코드, 메뉴, 로그 |

**(2) 데이터 모델 검증 (DAR-004)**

- 설계된 데이터 모델은 설계자, 개발자, 발주기관, 전문가 등이 참여하여 검증
- 각 단계별 데이터 모델 검증 시 기준을 정의하고 실시

| 검증 단계 | 검증 기준 | 참여자 |
|---------|---------|------|
| 개념 모델 검증 | 주제영역 적절성, 엔티티 식별 완전성 | 설계자, 발주기관 |
| 논리 모델 검증 | 정규화, 관계 정의, 속성 적절성 | 설계자, 개발자, 전문가 |
| 물리 모델 검증 | 인덱스, 파티셔닝, 성능 최적화 | 설계자, DBA, 발주기관 |

---

#### 3.2.3 구조관리/값검증 (DAR-005, DAR-006)

**[요구사항 개요]**

DAR-005는 시스템 구축기간 동안 데이터 구조 관리 방안(변경이력 포함)을, DAR-006은 데이터 값 검증 방안(진단 계획, 업무규칙 정의, 오류 데이터 관리)을 요구합니다.

**[구현 방안]**

**(1) 데이터 구조 관리 (DAR-005)**

- 시스템 구축기간 동안 데이터 구조 관리를 위한 방안(변경이력 포함) 제시
- DB의 형상과 물리데이터 모델의 형상 일치 여부를 주기적으로 점검
- 데이터 구조 변경 시 영향도 분석(Impact Analysis) 수행 후 변경 적용

**(2) 데이터 값 검증 (DAR-006)**

- 데이터 값 진단 계획 수립: 데이터 값 검증 범위(전체 또는 일부), 시기, 방법을 사업 계획에 명시
- 데이터 값 진단 수행 및 검증: 목표시스템에 추가되는 데이터의 특성을 분석하고, 대상 DB(시스템)의 구축/운영 근거(법령, 규정, 규칙, 매뉴얼 등)를 참고하여 데이터 특성에 부합하는 업무 규칙(BR)을 정의
- 공공데이터 범정부 품질진단 기준 및 업무규칙(BR)에 따라 값 검증을 수행
- 데이터 값 검증 결과에 따른 오류데이터를 개선하고, 오류로 인한 영향도, 오류 유형별 조치 방법 등을 관리
- 반복적으로 발생하는 동일 오류가 있는 경우 재발 방지 방안 제시
- 오류 데이터 입력 방지(사전검증) 방안 제시

---

#### 3.2.4 메타데이터/정합성 (DAR-007, DAR-008)

**[요구사항 개요]**

DAR-007은 메타데이터 관리체계 현행화를, DAR-008은 데이터 전환, 이관, 운영과 관련한 데이터 정제 및 품질 검증을 요구합니다.

**[구현 방안]**

**(1) 메타데이터 관리체계 (DAR-007)**

- 목표시스템의 메타데이터 표준 관리 항목이 발주기관의 메타데이터 관리시스템과 중앙메타시스템에 등록 및 현행화 되도록 지원
- 발주기관에서 운영 중인 메타데이터 관리시스템이 없는 경우 문서, 파일 등으로 메타데이터 항목을 작성하여 제출

**(2) 데이터 정합성 검증 (DAR-008)**

- 데이터 전환, 이관, 운영과 관련하여 데이터 정제 및 품질 검증을 수행
- 정제/검증 결과에 대해 중진공의 승인을 받아야 함

| 검증 유형 | 검증 내용 | 검증 시점 |
|---------|---------|---------|
| 마이그레이션 전 | 원본 데이터 품질 진단, 정제 대상 식별 | 데이터 이관 전 |
| 마이그레이션 중 | 건수 비교, 값 비교, 매핑 검증 | 이관 수행 시 |
| 마이그레이션 후 | 정합성 검증, 업무 시나리오 검증 | 이관 완료 후 |

---

#### 3.2.5 복구/개방 (DAR-009, DAR-010)

**[요구사항 개요]**

DAR-009는 보안 사고 발생 시 데이터 복구 체계를, DAR-010은 공공 데이터 개방 및 API 재개발을 요구합니다.

**[구현 방안]**

**(1) 데이터 복구 (DAR-009)**

- 보안 사고가 발생하여 데이터가 위변조, 손실되면 중진공이 제시한 복구소요시간 이내에 데이터를 복구
- 시스템은 데이터 복구를 관리해야 하며, 관련 표준체계를 수립 및 적용

| 복구 유형 | 복구 방안 | 목표 복구시간 |
|---------|---------|----------|
| 전체 복구 | 전체 DB 백업으로부터 복원 | 중진공 제시 기준 |
| 시점 복구 | 트랜잭션 로그 기반 특정 시점 복구 | 중진공 제시 기준 |
| 부분 복구 | 테이블/스키마 단위 선택적 복구 | 중진공 제시 기준 |

**(2) 데이터 개방 (DAR-010)**

- 기존 공공데이터 및 API를 검토하여 현행화 및 재개발
- 개방 대상 API는 정부표준 준수(디지털서비스개방 표준 API 명세서)
- 데이터 구조변경에 따른 개방가능한 데이터 검토

---

**[데이터 마이그레이션 전략]**

![데이터 마이그레이션 전략도 (ETL 파이프라인)](./images/05_data_migration_strategy.png)

본 사업의 데이터 마이그레이션은 3단계 ETL(Extract-Transform-Load) 파이프라인으로 수행합니다.

| 단계 | 주요 활동 | 품질 체크포인트 |
|------|---------|------------|
| Phase 1: 추출(Extract) | 기존 Oracle DB 데이터 프로파일링, 스키마 분석 | 원본 데이터 건수/품질 확인 |
| Phase 2: 변환(Transform) | 데이터 정제, 포맷 변환, 표준 매핑, 검증 규칙 적용 | 변환 규칙 검증, 오류 데이터 처리 |
| Phase 3: 적재(Load) | MariaDB(신규) 및 Oracle(기존) 적재, 검증 | 건수 비교, 값 비교, 정합성 검증 |

- 단계별 롤백 전략 수립으로 마이그레이션 실패 시 원상 복구 보장
- 병렬 처리를 통한 대용량 데이터 이관 시간 최소화
- 마이그레이션 완료 후 중진공 승인을 받아 최종 확정

> **컨소시엄 강점**: 아스트라비전은 VLE 2.0 시스템을 11개사에 구축하며 기존 ERP 시스템으로부터의 대규모 데이터 마이그레이션을 다수 수행하였습니다. FTA Korea 시스템에서는 Oracle 기반 원산지관리 데이터를 8년간 설계/운영하며 데이터 표준화 및 품질관리 역량을 축적하였습니다. 이러한 실무 경험을 바탕으로, 본 사업의 하이브리드 DB 환경(MariaDB 신규 + Oracle 기존 활용)에서의 데이터 무결성을 보장합니다.

### 3.3 보안요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 7개 보안요구사항(SER-001~SER-007)에 대한 충족 방안을 기술합니다. 퀸텟시스템즈(총괄)와 아스트라비전 컨소시엄은 국정원 보안정책 준수 다층 보안 체계와 원격개발 보안 경험을 바탕으로, 시큐어코딩, 개인정보보호, 참여인력 보안을 빈틈없이 이행합니다.

---

#### 3.3.1 보안관리지침/계획 (SER-001, SER-003)

**[요구사항 개요]**

SER-001은 발주기관 및 상위 기관의 보안 규정 및 지침 준수를, SER-003은 사업 수행과 연관된 문서, 인원, 장비 등에 대한 물리적/관리적/기술적 보안대책 및 누출금지 대상정보에 대한 보안관리 계획 제출을 요구합니다.

**[구현 방안]**

**(1) 보안관리 지침 준수 (SER-001)**

발주기관 및 상위 기관의 보안 규정과 지침을 철저히 준수합니다.

| 준수 대상 | 세부 내용 |
|---------|---------|
| 국가정보원 국가 정보보안 기본지침 | 보안성검토 결과에 따른 보안정책 및 조치사항 적용 |
| 중소벤처기업부 보안 규정 | 부처 보안 규정 및 지침 준수 |
| 중소벤처기업진흥공단 보안 지침 | 공단 자체 정보보안 규정 준수 |
| 개인정보보호법 | 개인정보보호 관련 법규, 지침 준수 |
| 행정안전부 소프트웨어 개발보안 가이드 | SW 개발 전 단계 보안 적용 |
| 행정기관 정보시스템 구축/운영 지침 | 공공기관 정보시스템 보안 기준 준수 |
| 소프트웨어 보안약점 진단가이드 | 보안취약점 진단 및 조치 |
| 정보화 용역사업 보안관리 실무가이드 | 용역사업 보안관리 체계 적용 |

**(2) 보안관리계획 제출 (SER-003)**

계약상대자는 사업 수행과 연관된 문서, 인원, 장비 등에 대한 물리적, 관리적, 기술적 보안대책 및 '누출금지 대상정보'에 대한 보안관리 계획을 사업수행계획서에 포함하여 제출합니다.

- 해당 정보 누출시 계약상대자는 「국가를 당사자로 하는 계약에 관한 법률 시행령」 제76조에 따라 부정당업자로 등록
- 계약상대자는 발주자의 보안정책(정보보안지침, 개인정보보호지침 등)을 준수
- 위반 시 "외주 용역사업 보안 특약"에 따라 관련자에 대하여 조치하고 손해배상 책임

**[보안관리 조직체계]**

| 역할 | 담당자 | 주요 업무 |
|------|--------|---------|
| 보안총괄책임자 | PM (임진규) | 보안 정책 총괄, 보안 사고 대응 총괄 |
| 보안관리담당자 | PMO (남의석) | 보안 계획 수립/이행 점검, 보안 교육 |
| 보안감독자 | PL (이상진 CTO) | 기술적 보안 감독, 시큐어코딩 점검 |
| 보안실무자 | 각 개발팀 | 보안 규정 준수, 보안 점검 참여 |

---

#### 3.3.2 개인정보보호 (SER-002)

**[요구사항 개요]**

SER-002는 수집되는 개인정보의 「개인정보보호법」 및 「개인정보의 안전성 확보조치 기준」에 따른 관리를 요구합니다.

**[구현 방안]**

**(1) 개인정보 수집 최소화 원칙**

- 서비스 제공에 필수적인 최소한의 개인정보만 수집
- 수집 목적 달성 후 즉시 파기 절차 이행

**(2) 개인정보 암호화 저장**

- 고유식별번호 등은 암호화하여 저장
- 중진공은 DB 보안(암호화)을 위해 전용 솔루션(DGuard) 사용 중

**(3) 개인정보 접속기록 관리**

- 개인정보 접속기록을 저장하고, 비정상적인 접속여부 등을 관리
- 접속기록 최소 2년 보관

**(4) 개인정보 파기 절차**

- 수집된 개인정보의 파기 절차를 구현
- 보유기간 경과 시 자동 파기 알림 및 처리

**(5) 가명정보 처리**

- 개인정보의 안정성을 고려하여 특정 정보는 가명정보, 합성정보 등으로 변환 수행

> **컨소시엄 강점**: 아스트라비전은 FTA Korea 시스템에서 기업의 원산지정보, 무역거래정보 등 민감한 비즈니스 데이터를 8년간 안전하게 관리한 경험이 있으며, SSL/시큐어코딩/관리자권한관리/SW개발보안을 실무에 적용하고 있습니다.

---

#### 3.3.3 응용프로그램/DB 보안 (SER-004)

**[요구사항 개요]**

SER-004는 DB 및 응용 프로그램 보안사항 준수/적용, 보안취약점 대응, 정보보안 지침 제29조(소프트웨어 개발보안) 준수를 요구합니다.

**[구현 방안]**

![다층 보안 체계도](./images/06_security_architecture.png)

**(1) DB 보안**

- 일반 사용자는 직접적으로 DB에 접근할 수 없도록 함
- 사용자 인증정보, 주민등록번호, 계좌번호, 패스워드 등은 소스코드에 직접 하드코딩 불가
- 패스워드, 주민등록번호, 계좌번호 등 중요한 개인 정보는 반드시 암호화하여 DB에 저장
- 관리자인 경우에도 화면에 해당 정보를 그대로 노출시키지 않아야 함
- 중진공은 DB 보안(암호화)을 위해 전용 솔루션(DGuard) 사용 중
- 사용자의 웹 브라우저와 웹 서버 사이에 소통되는 ID, 비밀번호 등 개인 정보의 암호화 처리

**(2) 응용프로그램 보안**

- 소프트웨어 보안약점 진단가이드(2019.6, 행정안전부, 한국인터넷진흥원) 준용하여 SW 개발보안 준수
- 보안 취약점 지적에 대하여 적극적으로 대응하며, 외부기관 및 발주기관에서 요구하는 일정 수준 이상으로 보안 수준을 업그레이드

**(3) 홈페이지 보안취약점 대응**

| 취약점 유형 | 대응 방안 |
|---------|---------|
| SQL Injection | PreparedStatement 사용, 입력값 검증 |
| XSS (Cross-Site Scripting) | 출력 인코딩, CSP 헤더 적용 |
| 파일다운로드 공격 | 경로 검증, 확장자 필터링 |
| URL 임의변경 | 세션 기반 권한 검증, URL 파라미터 암호화 |

**(4) 정보보안 지침 제29조 준수**

- 행정안전부 고시 제50조부터 제53조까지 반영하여 사업 수행
- 국정원, 중기부, 중진공 정보보안실 취약점 점검에 따른 보안취약점 이행

> **컨소시엄 강점**: 아스트라비전은 FTA Korea, FECT 플랫폼에서 SSL, 시큐어코딩, 관리자권한관리, SW개발보안을 실무 적용하고 있으며, 공공기관(산업통상자원부, 질병관리청, KISTI) 사업에서 보안성검토를 통과한 경험이 있습니다.

#### 3.3.4 참여인력/문서/장소 보안 (SER-005, SER-006, SER-007)

**[요구사항 개요]**

SER-005는 참여 인력에 대한 보안관리를, SER-006은 문서 및 전산자료에 대한 보안을, SER-007은 장소 및 장비, 네트워크 접근 보안 및 위반 시 조치사항을 요구합니다.

**[구현 방안]**

**(1) 참여인력 보안 (SER-005)**

| 보안 조치 | 세부 내용 |
|---------|---------|
| 보안서약서 제출 | 제안사 대표자는 별도 양식에 의한 보안서약서 제출(투입인력용, 업체대표용) |
| 신원조사 | 3개월 이상 투입되거나 비밀취급업무 수행 시 신원조사 실시 |
| 보안교육 | 전 참여인원에 대해 법률 및 규정에 의한 비밀 유지 의무 및 위반 시 처벌내용 교육 |
| 보안감독 분리 | 용역사업 감독과 보안감독 권한을 분리하여 별도의 보안 관리책임자 지정 |
| 인력 철수 시 | 전자적으로 기록된 자료는 국가정보원의 '정보시스템 저장매체 불용처리 지침'에 따라 완전삭제 후 반출 |

**(2) 문서 및 전산자료 보안 (SER-006)**

- 발주기관 정보보안지침에 의거 정보통신망 구성도, 정보시스템 구성도, 용역사업 산출물 및 개인정보 등 누출금지 정보는 비공개자료로 분류하여 관리
- 비공개자료 중 출력물 형태로 제공받는 자료에 대해 '자료관리대장'을 작성하여 인계자와 인수자(계약상대자)가 직접 서명한 후 인계/인수
- 용역사업 관련자료 및 사업과정에서 생산된 모든 산출물은 통합파일서버에 저장하거나 발주기관이 지정한 PC에 저장/관리
- 사업완료 후 생산된 최종 산출물은 인수/인계서를 작성하여 발주자에 반납하고, PC 완전삭제 등 보안조치를 통해 사업관련 산출물(중간산출물 포함)을 모두 삭제

**온라인 보안 관리:**
- 계약상대자가 사용하는 노트북/PC는 인터넷 연결을 금지함 (발주자 제공 별도 승인 장소 제외)
- 발주자 내/외부 간 자료 전송이 필요한 경우에는 암호화 등의 보안대책을 마련하여 수/발신
- 인터넷 파일공유 사이트 및 개인 메일함에 사업 관련 자료 저장 금지

**(3) 장소 및 장비 보안 (SER-007)**

| 보안 영역 | 보안 조치 |
|---------|---------|
| 사업수행 장소 | 시건장치 등 통제가 가능한 장소 사용 |
| 장비 반입/반출 | 반입반출은 반출/출입대장에 기재하여 관리 |
| PC/노트북 | 백신 프로그램 설치, 악성코드 등 위험요소 제거, 발주기관 보안프로그램 설치 |
| 보안검사 | 정기적으로 보안검사를 수행하고, 개인정보 보유현황 점검 |
| 물리적 보호 | 노트북 등 장비 유출방지를 위한 물리적 보호대책 강구 |
| 저장매체 | 개발장비, 저장매체 등 발주기관 보안담당자의 확인을 득한 후, 반출입 |

**네트워크 접근 보안:**
- 제안사가 사업수행과 관련하여 사용하는 전산망은 방화벽 등 보안시스템을 활용하여 네트워크에 제한적으로 접근이 가능하도록 통제

**보안위반 시 조치사항:**
- 중요 정보를 유출한 제안사는 '국가계약법 시행령'에 따라 입찰참가자격 제한 조치 적용
- 악성코드의 전염, 해킹 등 사고 유형별 위반자가 발생 시 위반자, 제안사, 감독자 등 대상별 제재기준에 따라 처벌
- 보안위반사항에 대한 내용은 '보안 특약사항'에 따름

> **컨소시엄 강점**: 아스트라비전은 한국-베트남 원격 협업 체계(하노이, 호치민 2개 거점, 20명)를 운영하며 원격개발 보안을 실무에 적용하고 있습니다. VPN 기반 보안 통신, 접근 권한 관리, 코드 리뷰 체계 등 원격 환경에서의 보안 관리 경험을 본 사업에 적용합니다. 또한 ODA 사업(중소벤처기업진흥공단 발주)을 수행하며 중진공의 보안 규정과 절차를 숙지하고 있습니다.

#### 3.4.2 작업장소 상호협의 결정 (COR-002)

**[요구사항 개요]**

COR-002는 사업 수행장소를 사업추진 효율성을 고려하여 상호 협의하여 정하도록 하며, 수행장소 및 환경구성(NW 등)에 대한 제반비용은 수행사가 부담하고, 중진공 보안지침에 위배되지 않아야 함을 요구합니다.

**[충족 방안]**

- 본 컨소시엄은 사업 착수 시 발주기관과 작업장소를 상호 협의하여 결정합니다
- 제안요청서 내 명시된 보안 요구사항을 준수한 작업장소를 제시하며, 중진공에서 제시된 작업장소에 관하여 우선 검토합니다
- 수행장소 및 환경구성(네트워크 등)에 대한 제반비용은 본 컨소시엄이 부담합니다
- 다만, 중진공에서 사업자가 제시한 작업장소가 보안 요구사항을 준수하지 못한 경우 거부할 수 있음을 인지합니다

| 구분 | 방안 |
|------|------|
| 작업장소 | 발주기관 협의 후 결정, 보안지침 준수 |
| 네트워크 | VPN 기반 보안 통신 구성, 비용 자체 부담 |
| 보안 환경 | 시건장치, CCTV, 출입통제 등 물리적 보안 확보 |

> **컨소시엄 강점**: 아스트라비전은 ODA 사업 수행 시 중소벤처기업진흥공단 내부 및 외부 작업장소에서의 협업 경험이 있어, 발주처의 보안 환경 요구사항을 충분히 이해하고 대응합니다.

#### 3.4.3 지식재산권 (COR-003)

**[요구사항 개요]**

COR-003은 본 사업과 관련하여 구현된 산출물에 대한 지적재산권은 중진공과 제안사가 공동 소유하며, 지식재산권/저작권/특허권에 대한 비용발생 시 본 사업에 포함하여 진행할 것을 요구합니다.

**[충족 방안]**

- 본 사업을 통해 개발되는 SW는 용역계약일반조건 제56조에 따라 타 기관과공동활용할 계획이 없음을 사전 명시합니다
- 구현된 산출물에 대한 지적재산권은 중진공과 제안사가 공동 소유합니다(계약예규 35조의 2)
- 국내외 지식재산권, 저작권, 특허권에 대한 비용발생 시 본사업에 포함하여 진행합니다
- 사업자는 최종 산출물이 국내외 지식재산권, 저작권, 특허권 등 침해이유로 중진공을 상대로 소송이 제기되었을 경우, 중진공은 책임지지 않으며 사업자는 관련소요경비 일체를 제공합니다

**오픈소스 라이선스 관리:**

| 관리 항목 | 세부 방안 |
|---------|---------|
| 라이선스 식별 | 사용 오픈소스 전수 조사 및 라이선스 유형 식별 |
| 호환성 검증 | GPL, LGPL, MIT, Apache 등 라이선스 호환성 검증 |
| 의무사항 이행 | 각 라이선스별 의무사항(고지, 소스 공개 등) 이행 |
| 관리 대장 | 오픈소스 사용 현황 관리 대장 작성/유지 |

> **컨소시엄 강점**: 아스트라비전은 '엑셀 문서 인공 지능 변환 시스템' 특허(제10-2563961호)를 보유하고 있어 지식재산권 관리에 대한 실무 경험이 있습니다. VLE 2.0 상표권 관리, 오픈소스 라이선스 검증(FTA Korea 시스템) 등 체계적인 지식재산권 관리를 수행합니다.

#### 3.4.4 기존 운영환경 호환 (COR-004)

**[요구사항 개요]**

- 시스템 통합 및 분산 설계, 데이터 유형, 프로세스 환경 유형, 사용자 유형, 시스템 간 물리적 네트워크 연결구성을 고려하여 구조 설계

**(3) 하이브리드 환경 호환**

## IV. 성능 및 품질

### 4.1 성능요구사항

본 절에서는 온라인수출플랫폼의 클라우드 전환에 따른 성능 요구사항 대응 방안을 제시합니다. 퀸텟시스템즈-아스트라비전 컨소시엄은 Kubernetes 기반 오토스케일링, CDN 글로벌 가속, APM 통합 모니터링을 적용하여 **응답시간 3초 이내**, **동시 사용자 1,000명 이상**, **시스템 자원 사용률 90% 이하**의 성능 목표를 달성합니다.

아스트라비전은 RabbitMQ 기반 분산 메시지 시스템으로 **일 100만건 이상의 트랜잭션 처리**, **평균 응답시간 5초 이내**, **가용성 99.9% 이상**을 실제 운영 환경에서 달성한 바 있으며, 이러한 대용량 처리 역량을 본 사업에 적용합니다.

#### 4.1.1 성능 일반 요구사항 (PER-001)

| 안정적 운영지원 및 사용자 지원 방안 | APM + 인프라 모니터링 이중 감시 체계 구축 |

**가. 성능 최적화 설계 원칙**

본 사업의 성능 최적화는 다음 4가지 원칙에 기반하여 설계합니다.

- **계층별 캐싱 전략**: CDN(정적 콘텐츠) - Redis(세션/데이터) - DB Query Cache(조회) 3단계 캐싱으로 서버 부하를 최소화합니다.
- **비동기 처리 아키텍처**: 대량메일/문자/카카오톡 발송(SFR-010), 통계 집계 등 장시간 소요 작업은 RabbitMQ 메시지 큐를 활용하여 비동기 처리합니다. 아스트라비전은 RabbitMQ 기반 분산 시스템으로 일 100만건 이상의 메시지를 안정적으로 처리한 실적을 보유하고 있습니다.
- **DB 성능 최적화**: MariaDB 쿼리 실행계획 분석, 인덱스 최적화, 슬로우 쿼리 모니터링을 상시 수행합니다. 대량 데이터 조회 시 페이징 처리 및 커서 기반 조회를 적용합니다.
- **프론트엔드 성능 최적화**: Nexacro N V24 기반 화면 렌더링 최적화, 이미지 Lazy Loading, 번들 사이즈 최소화를 적용합니다.

**나. 성능 모니터링 체계**

3계층 통합 모니터링 아키텍처를 구축하여 성능 이슈를 사전에 탐지하고 대응합니다.

#### 4.1.2 시스템 자원 사용 (PER-002)

| 요구사항 | 대응 방안 |
|---------|---------|
| WEB, WAS, DB 서버 성능 보장 | 계층별 자원 할당 최적화 및 독립 스케일링 |
| 시스템 자원 평균 사용율 90% 이하 | 70% 임계치 사전 알림 + 오토스케일링 자동 확장 |

**가. 자원 할당 설계**

클라우드 환경의 민간클라우드 인프라(WEB 2대, WAS 7대, DB 2대, NAS 2대)에 대해 다음과 같이 자원을 설계합니다.

| 서버 유형 | 구성 | 자원 설계 기준 | 평균 사용률 목표 |
|---------|-----|-----------|-----------|
| WEB (Nginx) | 2대 | 정적 콘텐츠 서빙 + 리버스 프록시 | 60% 이하 |
| WAS (Tomcat) | 7대 | 4개 사이트(국문/영문/관리자/통합정보) 분산 배치 | 70% 이하 |
| DB (MariaDB) | 2대 | Master-Slave 이중화, 읽기 부하 분산 | 70% 이하 |
| NAS | 2대 | 파일 스토리지, 이미지 캐시서버 연동 | 80% 이하 |

**나. 자원 사용률 관리 방안**

- **임계치 3단계 관리**: 주의(60%) - 경고(70%) - 위험(85%)으로 단계별 대응 체계를 수립합니다. 위험 단계 도달 전에 오토스케일링이 자동 작동하여 **평균 사용률 90% 이하**를 보장합니다.
- **용량 계획(Capacity Planning)**: 월별 자원 사용 추이를 분석하여 분기별 용량 계획을 수립하고, 수출 박람회 등 이벤트 시 사전 스케일업을 수행합니다.
- **기존 대국민서비스 및 내부서비스의 원활한 성능 보장**: WEB, WAS, DB 서버 각각에 대해 독립적인 모니터링을 수행하며, 특정 서비스의 자원 점유가 다른 서비스에 영향을 미치지 않도록 컨테이너 기반 격리를 적용합니다.

#### 4.1.3 응답시간 및 처리 성능 (PER-003)

|---------|---------|---------|

| WAS 구간 | Tomcat 커넥션풀 최적화, 스레드풀 튜닝 | 처리 대기시간 최소화 |
| DB 구간 | 쿼리 최적화, 인덱스 튜닝, Redis 캐시 | DB 응답 0.5초 이내 |

- **대량 자료 화면 출력/엑셀 다운로드**: 시스템에 영향을 주지 않도록 스트리밍 방식으로 처리하며, 대용량 엑셀 생성은 비동기 백그라운드 작업으로 전환합니다.
- **예외사항 관리**: 대량 데이터 검색요청, BATCH성 작업요청, 대용량 이미지(500KB 이상) 포함 화면은 3초 기준 적용을 제외하며, 기타 타당한 사유 발생 시 중진공과 협의하여 결정합니다.

**가. 오토스케일링 기반 동시 처리 보장**

![오토스케일링 정책 다이어그램](./images/14_autoscaling_policy_diagram.png)

[그림 4.1-2] 오토스케일링 정책 다이어그램

Kubernetes HPA(Horizontal Pod Autoscaler)를 활용한 동적 자원 관리 정책을 적용합니다.

| 정책 항목 | 설정값 | 비고 |
|---------|------|-----|
| 최소 Pod 수 | 2개 | 가용성 보장 (최소 이중화) |
| 최대 Pod 수 | 10개 | 비용 효율성 고려 상한 |
| Scale-out 임계치 | CPU 70% / Memory 80% | 부하 증가 시 자동 확장 |
| Scale-in 임계치 | CPU 30% / Memory 40% | 부하 감소 시 자동 축소 |
| Cooldown 기간 | 300초 | 빈번한 스케일링 방지 |

**나. 동시 접속 부하 대응 체계**

- **L4/L7 로드밸런서**: WEB 서버 앞단에 로드밸런서를 배치하여 트래픽을 균등 분산합니다. 라운드로빈 + 최소접속(Least Connection) 방식을 혼합 적용합니다.
- **세션 클러스터링**: Redis 기반 세션 공유를 통해 WAS 간 세션 일관성을 보장하며, 특정 WAS 장애 시에도 사용자 세션이 유지됩니다.
- **그레이스풀 디그레이드**: 최대 동시접속자 수 임계치의 90% 이상 시 서비스 지연 안내 페이지를 제공하여 사용자 경험을 보호합니다.
- **피크타임 대응**: 수출 박람회, 기업 모집 마감일 등 예측 가능한 트래픽 급증 시점에는 사전 스케일업을 수행합니다.

**다. 동시 처리 관련 보장 사항**

- 동시처리에 있어 시스템 부하를 최소화하고 성능저하가 없어야 하며, 필요시 HW/SW를 추가하여 이를 보완합니다.
- 성능 저하의 문제가 없음에 대한 공급사의 보장과 책임을 문서로써 확인합니다.
- 주어진 HW/SW 운영환경에서 동시처리가 목표값 달성이 어려운 경우 중진공과 상호 협의하여 변경 가능합니다.

### 4.2 인터페이스요구사항

본 절에서는 온라인수출플랫폼의 사용자 인터페이스(UI) 및 시스템 인터페이스 요구사항에 대한 대응 방안을 제시합니다. 퀸텟시스템즈-아스트라비전 컨소시엄은 **디지털 정부서비스 UI/UX 가이드라인**(2024.2.29 배포)을 전면 준수하며, **UX기획 3명 + UI디자인 3명 = 6명 전담 조직**을 편성하여 4개 사이트(국문/영문/관리자/통합정보시스템)의 UI/UX 재설계를 수행합니다.

김지예 PL(그래픽/웹 디자이너 12년)이 디자인 품질을 총괄하며, AIG 손해보험 캠페인, 삼성 LCDS SI, 이베이재팬(Qoo10) 기획전 디자인 등 대규모 프로젝트 디자인 경험을 본 사업에 적용합니다.

#### 4.2.1 사용자 인터페이스 방안 (INR-001)

| 요구사항 | 대응 방안 |
|---------|---------|
| 온라인 도움말 제공 | 컨텍스트 기반 온라인 도움말 + 서식 관련 정보 연동 |
| 사용자가 원하는 기능을 쉽게 찾을 수 있는 체계 | 직관적 메뉴 구조 + 통합 검색 + 퀵 메뉴 |
| 현재 위치 정보 표시 | 브레드크럼(Breadcrumb) 네비게이션 전 페이지 적용 |
| 사이트별 바로가기 퀵 메뉴 | 국문/영문/관리자/통합정보 4개 사이트별 맞춤 퀵 메뉴 |
| 직관적 인터페이스 | 사용자 유형별 대시보드 + 시각적 계층 구조 설계 |

**가. 사용자 편의성 중심 인터페이스 설계**

- **직관적 메뉴 구성**: 수출지원(공동물류/온라인수출플랫폼/전자상거래), 상품서비스, 정보제공, 애로센터, 마케팅서비스 등 주요 업무를 최상위 메뉴로 구성하여 2클릭 이내에 목표 기능에 도달할 수 있도록 설계합니다.
- **온라인 서식 지원**: 온라인 서식을 포함한 콘텐츠는 필요한 경우 서식과 관련 정보를 함께 제공합니다. 시스템은 콘텐츠의 모양이나 배치를 이해하기 쉽게 구성합니다.
- **통합 검색**: 상품, 기업, 바이어 정보를 통합 검색할 수 있는 전역 검색 기능을 제공합니다.
- **위치 정보**: 모든 페이지에 브레드크럼을 표시하여 사용자의 현재 위치를 명확히 인지할 수 있도록 합니다.

#### 4.2.2 온라인 도움말 및 절차서 (INR-002)

| 요구사항 | 대응 방안 |
|---------|---------|
| 온라인 도움말 기능 체계적 구성 | 기능별/업무별 도움말 트리 구조 설계 |
| 별도 교육 없이 이용 가능 | 단계별 가이드(Step-by-step) + 툴팁 제공 |
| 도움말 기능 지속 현행화 | 관리자 도움말 편집 기능 + 버전 관리 |

**가. 온라인 도움말 체계**

- **컨텍스트 도움말**: 각 화면에서 물음표(?) 아이콘을 클릭하면 해당 기능의 도움말이 팝업으로 표시됩니다. 사용자가 별도로 교육을 받지 않더라도 온라인 도움말 및 절차서를 이용하여 기능을 이용할 수 있도록 체계적으로 구성합니다.
- **업무 절차서**: 회원가입, 상품등록, 수출지원 신청 등 주요 업무에 대해 단계별 절차서를 제공합니다.
- **FAQ 시스템**: 자주 묻는 질문을 카테고리별로 정리하고, 검색 기능을 제공합니다.
- **현행화 관리**: 관리자 사이트에서 도움말 콘텐츠를 직접 편집할 수 있으며, 시스템 변경 시 도움말을 지속적으로 현행화할 수 있는 방안을 제시합니다.

#### 4.2.3 사이트 구성 및 디자인 설계 (INR-003)

| 요구사항 | 대응 방안 |
|---------|---------|

**가. UI/UX 설계 프로세스**

![UI/UX 설계 프로세스](./images/15_uiux_design_process.png)

[그림 4.2-1] UI/UX 설계 프로세스

6명 전담 디자인팀(UX기획 3명 + UI디자인 3명)이 5단계 프로세스를 통해 UI/UX를 설계합니다.

| 단계 | 활동 | 담당 | 산출물 |
|------|------|------|--------|
| 1. Research | 사용자 인터뷰, 현행 시스템 분석, 경쟁 분석 | UX기획팀 | 사용자 리서치 보고서 |
| 2. UX Planning | 사이트맵, 사용자 플로우, 와이어프레임 | UX기획팀 | IA 문서, 와이어프레임 |
| 3. UI Design | 비주얼 스타일, 컴포넌트 설계, 목업 | UI디자인팀 | UI 스타일가이드, 화면설계서 |
| 4. Usability Test | 사용자 테스트 세션, 피드백 수집 | UX기획팀 + 발주처 | 사용성 테스트 결과서 |
| 5. Design QA | 접근성 점검, 비주얼 리뷰, 이슈 리포팅 | 김지예 PL | 디자인 QA 체크리스트 |

**나. 디자인 원칙**

- **Layout**: 고정 창/팝업 창을 사용자 편의성 관점에서 구성하고, 모든 팝업 창은 사용자가 선택하여 닫을 수 있도록 설계합니다.
- **Color**: 사용자 피로도를 최소화하고 가독성을 고려한 색상 계열로 구성합니다. 고비즈코리아 브랜드 아이덴티티를 유지하면서 접근성 기준(명도대비 4.5:1 이상)을 준수합니다.
- **Icon**: 기능과 용도를 쉽게 파악할 수 있는 직관적 아이콘을 사용합니다.
- **Button**: 사용자에게 직관적이고 친화적인 느낌의 버튼을 사용합니다.

#### 4.2.4 사용단말을 고려한 화면 사이즈 (INR-004) / 스크린 사이즈 (INR-005)

|---------|---------|

| 화면 UI 기획/디자인 발주처 협조 | 주간 디자인 리뷰 회의 + 프로토타입 공유 |

**가. 반응형 웹 설계 전략**

![주요 화면 UI 와이어프레임 개념](./images/16_ui_wireframe_concept.png)

[그림 4.2-2] 주요 화면 UI 와이어프레임 개념 (Desktop/Mobile)

| Breakpoint | 대상 디바이스 | 레이아웃 |
|-----------|----------|---------|
| 1200px 이상 | Desktop (PC) | 3~4컬럼 그리드, 사이드바 표시 |
| 768px~1199px | Tablet | 2컬럼 그리드, 사이드바 접기 |
| 767px 이하 | Mobile | 1컬럼 스택, 햄버거 메뉴 |

- 특정 해상도에 최적화된 디자인을 지양하고, 스크린 및 구성요소 크기는 CSS를 사용하여 유동적으로 지정합니다.
- 수평스크롤은 사용하지 않도록 설계합니다(단, 출력용 조회 화면 등은 예외 가능).
- 필요시 사용자의 스크린 사이즈별 유동적인 서비스 제공을 위한 서비스 대상 콘텐츠를 선정하여 중진공과 협의 후 확정합니다.

#### 4.2.5 사용자 수행 결과에 대한 메시지 제공 (INR-006) / 확인 메시지 제공 방안 (INR-007)

| 요구사항 | 대응 방안 |
|---------|---------|
| 수행활동 확인 메시지 제공 | 등록/수정/삭제 완료 시 확인 메시지 표시 |
| 오류 메시지 제공 | 시스템/프로그램 오류 시 사용자 친화적 알림 |
| 개인정보 포함 정보 게시 경우 알림 | 개인정보 감지 시 경고 메시지 표시 |
| 미완료 저장 등 수행 활동 확인 | 이탈 방지 메시지(unsaved changes alert) |
| 표준 UI 채택 | 표준 운영체제 확인 메시지 UI 준수 |

**가. 메시지 분류 체계**

| 메시지 유형 | 표시 방식 | 예시 |
|----------|---------|------|
| 성공(Success) | 상단 토스트 알림(녹색) | "상품이 등록되었습니다" |
| 경고(Warning) | 모달 다이얼로그(노란색) | "입력하지 않은 필수항목이 있습니다" |
| 오류(Error) | 모달 다이얼로그(빨간색) | "시스템 오류가 발생하였습니다. 관리자에게 문의하세요" |
| 확인(Confirm) | 확인/취소 모달(파란색) | "삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다" |
| 정보(Info) | 인라인 메시지(파란색) | "개인정보가 포함된 정보입니다" |

- 등록, 수정, 저장, 삭제와 같이 사용자와 시스템 간의 상호작용에 대한 확인 메시지를 제공하되, 표준 운영체제 환경에서 일반적으로 사용되는 표준 UI를 채택하여 구현합니다.
- 삭제, 입력정보 완료 혹은 미완료 후 저장 등 사용자의 수행 활동에 대한 메시지를 제공하여 사용자가 작업 상태를 명확히 인지할 수 있도록 합니다.

#### 4.2.6 통합 UI (INR-008) / 통합UI 멀티플랫폼/브라우저 지원 (INR-009)

| 요구사항 | 대응 방안 |
|---------|---------|
| 전체 시스템 간 통일성 부여 UI | UI 컴포넌트 라이브러리(Design System) 구축 |
| UI/UX 공통가이드 준수 | 디지털 정부서비스 UI/UX 가이드라인(24.2.29) 전면 적용 |
| 중진공과 긴밀한 협조 | 주간 디자인 리뷰 + 시안 승인 프로세스 |
| 멀티플랫폼/멀티브라우저 | 웹 표준 기반 크로스브라우저 대응 |
| 기 구축 시스템 통합사용 환경 고려 | Nexacro N V24 기반 통합 UI 프레임워크 |

**가. 통합 UI 설계 체계**

4개 사이트(국문/영문/관리자/통합정보시스템) 간 통일된 UI를 구현하기 위해 공통 디자인 시스템을 구축합니다.

| 구성 요소 | 내용 |
|---------|------|
| 공통 레이아웃 | 헤더/푸터/사이드바/콘텐츠 영역 표준화 |
| 컴포넌트 라이브러리 | 버튼, 입력폼, 테이블, 카드, 모달 등 표준 컴포넌트 |
| 색상 팔레트 | 주색/보조색/배경색/텍스트색 통일 |
| 타이포그래피 | 제목/본문/캡션 서체 및 크기 표준 |
| 아이콘 세트 | 업무별 표준 아이콘 통일 |

- UI는 직관적이면서도 사용자의 효율적인 상호작용이 가능하여야 하며, 다양한 사용자 요구를 수용할 수 있도록 시스템의 유연한 변경이 가능하여야 합니다.
- **UI/UX 공통가이드**(디지털 정부서비스 UI/UX 가이드라인, 2024.2.29 배포)를 준수합니다.
- 화면 UI 기획 및 디자인에 관련된 제반 사항을 발주기관과 긴밀한 협조를 통하여 시행토록 합니다.

**나. 크로스브라우저 대응**

| 브라우저 | 지원 버전 | 테스트 방식 |
|---------|---------|---------|
| Chrome | 최신 2개 버전 | 자동화 테스트(Selenium) |
| Edge | 최신 2개 버전 | 자동화 테스트 |
| Safari | 최신 2개 버전 | 수동 + 자동화 병행 |
| Firefox | 최신 2개 버전 | 자동화 테스트 |
| Mobile (iOS/Android) | 최신 OS | 실기기 테스트 |

- 웹 표준 기반의 Client 표현기술을 적용하여 일반사용자가 다양한 OS 플랫폼과 다양한 브라우저 환경에서 서비스를 구현합니다.
- 브라우저에서 웹 페이지 깨짐 및 오작동 없이 구현합니다.
- 목표시스템 설계는 기 구축한 시스템을 포함한 통합사용 환경을 고려 하여 최적화된 Cross Browser 환경과 구축방안을 제시합니다.

| 통합정보시스템(Oracle DB) | IPsec VPN + API | 수출 정보 | SFR-012/013 |

- 데이터 연계의 정확성을 위해 개발 및 테스트 환경을 구성하고, 연계 어플리케이션의 기능적 정확성을 검증한 후 운영 시스템에 적용합니다.
- 사용자 또는 정보 증가 시에도 안정적인 성능을 제공하여야 하며, 추가 라이선스 비용 등이 발생하지 않아야 합니다.

**나. 연계데이터 정합성 확보 방안**

| 검증 단계 | 검증 방법 | 시점 |

| 연계 중 검증 | 트랜잭션 무결성, 타임아웃 관리 | API 호출 중 |

- 기관별 연계 방식에 따른 각 유형별 정합성 확보 방안을 제시합니다.

- 연계 대상 시스템과 원활한 연동이 이루어져야 하며, 과업에 제시한 사항을 구현하기 위해 기 도입된 시스템의 연동 및 업그레이드가 필요한 경우 기 도입시스템 업체와 협의하여 진행합니다.
- 타 행정정보 시스템과의 연계 또는 관련 표준을 준수하여야 합니다.

### 4.3 테스트요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 테스트 요구사항 대응 방안을 제시합니다. 퀸텟시스템즈-아스트라비전 컨소시엄은 **단위 - 통합 - 시스템 - 성능 - 인수** 5단계 체계적 테스트 프로세스를 적용하며, CI/CD 파이프라인 기반 자동화 테스트로 회귀결함을 방지합니다.

아스트라비전은 GitHub Action 기반 CI/CD 파이프라인을 운영하며, 퀸텟시스템즈는 CALS 플랫폼의 빌드 파이프라인을 통해 배포 시간을 40% 단축한 실적을 보유하고 있습니다. 이러한 자동화 테스트 역량을 본 사업에 적용하여 품질을 보장합니다.

- 기존 시스템과의 병행테스트 방안 수립 후 발주처 승인 하에 진행합니다.
- 해당 테스트서버의 테스트환경은 사업자가 사전에 직접 구성합니다.

**다. 통합테스트 관리**

- **요구사항 추적 매트릭스(RTM)**: 요구사항에 대한 시스템 반영결과 추적을 위한 요구사항 추적 매트릭스를 점검하고 기능의 정상적 수행여부를 검증합니다.
- **결함 추적**: 결함을 파악하고 원인을 추적하여 결함을 제거합니다.
- **테스트 결과 이력관리**: 테스트 결과 및 조치내역의 이력을 관리합니다.
- **시범운영 전환 시나리오**: 시범운영 및 전환가동을 위한 목표치, 검증 상세 시나리오를 제시합니다.

#### 4.3.4 인수테스트 (TER-004)

| 요구사항 | 대응 방안 |
|---------|---------|
| 승인(인수) 검사 및 테스트 시나리오 제시 | 수요기관 관점 인수테스트 시나리오 수립 |
| 검사/테스트 수행방법, 절차, 참여조직 기술 | 인수테스트 계획서 상세 기술 |
| 발주자에게 승인검사 및 테스트 요청 | 최종 산출물 + 테스트 결과물 첨부 제출 |
| 만족한 결과 시까지 보완/테스트 반복 | 결함 발견 시 보완 후 재테스트 수행 |
| 14일 이내 검사/테스트 | 요청일로부터 14일 이내 수행, 기간 연장 가능 |

**가. 인수테스트 수행 계획**

| 항목 | 내용 |
|------|------|
| 수행 방법 | 시나리오 기반 검증 + 자유 탐색 테스트 병행 |
| 수행 절차 | 계획 수립 - 환경 준비 - 테스트 실행 - 결함 보완 - 재테스트 - 승인 |
| 참여 조직 | 중진공(검수 담당), 퀸텟시스템즈(PM), 아스트라비전(기술지원) |
| 점검 사항 | 기능 완전성, 성능 목표 달성, 보안 준수, 데이터 정합성 |
| 최종 검수 기준 | 치명결함 0건, 주요결함 해소율 100% |

**나. 인수테스트 절차**

| 단계 | 활동 | 담당 | 산출물 |
|------|------|------|--------|
| 1. 계획 | 인수테스트 계획 수립, 시나리오 작성 | 제안사 + 중진공 | 인수테스트 계획서 |
| 2. 환경 | 운영환경 동일 테스트 환경 구성 | 제안사 | 테스트 환경 구성서 |
| 3. 실행 | 시나리오 기반 테스트 수행 | 중진공(주도) + 제안사(지원) | 테스트 실행 결과 |
| 4. 보완 | 발견 결함 수정 및 보완 | 제안사 | 결함 조치 보고서 |
| 5. 재테스트 | 보완 사항 확인 테스트 | 중진공 | 재테스트 결과 |
| 6. 승인 | 최종 승인 및 인수 | 중진공 | 인수 확인서 |

**다. 인수테스트 보장 사항**

- 수요기관의 승인(인수) 검사 및 테스트 요청과 관련된 시나리오를 제시합니다.
- 개발 완료 후 최종 산출물 및 테스트 결과물을 첨부하여 수요기관에게 승인 검사 및 테스트를 요청하여야 합니다.
- 승인 검사 및 테스트 과정에서 발견된 하자사항은 만족한 결과를 얻을 때까지 보완/테스트를 반복적으로 실시합니다.
- 본 사업 구축 완료 후 승인 검사 및 테스트를 요청할 수 있으며, 발주자는 검사 및 테스트 요청일로부터 14일 이내에 검사/테스트하여야 합니다. 검사/테스트 과정에서 기일이 추가로 소요되는 경우 상호 합의에 의해서 기간 연장이 가능합니다.
- 수요기관은 현장에 시급하게 적용해야 할 프로그램에 대해서는 사업 중이라도 승인 검사 및 테스트를 실시할 수 있습니다.

### 4.4 품질요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 품질 요구사항 대응 방안을 제시합니다. 퀸텟시스템즈-아스트라비전 컨소시엄은 **분석 - 설계 - 구현 - 테스트** 전 단계에 걸친 체계적 품질관리를 수행하며, PMO 남의석(공공프로젝트 관리 10년)이 품질관리를 전담합니다.

남의석 PMO는 2017년부터 2025년까지 연속 8년간 원산지관리시스템 종합컨설팅 사업의 산출물/일정 관리 및 품질 점검을 수행하며 축적한 공공프로젝트 품질관리 노하우를 본 사업에 적용합니다.

시스템은 제공되기로 한 요구사항을 모두 제공하며, 초기 협의한 요구사항에서 변경관리 절차를 통해 승인을 획득한 요구사항을 최종 기준요건으로 간주합니다.

| 추적 항목 | 관리 방법 |
|---------|---------|
| 요구사항 ID (SFR/PER/INR/TER/QUR 등) | RTM에 전체 등록 |
| 설계 매핑 | 요구사항 - 설계서 - 테스트케이스 연결 |
| 구현 매핑 | 설계서 - 소스코드 - 단위테스트 연결 |
| 검증 매핑 | 테스트케이스 - 테스트결과 연결 |

**나. 변경관리 절차**

| 단계 | 활동 | 승인 주체 |
|------|------|---------|
| 1. 변경 요청 | 변경사항 식별, 변경요청서 작성 | 요청자 |
| 2. 영향 분석 | 일정/비용/품질 영향도 분석 | 기술팀 |
| 3. 승인/반려 | 변경통제위원회(CCB) 심의 | PM + 중진공 |
| 4. 반영 | 변경 사항 구현 및 검증 | 개발팀 |

- 제공되기로 한 요구사항을 제공하는지 여부는 각 기능 요구사항의 검증(테스트) 활동을 통해 예상된 결과가 도출되었을 경우 요구사항을 제공한 것으로 평가합니다.
- 기능 구현 정확성은 사용자가 직접 테스트를 수행 기간에 테스트를 수행함으로써 평가합니다.
- 신규 기능 개발로 인해 기존 기능 및 성능에 영향을 미치지 않아야 합니다.
- 신규 기능 구축 시 정보시스템 운영 및 유지관리의 효율성 제고를 위해 기능별 활용 여부 점검이 가능하도록 구현하여야 합니다.

| 오류 방지 및 오류 발생 시 메시지 공지 | 입력 검증 + 사용자 친화적 오류 메시지 |
| 결함 수/지속시간 측정, 5% 이상 시 오픈 연장 | 결함 추적 시스템으로 정량 관리 |
| 결함지속시간 최대 한계값 1시간 이내 | 장애 대응 체계 + 자동 복구 메커니즘 |
| 시스템 목표 달성 신뢰성 측정/평가 | SLA 기반 신뢰성 지표 모니터링 |
| 고객 수 증가에 제한 없는 확장 | K8s 오토스케일링 + 수평 확장 아키텍처 |
| 지속적 업그레이드 및 유지보수 방안 | 컨테이너 기반 무중단 배포, 모듈 독립 업데이트 |

**가. 신뢰성 확보 방안**

| 신뢰성 지표 | 목표값 | 확보 방안 |
|---------|------|---------|
| 가용성 | 99.9% 이상 | 이중화(WEB/WAS/DB) + 자동 복구 |
| 결함 발생률 | 5% 미만 | CI/CD 자동테스트 + 코드리뷰 |
| 결함 지속시간 | 1시간 이내 | 장애 감지(5분) - 분석(15분) - 조치(40분) |
| 데이터 무결성 | 100% | 트랜잭션 관리 + 백업/복구 체계 |

- 복구할 수 없는 자료의 손실로 이어질 수 있는 오류를 방지하고 오류 발생 시 즉시 사용자에게 관련 메시지를 공지해야 합니다.
- 시험운영 기간동안 발견된 결함 수와 결함의 지속시간을 측정해야 하며, 결함 발생율이 5% 이상인 경우에는 시스템 오픈기간을 연장합니다.
- 시험운영 기간동안 결함지속시간의 최대 한계값은 1시간 이내여야 합니다.

**나. 확장성 확보 방안**

| 확장 영역 | 현재 설계 | 확장 방안 |
|---------|---------|---------|
| 사용자 수 | 동시 1,000명 | K8s HPA로 Pod 자동 증설, 최대 10배 확장 |
| 데이터량 | 현행 수준 | MariaDB 파티셔닝, NAS 용량 자동 확장 |
| 기능 추가 | 4개 사이트 | MSA 기반 서비스 독립 추가/변경 |
| 외부 연계 | 현행 연계 시스템 | API Gateway 통한 신규 연계 용이 |

- 시스템에 포함된 모든 제품은 고객의 수에 제한되지 않고 증가를 수용할 수 있도록 확장될 수 있어야 합니다.
- 지속적으로 업그레이드 및 유지보수하기 위한 방안을 제시합니다. 컨테이너 기반 무중단 배포(Blue/Green, Rolling)를 적용하여 서비스 중단 없이 업그레이드가 가능합니다.
- 시스템의 목표 달성을 검증할 신뢰성 있는 측정 및 평가 방법을 제시합니다. 가용성, 응답시간, 처리량, 에러율 등 핵심 지표를 실시간 모니터링하여 SLA 달성 여부를 지속적으로 검증합니다.

#### 5.1.1 사업위험관리

위험관리는 사업 수행 과정에서 발생 가능한 위험요소를 사전에 식별하고, 체계적으로 대응하여 사업 목표 달성에 미치는 부정적 영향을 최소화하는 활동입니다.

**가. 위험 식별 및 분류 체계**

| 위험 유형 | 위험 항목(예시) | 발생 가능성 | 영향도 | 대응 전략 |
|---------|------------|---------|------|--------|
| 일정 위험 | 클라우드 전환 일정 지연 | 중 | 상 | 4주 스프린트 단위 진척 점검, 버퍼 일정 확보 |
| 기술 위험 | 레거시 데이터 마이그레이션 오류 | 중 | 상 | 사전 데이터 정합성 검증, 단계별 이행 리허설 |
| 자원 위험 | 핵심인력 이탈 | 하 | 상 | 백업 인력 사전 지정, 기술 문서화 강화 |
| 외부 위험 | 연계기관(관세청, KTNET 등) 협조 지연 | 중 | 중 | 조기 협의 착수, 주관기관 지원 요청 |
| 품질 위험 | 요구사항 변경 빈발 | 중 | 중 | 변경관리 프로세스 적용, 영향도 사전 분석 |

**나. 위험관리 프로세스**

위험관리는 다음 4단계 프로세스를 반복 수행합니다.

1. **위험 식별**: 착수 단계에서 위험 목록을 초기 작성하고, 매주 주간회의에서 신규 위험을 식별합니다.
2. **위험 분석**: 발생 가능성과 영향도를 기준으로 위험 등급(상/중/하)을 산정합니다.
3. **대응 계획 수립**: 위험 등급별 대응 전략(회피/전가/완화/수용)을 수립합니다.
4. **모니터링 및 통제**: 주간 단위로 위험 상태를 추적하고, 위험 발생 시 즉시 대응합니다.

**다. 위험관리 도구 및 산출물**

- **위험관리 대장(Risk Register)**: 전체 위험 항목, 등급, 담당자, 대응 현황을 일원화 관리합니다.
- **위험 현황 보고**: 주간보고서에 위험 현황을 포함하여 주관기관에 보고합니다.
- **이슈 및 위험관리 식별 시 수시 보고**: PMR-002에 따라 성공적 완수에 영향을 미치는 이슈 및 위험관리 대상 항목이 식별된 경우 수시로 주관기관/전문기관과 협의하고, 이슈 및 위험이 해결될 때까지 추적관리를 실시합니다.

> **PMO 남의석 팀장의 역량**: 원산지관리시스템 8년 연속 수행(2017~2025) 과정에서 매년 위험관리 대장을 작성하고 운영한 경험을 본 사업에 그대로 적용합니다. 특히, KTNET/산업통상자원부 발주 공공사업에서의 위험 식별 및 대응 노하우를 보유하고 있습니다.

#### 5.1.2 사업진도관리

270일(약 9개월) 사업 기간 내 모든 과업을 완료하기 위해, WBS 기반의 진도관리를 수행합니다.

**가. 진도관리 체계**

| 관리 항목 | 주기 | 방법 | 담당 |
|---------|------|------|------|
| WBS 기반 진척률 관리 | 주간 | WBS 항목별 계획 대비 실적 산출 | PMO(남의석) |
| 스프린트 진도 점검 | 4주 | 스프린트 목표 달성률 확인 | PM(임진규) |

- 지연이 2주 이상 지속될 경우, 중진공 주관기관에 즉시 보고하고 대응 방안을 협의합니다.

| 관리 항목 | 방법 | 도구 |

**나. 문서 분류 및 코드 체계**

| 테스트 | TS | TS-001 테스트 계획서 |

**다. 용역 완료 시 산출물 제출**

PMR-001에 따라 용역 완료 시 제출하는 최종 결과물은 발주자의 승인을 득한 후 원본 파일 매체를 상호협의하여 수록하고 산출물과 같이 제출합니다.

#### 5.1.5 보고관리

PMR-002에 따라, 과업기간 동안 주관기관과의 협의에 따라 착수보고회, 중간보고회, 최종보고회를 실시하며, 매주/매월 사업 진척사항, 이슈사항 및 대응방안 등을 포함하여 보고합니다.

**가. 보고 체계**

| 보고 유형 | 주기 | 참석자 | 주요 내용 | 산출물 |
|---------|------|--------|---------|--------|
| **착수보고** | 사업 착수 시(M) | PM, PMO, 중진공 담당 | 사업수행계획, 조직, 일정, 위험 | 착수보고서 |
| **주간보고** | 매주 | PM, PMO, 중진공 담당 | 주간 진척, 이슈, 차주 계획 | 주간보고서 |
| **월간보고** | 매월 | PM, PMO, 중진공 담당 | 월간 성과, 일정 현황, 위험 | 월간보고서 |
| **중간보고** | M+4 | PM, PMO, PL, 중진공 | 행정서비스/대량메일 시범오픈 결과, 진행 현황 | 중간보고서 |
| **수시보고** | 이슈 발생 시 | PM, 관련 담당 | 긴급 이슈, 위험 발생 | 이슈보고서 |
| **최종보고** | M+8 | 전체 | 사업 완료 결과, 성과, 인수인계 | 최종보고서 |

**나. 보고서 작성 기준**

- 모든 보고서는 중진공의 서식 및 양식을 준수하여 작성합니다.
- 주간보고서에는 금주 실적, 차주 계획, 이슈/위험 현황, 진척률을 필수 포함합니다.
- 이슈 및 위험관리 대상 항목이 식별된 경우, 해결될 때까지 추적관리를 실시(회의록 관리)하고, 타 사업산출물과 함께 발주기관에도 동일하게 제출합니다.

|---------|---------|--------|

> QUR-002에 따라, 시스템은 제공되기로 한 요구사항을 모두 제공하며, 기능 구현 정확성은 사용자가 직접 테스트를 수행함으로써 평가합니다. 신규 기능 개발로 인해 기존 기능 및 성능에 영향을 미치지 않도록 회귀 테스트를 반드시 실시합니다.

#### 6.1.3 품질 지표 및 측정

QUR-004에 따라, 시스템의 목표 달성을 검증할 신뢰성 있는 측정 및 평가 방법을 다음과 같이 제시합니다.

| 품질 지표 | 측정 기준 | 목표 값 |
|---------|---------|--------|
| 요구사항 충족률 | 구현된 기능 수 / 전체 요구 기능 수 x 100 | 100% |
| 결함 밀도 | 결함 수 / KLOC(천 라인) | 5건 이하/KLOC |
| 시험운영 결함율 | 결함 발생률 5% 미만 시 정상 오픈 | 5% 미만 |
| 결함지속시간 | 시험운영 기간 최대 한계값 | 1시간 이내 |
| 코드 커버리지 | 테스트 커버리지 비율 | 80% 이상 |
| 보안 취약점 | 보안점검 결과 미해결 취약점 | 0건 |

> QUR-004에 따라, 시험운영 기간동안 발견된 결함 수와 결함의 지속시간을 측정하며, 결함 발생율이 5% 이상인 경우에는 시스템 오픈기간을 연장합니다. 시험운영 기간 결함지속시간의 최대 한계값은 1시간 이내로 관리합니다.

**외부 요인 대응(QUR-001)**

본 과업범위 외의 요인(운영서버 등 정보자원)으로 인해 사업 결과에 영향이 있거나 예상되는 경우, 원인과 해결방안을 발주기관에 제시하여 위험요소를 최소화하거나 문제가 해결될 수 있도록 적극 협조합니다. DB구축을 포함하는 경우 사업완료(검수) 시 데이터품질진단 및 오류데이터 개선을 실시합니다.

### 6.2 교육훈련

체계적이고 효율적인 교육을 위하여 사용자, 관리자, 시스템운영자 대상별로 맞춤형 교육훈련 계획을 수립합니다. 교육훈련계획 작성 시 교육대상, 교육일정 및 장소, 내용, 교재 등 기타 제반사항은 발주기관과 협의하여 결정합니다.

> **[요구사항 대응]** PSR-001(기술이전 교육)

---

#### 6.2.1 교육 대상 구분

PSR-001에 따라, 교육대상을 사용자, 관리자 등으로 구분하여 대상별 맞춤 교육을 실시합니다.

| 교육 대상 | 대상자 | 교육 목표 | 예상 인원 |
|---------|-------|---------|---------|
| **사용자(일반)** | 중진공 온라인수출처 실무자, 사업관리 담당 | 시스템 주요 기능 활용, 업무 프로세스 이해 | 20~30명 |
| **관리자** | 시스템 관리 담당자, 콘텐츠 관리자 | 관리자 메뉴 활용, 권한 관리, 콘텐츠 운영 | 5~10명 |
| **시스템운영자** | IT 운영팀, 인프라 관리자 | 시스템 운영, 모니터링, 장애 대응, 보안 관리 | 3~5명 |

---

#### 6.2.2 교육 일정 및 장소

| 구분 | 일정 | 장소 | 비고 |
|------|------|------|------|
| 사용자 교육(1차) | M+7 (시범운영 시작 시점) | 중진공 회의실 또는 온라인 | 기본 기능 교육 |
| 사용자 교육(2차) | M+8 (안정화 단계) | 중진공 회의실 또는 온라인 | 심화 기능 교육 |
| 관리자 교육 | M+7 ~ M+8 | 중진공 회의실 | 관리자 메뉴, 운영 교육 |
| 시스템운영자 교육 | M+8 | 중진공 전산실 또는 온라인 | 운영/감시/보안/복구 교육 |

- 교육 일정은 발주기관과 협의하여 확정합니다.
- 대면 교육을 원칙으로 하되, 참석자 편의를 고려하여 온라인 병행 교육을 지원합니다.

- 교육 동영상을 제작하여 신규 인력 입사 시에도 자체 학습이 가능하도록 합니다.

### 6.3 하자보수방안

본 컨소시엄은 최종검수 완료일로부터 **12개월간** 무상 하자보수를 수행합니다. 한국 팀과 베트남 팀의 시차(2시간)를 활용하여 **근무시간 외에도 장애 대응이 가능한 24시간 지원 체계**를 구축하며, FTA Korea 시스템 유지보수 6년 연속 수행(2018~2023) 경험을 기반으로 안정적인 하자보수 서비스를 제공합니다.

#### 6.3.1 하자보수 계획 및 조직

**하자보수 기본 계획:**

| 항목 | 내용 |
|------|------|
| **하자보수 기간** | 최종검수 완료일로부터 **12개월** |
| **하자보수 범위** | 본 사업에서 개발한 전체 시스템 (국문/영문/관리자/통합정보시스템) |
| **대응 채널** | 전화, 이메일, 온라인 이슈트래커(Jira), 긴급 핫라인 |
| **대응 시간** | 긴급(1시간 이내), 중요(4시간 이내), 일반(24시간 이내) |
| **비용** | 무상 (하자보수 기간 내) |

**하자보수 전담 조직:**

| 역할 | 담당자 | 주요 업무 |
|------|--------|---------|
| **하자보수 총괄** | PM(임진규) | 하자보수 의사결정, 발주처 보고 |
| **기술 지원** | CTO(이상진) | 아키텍처 관련 장애 분석, 기술 의사결정 |
| **개발 대응** | 개발리드(지원호) | 코드 수정, 버그 패치, 배포 |
| **품질 관리** | PMO(남의석) | 하자보수 이력 관리, 보고서 작성 |
| **베트남 개발팀** | ASTRA TECHNOLOGY (20명) | 야간/휴일 긴급 대응, 코드 수정 지원 |

#### 6.3.2 하자보수 절차

하자보수는 **접수 → 분류 → 대응 → 검증 → 완료** 5단계 프로세스로 수행합니다.

![하자보수 대응 프로세스](./images/22_warranty_process_flow.png)

**단계별 상세 절차:**

| 단계 | 활동 | 산출물 | 소요시간 |
|------|------|--------|---------|
| **1. 접수** | 발주처 하자 접수, 접수번호 부여, 담당자 배정 | 하자접수대장 | 즉시 |

### 6.4 기밀보안

본 컨소시엄은 사업 수행 전 과정에서 기밀보안 체계를 수립하고, 저작권 존중, 시스템 보안성 확보, 개인정보보호 대책을 포괄하는 종합 보안 방안을 적용합니다. 특히 한국-베트남 원격 협업 환경에서의 보안 관리 경험(ODA 사업 수행)을 기반으로 원격개발 보안 방안을 구체적으로 제시합니다.

#### 6.4.1 기밀보안 체계 및 대책

**기밀보안 관리 원칙:**

본 사업에서 취급하는 모든 정보는 「국가정보보안기본지침」, 「전자정부법」, 「개인정보보호법」 등 관련 법령에 따라 관리합니다.

| 보안 영역 | 관리 항목 | 구체적 대책 |
|---------|---------|----------|
| **관리적 보안** | 보안 조직, 보안 교육, 보안 서약 | 전 참여인력 보안서약서 징구, 분기별 보안 교육 실시, 보안 담당자 지정 |
| **물리적 보안** | 작업장소, 장비, 출입통제 | 개발 장소 CCTV 설치, 외부 저장매체 반입 통제, 출입대장 관리 |
| **기술적 보안** | 네트워크, 시스템, 데이터 | VPN 기반 접근, 다단계 인증(MFA), 암호화(전송/저장), 접근 로그 관리 |
| **인적 보안** | 인력 관리, 퇴직자 관리 | 보안 등급별 접근 권한 차등 부여, 퇴직/이직 시 접근 권한 즉시 회수 |

**참여인력 보안 관리 (SER-005):**

| 항목 | 내용 |
|------|------|
| 보안서약서 | 사업 착수 전 전 인력 보안서약서 징구 (한국/베트남 동일 적용) |
| 보안 교육 | 사업 착수 시 + 분기별 정기 교육 (공공기관 보안 정책, 개인정보보호) |
| 접근 권한 | 역할 기반 접근 통제(RBAC), 최소 권한 원칙(Least Privilege) 적용 |
| 퇴직/이직 관리 | 접근 권한 즉시 회수, 보유 자료 반납, 비밀유지 의무 확인 |

**원격개발 보안 방안 (COR-002):**

한국-베트남 원격 협업 환경에서의 보안 대책을 수립합니다. 본 컨소시엄은 ODA 사업 수행 과정에서 동일한 원격 협업 환경을 운영한 경험을 보유하고 있습니다.

| 보안 항목 | 적용 방안 |
|---------|---------|
| **네트워크 보안** | IPsec VPN 전용 터널 구성, 접속 IP 화이트리스트, SSL/TLS 암호화 통신 |
| **접근 통제** | 다단계 인증(MFA) 필수, 역할 기반 접근 권한, 접속 이력 실시간 모니터링 |
| **코드 관리** | GitHub Enterprise 접근 통제, Branch Protection Rule, 코드리뷰 필수 승인 |
| **데이터 보호** | 운영 데이터 개발 환경 반입 금지, 테스트 데이터 마스킹 처리, 개인정보 비식별화 |
| **단말기 보안** | 개발 PC 보안 솔루션 설치 필수, 외부 저장매체 사용 통제, 화면 캡처 제한 |
| **감사 추적** | 전 접근 이력 로깅, 월간 보안 감사, 이상 접근 시 즉시 차단 및 보고 |

**문서 보안 (SER-006):**

| 항목 | 대책 |
|------|------|
| 문서 분류 | 기밀/대외비/일반 3등급 분류 체계 적용 |
| 문서 관리 | 형상관리 시스템(Git)을 통한 문서 버전 관리, 접근 이력 추적 |
| 인쇄/복사 통제 | 기밀 문서 인쇄 승인제, 워터마크 적용 |
| 폐기 관리 | 사업 종료 후 보유 자료 완전 삭제, 폐기 확인서 발급 |
