## 1. CI/CD 파이프라인 아키텍처

### 1.1 파이프라인 구성

```
[소스 커밋] → [코드 검증] → [이미지 빌드] → [레지스트리 푸시] → [배포 전략 선택] → [K8s 배포]
```

| 단계 | 도구 | 세부 내용 |
|------|------|---------|
| 소스 관리 | GitLab CE | 브랜치 전략(feature → develop → main), MR 기반 코드 리뷰, 이슈 트래킹 |
| 코드 검증 | Jenkins | 정적 분석(SonarQube), 단위 테스트, 보안 스캔 |
| 이미지 빌드 | Docker + Jenkins | Dockerfile 기반 컨테이너 이미지 빌드 |
| 레지스트리 | Harbor | HTTPS 암호화 전송, 취약점 스캐닝, RBAC 기반 접근 제어, 개발/운영 분리 |
| 배포 | Argo CD + K8s | GitOps 기반 선언적 배포, Helm/Kustomize 매니페스트 동기화 |

### 1.2 환경 분리

단일 클러스터에서 네임스페이스 기반으로 환경을 분리한다.

| 환경 | 클러스터 | 네임스페이스 | Argo CD Sync | 용도 |
|------|---------|------------|-------------|------|
| 스테이징(STG) | 운영 클러스터 | `staging` | 자동 Sync | QA·검수 검증 |
| 운영(PROD) | 운영 클러스터 | `production` | 수동 Sync (승인 후) | 실서비스 운영 |

## 2. 무중단 배포 전략

### 2.1 전략 개요

RFP(SFR-001, SFR-002)에서 요구하는 무중단 배포는 **Rolling Update, Blue/Green, Canary 세 가지 전략을 모두 지원**하되, 배포 상황에 따라 **최적 전략을 선택 적용**하는 방식으로 구현한다. 세 전략을 동시에 사용하는 것이 아니라, 각 배포 건마다 하나의 전략을 선택하여 실행한다.

### 2.2 전략별 정의 및 적용 기준

| 배포 전략 | 정의 | 적용 상황 | K8s 구현 방식 | 적용 비율 |
|---------|------|---------|-------------|---------|
| **Rolling Update** | 기존 Pod를 점진적으로 새 버전으로 교체 | 일상적인 배포, 소규모 변경, 패치 적용 | Deployment `strategy.type: RollingUpdate` | ~85% |
| **Blue/Green** | 신규 버전을 완전히 준비한 후 트래픽을 일괄 전환 | 대규모 변경, DB 스키마 변경, 전체 전환이 필요한 경우 | 두 개의 Deployment + Service selector 전환 | ~10% |
| **Canary** | 소수 트래픽을 신규 버전으로 먼저 라우팅하여 검증 | 리스크가 높은 변경, 신규 기능 출시, 성능 영향 검증 | 두 개의 Deployment + Ingress 트래픽 비율 조절 | ~5% |

### 2.3 Rolling Update (기본 전략)

K8s의 기본 배포 전략으로, 전체 배포의 약 85%에 적용한다.

**동작 방식**:
```
v1 Pod [■■■■] → v1 Pod [■■■] + v2 Pod [■] → v1 Pod [■■] + v2 Pod [■■] → v2 Pod [■■■■]
```

**K8s 설정**:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gobizkorea-web
  namespace: gobiz-kr
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # 최대 1개 Pod 추가 허용
      maxUnavailable: 0    # 서비스 중단 Pod 0 (무중단 보장)
  template:
    spec:
      containers:
      - name: web
        image: harbor.gobiz.kr/gobiz-kr/web:v2.0.1
        readinessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
```

**적용 기준**:
- DB 스키마 변경이 없는 애플리케이션 코드 변경
- 설정 변경, 보안 패치, 버그 수정
- 하위 호환성이 보장되는 업데이트

### 2.4 Blue/Green 배포

대규모 변경이나 즉시 롤백이 필요한 상황에 적용한다.

**동작 방식**:
```
[Blue v1 ■■■■] ← Service (현재 트래픽)
[Green v2 ■■■■]  (대기, 검증 중)

  ↓ 전환 시점

[Blue v1 ■■■■]  (대기, 롤백 대비)
[Green v2 ■■■■] ← Service (트래픽 전환)
```

**K8s 구현**:
```yaml
# Blue Deployment (현재 운영)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gobizkorea-web-blue
  labels:
    app: gobizkorea-web
    version: blue
spec:
  replicas: 4
  template:
    metadata:
      labels:
        app: gobizkorea-web
        version: blue
    spec:
      containers:
      - name: web
        image: harbor.gobiz.kr/gobiz-kr/web:v1.9.0

---
# Green Deployment (신규 버전)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gobizkorea-web-green
  labels:
    app: gobizkorea-web
    version: green
spec:
  replicas: 4
  template:
    metadata:
      labels:
        app: gobizkorea-web
        version: green
    spec:
      containers:
      - name: web
        image: harbor.gobiz.kr/gobiz-kr/web:v2.0.0

---
# Service — selector 전환으로 트래픽 라우팅
apiVersion: v1
kind: Service
metadata:
  name: gobizkorea-web
spec:
  selector:
    app: gobizkorea-web
    version: green   # blue → green 전환
  ports:
  - port: 80
    targetPort: 8080
```

**적용 기준**:
- DB 스키마 변경을 동반하는 대규모 릴리스
- 전자정부 프레임워크 버전 업그레이드
- 외부 연계 인터페이스 변경 (토스페이먼츠, 카페24, 아마존 API 등)
- 즉시 롤백이 반드시 필요한 배포

**롤백**: Service selector를 `version: blue`로 변경하면 즉시 이전 버전으로 복원 (수 초 이내)

### 2.5 Canary 배포

리스크가 높은 변경을 소수 트래픽으로 검증한 후 단계적으로 확대한다.

**동작 방식**:
```
[Stable v1 ■■■■] ← 90% 트래픽
[Canary  v2 ■]   ← 10% 트래픽 (모니터링)

  ↓ 검증 완료 시 단계적 확대

[Stable v1 ■■]   ← 50% 트래픽
[Canary  v2 ■■]  ← 50% 트래픽

  ↓ 최종 전환

[Canary  v2 ■■■■] ← 100% 트래픽
```

**K8s 구현 (Ingress 기반 트래픽 분할)**:
```yaml
# Stable Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gobizkorea-web-stable
spec:
  replicas: 4
  template:
    metadata:
      labels:
        app: gobizkorea-web
        track: stable
    spec:
      containers:
      - name: web
        image: harbor.gobiz.kr/gobiz-kr/web:v1.9.0

---
# Canary Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gobizkorea-web-canary
spec:
  replicas: 1
  template:
    metadata:
      labels:
        app: gobizkorea-web
        track: canary
    spec:
      containers:
      - name: web
        image: harbor.gobiz.kr/gobiz-kr/web:v2.0.0

---
# Ingress — 트래픽 비율 제어 (Nginx Ingress 어노테이션)
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: gobizkorea-web-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"  # 10% → 30% → 50% → 100%
spec:
  rules:
  - host: kr.gobizkorea.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: gobizkorea-web-canary
            port:
              number: 80
```

**단계적 확대 절차**:

| 단계 | Canary 비율 | 관찰 시간 | 판단 기준 |
|------|-----------|---------|---------|
| 1단계 | 10% | 30분 | 에러율 < 0.1%, 응답시간 정상 범위 |
| 2단계 | 30% | 1시간 | 에러율 유지, 리소스 사용량 정상 |
| 3단계 | 50% | 2시간 | 전체 지표 안정 |
| 4단계 | 100% | - | 전체 전환 완료, Stable 제거 |

**적용 기준**:
- 검색엔진 고도화 등 성능 영향이 불확실한 변경
- 신규 결제 연동 (토스페이먼츠) 최초 적용
- 대규모 UI/UX 재설계 반영 시

## 3. 서비스별 배포 전략 매핑

각 서비스(Namespace)별로 권장 기본 배포 전략을 정의한다.

| 서비스 (Namespace) | 기본 전략 | Blue/Green 적용 | Canary 적용 |
|-------------------|---------|---------------|------------|
| **국문사이트** (`gobiz-kr`) | Rolling Update | 프레임워크 업그레이드, 대규모 릴리스 | 검색엔진 변경, UI 전면 개편 |
| **영문사이트** (`gobiz-en`) | Rolling Update | 결제 연동 변경, CDN 설정 변경 | 토스페이먼츠 최초 적용, 아마존/카페24 연동 |
| **사업관리서비스** (`gobiz-admin`) | Rolling Update | 권한체계 변경, Nexacro 버전 업 | - (내부 사용자, Canary 불필요) |
| **통합정보시스템** (`gobiz-export`) | Rolling Update | Oracle 연계 변경, SSO 변경 | - (내부 사용자) |
| **외부연계 (API GW)** | Rolling Update | API 버전 변경, 인증 체계 변경 | 신규 외부 API 연동 최초 적용 |

## 4. 롤백 전략

| 배포 전략 | 롤백 방식 | 소요 시간 | 명령어 |
|---------|---------|---------|-------|
| Rolling Update | `kubectl rollout undo` | 1~3분 | `kubectl rollout undo deployment/gobizkorea-web -n gobiz-kr` |
| Blue/Green | Service selector 전환 | 수 초 | Service의 `version` label을 이전 버전으로 변경 |
| Canary | Canary Deployment 삭제 | 수 초 | `kubectl delete deployment gobizkorea-web-canary -n gobiz-kr` |

**자동 롤백 조건** (CI/CD 파이프라인에서 설정):
- 에러율이 배포 전 대비 2배 이상 증가
- 평균 응답시간이 임계값(예: 3초) 초과
- Health Check 연속 실패 3회 이상

## 5. CI/CD 파이프라인 배포 전략 통합

CI/CD 파이프라인에서 배포 전략을 파라미터로 선택할 수 있도록 구성한다.

```
┌─────────────────────────────────────────────────────────┐
│                    CI/CD 파이프라인                        │
│                                                         │
│  [소스 커밋] → [빌드] → [테스트] → [이미지 빌드/푸시]       │
│                                      │                  │
│                           ┌──────────┼──────────┐       │
│                           ▼          ▼          ▼       │
│                      [Rolling]  [Blue/Green] [Canary]   │
│                       (기본)     (선택)       (선택)      │
│                           │          │          │       │
│                           ▼          ▼          ▼       │
│                    [K8s 배포 실행 + 모니터링]               │
│                           │                             │
│                    [자동 롤백 판단]                        │
│                     ├─ 정상 → 배포 완료                    │
│                     └─ 이상 → 자동 롤백                    │
└─────────────────────────────────────────────────────────┘
```

**파이프라인 배포 파라미터 예시**:
```yaml
# Jenkinsfile / GitLab CI 배포 스테이지
deploy:
  parameters:
    DEPLOY_STRATEGY:
      type: choice
      choices:
        - rolling    # 기본값
        - blue-green
        - canary
    TARGET_NAMESPACE:
      type: choice
      choices:
        - gobiz-kr
        - gobiz-en
        - gobiz-admin
        - gobiz-export
    CANARY_WEIGHT:
      type: string
      default: "10"
      description: "Canary 배포 시 초기 트래픽 비율(%)"
```

## 6. 모니터링 연동

배포 시 Prometheus + Grafana 기반 모니터링과 연동하여 배포 상태를 실시간 추적한다.

| 모니터링 항목 | 메트릭 | 임계값 |
|------------|-------|-------|
| HTTP 에러율 | `rate(http_requests_total{code=~"5.."}[5m])` | < 0.1% |
| 응답 시간 (P99) | `histogram_quantile(0.99, http_request_duration_seconds)` | < 3s |
| Pod 재시작 횟수 | `kube_pod_container_status_restarts_total` | 0 |
| CPU/메모리 사용량 | `container_cpu_usage_seconds_total`, `container_memory_usage_bytes` | 리소스 요청의 80% 미만 |

## 7. 브랜치 전략 및 환경 배포

### 7.1 브랜치 구성

스테이징 서버 기반의 단순화된 브랜치 전략을 적용한다.

```
feature/* ──→ develop ──→ main
   │              │          │
  개발 작업     CI 빌드     스테이징 배포 + 운영 배포
              + 테스트      (Argo CD)
```

| 브랜치 | 역할 | 트리거 |
|--------|------|--------|
| `feature/*` | 기능 개발. MR을 통해 develop에 병합 | 개발자 push |
| `develop` | CI가 빌드·테스트를 수행하고 Harbor에 이미지를 push. 이 단계에서는 배포하지 않음 | MR 병합 |
| `main` | 스테이징 배포와 운영 배포를 순차적으로 처리하는 브랜치 | develop → main MR 병합 |

### 7.2 단일 클러스터 네임스페이스 분리 (권장)

하나의 클러스터에서 Kubernetes 네임스페이스로 스테이징과 운영 환경을 분리한다.

```
클러스터 (1대)
├── namespace: staging    ← 검증용
└── namespace: production ← 운영용
```

### 7.3 배포 흐름

```
main 병합
  → Argo CD가 staging 네임스페이스에 자동 배포
  → QA / 검수 완료
  → 담당자가 Argo CD에서 production 수동 Sync
```

| 단계 | 환경 | Argo CD Sync 정책 | 승인 |
|------|------|-------------------|------|
| 1단계 | staging | **자동 Sync** (Auto Sync) | 불필요 |
| 2단계 | QA/검수 | - | QA 담당자 검증 |
| 3단계 | production | **수동 Sync** (Manual Sync) | 담당자 승인 후 배포 |

### 7.4 매니페스트 저장소 구조

GitOps 방식으로 배포 매니페스트를 별도 저장소에서 관리한다.

```
deploy-repo/
├── staging/
│   └── values.yaml      ← auto sync
└── production/
    └── values.yaml      ← manual sync (승인 후 배포)
```

> staging과 production은 동일한 이미지 태그를 사용하되, 환경별 설정(리소스 제한, 레플리카 수, 외부 연동 엔드포인트 등)은 `values.yaml`에서 분리하여 관리한다.

## 8. CI/CD 도구 스택 상세

### 8.1 파이프라인 흐름

```
GitLab (SCM) → Jenkins (CI) → Harbor (Registry) → Argo CD (CD/GitOps)
```

### 8.2 구성요소별 역할 및 라이선스

#### GitLab (SCM)

- **역할**: 소스 코드 관리(Source Code Management). Git 기반 코드 저장소로, 코드 버전 관리, 코드 리뷰(Merge Request), 이슈 트래킹 등을 담당한다. 개발자가 코드를 push하면 파이프라인의 시작점이 된다.
- **라이선스**: MIT License (Community Edition). Enterprise Edition은 독자적인 상용 라이선스를 적용한다. CE는 무료 오픈소스이고, EE는 Free / Premium / Ultimate 유료 티어로 나뉜다.

#### Jenkins (CI Pipeline)

- **역할**: 지속적 통합(Continuous Integration). GitLab에서 코드 변경이 감지되면 자동으로 빌드, 테스트, 코드 품질 검사를 수행한다. 빌드가 성공하면 컨테이너 이미지를 생성하여 Harbor에 push한다.
- **라이선스**: MIT License. 완전 무료 오픈소스 프로젝트로, 플러그인 생태계가 매우 넓다. (일부 서드파티 플러그인은 별도 라이선스일 수 있음)

#### Harbor (Container Registry)

- **역할**: 컨테이너 이미지 저장소. Jenkins에서 빌드된 Docker/OCI 이미지를 저장·관리한다. 이미지 취약점 스캐닝, 서명, 복제(replication), RBAC 기반 접근 제어 등 엔터프라이즈급 기능을 제공한다.
- **라이선스**: Apache License 2.0. CNCF Graduated 프로젝트로, 완전 무료 오픈소스이다.

#### Argo CD (CD / GitOps)

- **역할**: 지속적 배포(Continuous Deployment) 및 GitOps. Git 저장소에 선언된 Kubernetes 매니페스트(Helm chart, Kustomize 등)와 클러스터의 실제 상태를 지속적으로 비교·동기화한다. Harbor에 새 이미지가 올라오면 Git 매니페스트를 업데이트하고, Argo CD가 이를 감지하여 자동 배포한다.
- **라이선스**: Apache License 2.0. CNCF Graduated 프로젝트로, 완전 무료 오픈소스이다.

## 9. 핵심 요약

> **세 가지 배포 전략(Rolling Update, Blue/Green, Canary)을 모두 지원하는 플랫폼을 구축**하되, 각 배포 건마다 **상황에 맞는 최적 전략을 선택 적용**한다.
>
> - **Rolling Update**: 기본 전략 (~85%), 일상적 배포에 사용
> - **Blue/Green**: 대규모 변경 시 즉시 전환/롤백이 필요한 경우 (~10%)
> - **Canary**: 리스크가 높은 변경을 소수 트래픽으로 사전 검증 (~5%)
>
> CI/CD 파이프라인에서 배포 전략을 파라미터로 선택할 수 있도록 통합하고, Prometheus/Grafana 모니터링과 연동하여 이상 감지 시 자동 롤백을 수행한다.
