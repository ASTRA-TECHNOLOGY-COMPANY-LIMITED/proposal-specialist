### 4.4 품질요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 품질 요구사항 대응 방안을 제시합니다. 퀸텟시스템즈-아스트라비전 컨소시엄은 **분석 - 설계 - 구현 - 테스트** 전 단계에 걸친 체계적 품질관리를 수행하며, PMO 남의석(공공프로젝트 관리 10년)이 품질관리를 전담합니다.

남의석 PMO는 2017년부터 2025년까지 연속 8년간 원산지관리시스템 종합컨설팅 사업의 산출물/일정 관리 및 품질 점검을 수행하며 축적한 공공프로젝트 품질관리 노하우를 본 사업에 적용합니다.

---

#### 4.4.1 품질관리 일반사항 (QUR-001)

| 요구사항 | 대응 방안 |
|---------|---------|
| 품질관리 조직과 운영절차 구체적 제시 | PM-PMO-QA 3단계 품질관리 조직 편성 |
| 품질관리계획을 제안서 및 사업수행계획서 기술 | 품질관리계획서 별도 수립 및 상세 기술 |
| 품질관리 조직을 통해 품질보증활동 수행 | PMO 남의석 전담, 단계별 품질 점검 |
| 과업범위 외 요인 해결방안 제시 | 위험 요소 사전 식별 및 대응계획 수립 |
| DB구축 시 데이터품질진단 및 오류데이터 개선 | 데이터 마이그레이션 품질검증 수행 |

**가. 품질관리 조직 체계**

| 역할 | 담당자 | 주요 활동 |
|------|--------|---------|
| PM (프로젝트 총괄) | 임진규 | 품질 목표 수립, 의사결정, 발주처 보고 |
| PMO (품질관리 전담) | 남의석 | 산출물 검토, 일정 점검, 품질 감사, 이슈 관리 |
| QA (품질보증) | 개발팀 QA | 코드리뷰, 테스트, 결함 추적 |
| CTO (기술품질) | 이상진 | 아키텍처 리뷰, 기술 표준 준수 점검 |

**나. 단계별 품질관리 활동**

| 단계 | 품질관리 활동 | 품질 점검 항목 | 산출물 |
|------|----------|----------|--------|
| **분석** | 요구사항 검토, 완전성/일관성 점검 | 요구사항 추적성, 명확성 | 요구사항 검토 결과서 |
| **설계** | 설계 표준 준수 점검, 설계 리뷰 | 아키텍처 적합성, DB 설계 표준 | 설계 리뷰 결과서 |
| **구현** | 코드리뷰, 정적분석, 코딩표준 준수 | 코드 품질, 보안 취약점, 중복도 | 코드 품질 보고서 |
| **테스트** | 테스트 커버리지 점검, 결함 분석 | 시험 통과율, 결함 해소율 | 테스트 품질 보고서 |

**다. 품질관리 계획 주요 내용**

품질활동의 제반절차 및 산출물을 명시한 품질관리계획을 제안서 및 사업수행계획서에 상세하게 기술하여야 하며, 이에 근거하여 체계적이고 효과적인 프로젝트 진행을 위하여 사업기간동안 품질관리 조직을 통해 품질보증활동을 수행하고 결과물을 제출하여야 합니다.

| 계획 항목 | 내용 |
|---------|------|
| 품질 목표 | 결함밀도, 시험 커버리지, 일정 준수율 등 정량적 목표 |
| 품질 기준 | 전자정부 표준프레임워크 개발 가이드, 시큐어코딩 가이드 |
| 점검 주기 | 주간 코드리뷰, 격주 품질 점검, 월간 품질 보고 |
| 산출물 관리 | 형상관리(Git) 기반 산출물 버전관리 및 이력추적 |
| 위험 관리 | 위험 등록부 운영, 주간 위험 점검 회의 |

- 본 과업범위 외의 요인(운영서버 등 정보자원)으로 인해 사업 결과에 영향이 있거나 예상되는 경우 원인과 해결방안을 발주기관에 제시하여 위험요소를 최소화 하거나 문제가 해결될 수 있도록 적극 협조하여야 합니다.
- DB구축을 포함하는 경우 사업완료(검수)시 데이터품질진단 및 오류데이터 개선을 실시하여야 합니다.

#### 4.4.2 기능 구현 정확성 (QUR-002)

| 요구사항 | 대응 방안 |
|---------|---------|
| 제공되기로 한 요구사항 모두 제공 | 요구사항 추적 매트릭스(RTM)로 전수 관리 |
| 변경관리 절차를 통해 최종 기준요건 확정 | 변경요청-검토-승인-반영 4단계 변경관리 |
| 검증(테스트) 활동 통한 확인 | 단위/통합/시스템/인수 4단계 검증 |
| 기능 구현 정확성은 사용자 직접 테스트로 평가 | UAT(사용자 인수 테스트) 수행 |
| 신규 기능이 기존 기능/성능에 미영향 | 회귀테스트 + 영향도 분석 |
| 기능별 활용 여부 점검 기능 | 기능 활용 현황 대시보드 구현 |

**가. 요구사항 추적 관리**

시스템은 제공되기로 한 요구사항을 모두 제공하며, 초기 협의한 요구사항에서 변경관리 절차를 통해 승인을 획득한 요구사항을 최종 기준요건으로 간주합니다.

| 추적 항목 | 관리 방법 |
|---------|---------|
| 요구사항 ID (SFR/PER/INR/TER/QUR 등) | RTM에 전체 등록 |
| 설계 매핑 | 요구사항 - 설계서 - 테스트케이스 연결 |
| 구현 매핑 | 설계서 - 소스코드 - 단위테스트 연결 |
| 검증 매핑 | 테스트케이스 - 테스트결과 연결 |

**나. 변경관리 절차**

| 단계 | 활동 | 승인 주체 |
|------|------|---------|
| 1. 변경 요청 | 변경사항 식별, 변경요청서 작성 | 요청자 |
| 2. 영향 분석 | 일정/비용/품질 영향도 분석 | 기술팀 |
| 3. 승인/반려 | 변경통제위원회(CCB) 심의 | PM + 중진공 |
| 4. 반영 | 변경 사항 구현 및 검증 | 개발팀 |

- 제공되기로 한 요구사항을 제공하는지 여부는 각 기능 요구사항의 검증(테스트) 활동을 통해 예상된 결과가 도출되었을 경우 요구사항을 제공한 것으로 평가합니다.
- 기능 구현 정확성은 사용자가 직접 테스트를 수행 기간에 테스트를 수행함으로써 평가합니다.
- 신규 기능 개발로 인해 기존 기능 및 성능에 영향을 미치지 않아야 합니다.
- 신규 기능 구축 시 정보시스템 운영 및 유지관리의 효율성 제고를 위해 기능별 활용 여부 점검이 가능하도록 구현하여야 합니다.

#### 4.4.3 상호운영성 (QUR-003)

| 요구사항 | 대응 방안 |
|---------|---------|
| 기술표준과의 상호 운영성 확보 | 전자정부 표준프레임워크 5.0 기반 개발 |
| 기능 구현 정확성 + 정보 무결성 + 데이터 정합성 | 3대 품질 지표 통합 검증 |
| 통합테스트, 시스템테스트 수행기간 검증 | 테스트 단계에서 상호운용성 검증 수행 |
| 행정기관 정보시스템 구축 지침 준수 | 행정안전부 고시 표준 기술 적용 |

**가. 상호운영성 확보 방안**

| 영역 | 적용 표준 | 검증 방법 |
|------|---------|---------|
| 개발 표준 | 전자정부 표준프레임워크 5.0, Open JDK 17/21/23 | 프레임워크 호환성 테스트 |
| 인터페이스 표준 | REST API, JSON, XML | API 규격 검증 |
| 데이터 표준 | 범정부 데이터 표준, 공통표준용어 | 데이터 정합성 검증 |
| 보안 표준 | 시큐어코딩 가이드, 소프트웨어 보안약점 진단가이드 | 보안 점검 |

- 목표시스템은 본 사업과 관련된 정보시스템 및 기술표준과의 상호 운영성을 확보해야 합니다.
- 시스템 인터페이스 요구사항 및 어플리케이션과 정보간의 상호작용을 하는 기능은 기능 구현의 정확성뿐만 아니라 정보의 무결성, 데이터 정합성을 보장해야 합니다.
- 상호 운용성 검증은 통합테스트, 시스템테스트 수행 기간에 테스트를 수행함으로써 평가합니다.
- 행정기관 및 공공기관 정보시스템 구축 운영 지침(행정안전부 고시)에 따른 표준 기술을 적용해야 합니다.

#### 4.4.4 신뢰성, 확장성 (QUR-004)

| 요구사항 | 대응 방안 |
|---------|---------|
| 오류 방지 및 오류 발생 시 메시지 공지 | 입력 검증 + 사용자 친화적 오류 메시지 |
| 결함 수/지속시간 측정, 5% 이상 시 오픈 연장 | 결함 추적 시스템으로 정량 관리 |
| 결함지속시간 최대 한계값 1시간 이내 | 장애 대응 체계 + 자동 복구 메커니즘 |
| 시스템 목표 달성 신뢰성 측정/평가 | SLA 기반 신뢰성 지표 모니터링 |
| 고객 수 증가에 제한 없는 확장 | K8s 오토스케일링 + 수평 확장 아키텍처 |
| 지속적 업그레이드 및 유지보수 방안 | 컨테이너 기반 무중단 배포, 모듈 독립 업데이트 |

**가. 신뢰성 확보 방안**

| 신뢰성 지표 | 목표값 | 확보 방안 |
|---------|------|---------|
| 가용성 | 99.9% 이상 | 이중화(WEB/WAS/DB) + 자동 복구 |
| 결함 발생률 | 5% 미만 | CI/CD 자동테스트 + 코드리뷰 |
| 결함 지속시간 | 1시간 이내 | 장애 감지(5분) - 분석(15분) - 조치(40분) |
| 데이터 무결성 | 100% | 트랜잭션 관리 + 백업/복구 체계 |

- 복구할 수 없는 자료의 손실로 이어질 수 있는 오류를 방지하고 오류 발생 시 즉시 사용자에게 관련 메시지를 공지해야 합니다.
- 시험운영 기간동안 발견된 결함 수와 결함의 지속시간을 측정해야 하며, 결함 발생율이 5% 이상인 경우에는 시스템 오픈기간을 연장합니다.
- 시험운영 기간동안 결함지속시간의 최대 한계값은 1시간 이내여야 합니다.

**나. 확장성 확보 방안**

| 확장 영역 | 현재 설계 | 확장 방안 |
|---------|---------|---------|
| 사용자 수 | 동시 1,000명 | K8s HPA로 Pod 자동 증설, 최대 10배 확장 |
| 데이터량 | 현행 수준 | MariaDB 파티셔닝, NAS 용량 자동 확장 |
| 기능 추가 | 4개 사이트 | MSA 기반 서비스 독립 추가/변경 |
| 외부 연계 | 현행 연계 시스템 | API Gateway 통한 신규 연계 용이 |

- 시스템에 포함된 모든 제품은 고객의 수에 제한되지 않고 증가를 수용할 수 있도록 확장될 수 있어야 합니다.
- 지속적으로 업그레이드 및 유지보수하기 위한 방안을 제시합니다. 컨테이너 기반 무중단 배포(Blue/Green, Rolling)를 적용하여 서비스 중단 없이 업그레이드가 가능합니다.
- 시스템의 목표 달성을 검증할 신뢰성 있는 측정 및 평가 방법을 제시합니다. 가용성, 응답시간, 처리량, 에러율 등 핵심 지표를 실시간 모니터링하여 SLA 달성 여부를 지속적으로 검증합니다.
