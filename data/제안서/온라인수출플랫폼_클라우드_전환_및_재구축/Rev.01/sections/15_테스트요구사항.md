### 4.3 테스트요구사항

본 절에서는 온라인수출플랫폼 클라우드 전환 및 재구축 사업의 테스트 요구사항 대응 방안을 제시합니다. 퀸텟시스템즈-아스트라비전 컨소시엄은 **단위 - 통합 - 시스템 - 성능 - 인수** 5단계 체계적 테스트 프로세스를 적용하며, CI/CD 파이프라인 기반 자동화 테스트로 회귀결함을 방지합니다.

아스트라비전은 GitHub Action 기반 CI/CD 파이프라인을 운영하며, 퀸텟시스템즈는 CALS 플랫폼의 빌드 파이프라인을 통해 배포 시간을 40% 단축한 실적을 보유하고 있습니다. 이러한 자동화 테스트 역량을 본 사업에 적용하여 품질을 보장합니다.

---

#### 4.3.1 테스트 방안 수립 (TER-001)

| 요구사항 | 대응 방안 |
|---------|---------|
| 업무별 단위시험, 통합시험 방안 제시 | 5단계 테스트 체계 수립 + 단계별 계획서 작성 |
| 검수일 이전까지 시험운영 완료 | 사업기간(270일) 내 단계별 일정 수립 |
| 구축 완료시까지 지속적 단위테스트 | CI/CD 파이프라인 자동 단위테스트 실행 |
| 시스템 간 안정적 전환 방안 | 병행운영 + 전환 시나리오 수립 |
| 시험 운영결과 오류사항 과업 기간 내 보완 | 결함 추적 관리 시스템 운영 |

**가. 5단계 테스트 체계**

![5단계 테스트 체계 다이어그램](../images/17_test_framework_5stage.png)

[그림 4.3-1] 5단계 테스트 체계 다이어그램

| 단계 | 테스트 유형 | 수행 시기 | 수행 주체 | 주요 도구 |
|------|---------|---------|---------|---------|
| 1단계 | 단위 테스트 | 개발 중 (상시) | 개발자 | JUnit, Jest, CI/CD 자동실행 |
| 2단계 | 통합 테스트 | 모듈 개발 완료 후 | 개발팀 + QA | API 테스트 도구, Selenium |
| 3단계 | 시스템 테스트 | 통합 완료 후 | QA팀 | E2E 시나리오, 보안스캔 |
| 4단계 | 성능 테스트 | 시스템 테스트 후 | 성능 전담 | JMeter, 부하테스트 도구 |
| 5단계 | 인수 테스트 | 개발 완료 후 | 중진공 + 제안사 | 인수 체크리스트, 시나리오 |

**나. CI/CD 기반 자동화 테스트 체계**

코드 변경 시마다 자동으로 단위테스트 및 회귀테스트를 실행하여 결함을 조기에 발견합니다.

| CI/CD 단계 | 자동 실행 항목 | 실패 시 조치 |
|----------|----------|----------|
| 코드 커밋 | 정적 분석(코드 품질 검사) | 커밋 거부, 개발자 알림 |
| 빌드 | 컴파일 + 단위테스트 실행 | 빌드 실패 알림, 머지 차단 |
| 통합 | API 테스트 + 통합테스트 실행 | 배포 차단, 결함 티켓 생성 |
| 배포(Dev) | 개발환경 자동 배포 + 스모크 테스트 | 롤백, PM 보고 |
| 배포(Staging) | 운영 동일 환경 테스트 | 운영 배포 차단 |

**다. 테스트 일정 계획**

270일 사업기간 내 테스트 일정을 다음과 같이 수립합니다.

| 구분 | 시작 시점 | 종료 시점 | 비고 |
|------|---------|---------|------|
| 단위 테스트 | 개발 착수 시 | 개발 완료 시 | 개발과 병행, CI/CD 자동 |
| 통합 테스트 | 모듈 완료 시점 | 시스템 테스트 전 | 모듈별 순차 진행 |
| 시스템 테스트 | 통합 완료 후 | 성능 테스트 전 | 4개 사이트 전체 |
| 성능 테스트 | 시스템 테스트 후 | 인수 테스트 전 | 성능 목표 검증 |
| 인수 테스트 | 성능 테스트 후 | 검수일 전 | 중진공 참여 |

- 검수일 이전까지 제시한 방안에 따라 시험운영을 완료합니다.
- 구축 완료시까지 지속적으로 단위테스트를 실시합니다.
- 시스템 간의 안정적인 전환을 위한 방안을 제시합니다.
- 시험 운영결과 오류사항 등을 과업 기간 내 보완이 완료되도록 합니다.

#### 4.3.2 단위테스트 (TER-002)

| 요구사항 | 대응 방안 |
|---------|---------|
| 단위 테스트 수행 방안 제시 | 개발 프로그램별 단위 테스트 수행 절차 수립 |
| 요건 반영도, 기능 구현도 점검 | 유스케이스 기반 시험 커버리지 관리 |
| 테스트 계획서 사전 작성 | 시험 범위/절차/일정/환경/평가기준 사전 정의 |
| 결과보고서 발주기관 제출 | 결함유형 분석 + 시험 커버리지 포함 보고서 |
| 발주기관 요청사항 반영 | 자체 시정조치 + 발주기관 요청 반영 |

**가. 단위테스트 수행 절차**

| 순서 | 활동 | 산출물 |
|------|------|--------|
| 1 | 테스트 계획 수립 (범위, 절차, 조직, 일정, 시험환경, 평가기준) | 단위테스트 계획서 |
| 2 | 테스트 케이스 작성 (시나리오, 처리 절차, 수행 데이터, 예상결과) | 테스트 케이스 명세서 |
| 3 | 테스트 실행 (자동 + 수동) | 테스트 실행 로그 |
| 4 | 결함 분석 및 조치 | 결함 추적 대장 |
| 5 | 결과 보고서 작성 및 발주기관 제출 | 단위테스트 결과서 |

**나. 단위테스트 품질 지표**

단위 시험 시에 다음 내용을 점검합니다.

| 점검 항목 | 측정 방법 | 목표 |
|---------|---------|------|
| 결함유형 분석 | 결함발생건수, 결함비율 | 결함비율 5% 이하 |
| 결함심각도 분석 | 치명/주요/단순/사소 결함별 발생 건수 | 치명결함 0건 |
| 결함발견 추세분석 | 시험일시, 발견결함 수 | 추세 감소 확인 |
| 시험 커버리지 | (시험대상 유스케이스 / 전체 유스케이스) x 100 | 95% 이상 |

- 사전에 테스트 계획서를 작성하여 제공하여야 하며, 결과와 향후 조치 방안이 포함된 단위 테스트 결과보고서를 발주기관에 제출하여야 합니다.
- 테스트 결과에 따른 자체적인 시정조치 이외에 발주기관이 요청한 사항을 반영하여야 합니다.

#### 4.3.3 통합테스트 (TER-003)

| 요구사항 | 대응 방안 |
|---------|---------|
| 단위테스트 완료 프로그램 대상 검증 | 시나리오 기반 통합테스트 수행 |
| 기능/성능/설계사양 충족여부 검증 | 요구사항 추적 매트릭스(RTM) 기반 점검 |
| 접근권한 및 업무 권한 적절성 | 역할별 접근권한 테스트 수행 |
| 대내시스템 간 연계 및 업무흐름 검증 | E2E 시나리오 기반 연계 테스트 |
| 실제 운영환경과 동일 구성 | 스테이징 환경에서 테스트 실시 |
| 기존 시스템과 병행테스트 | 병행운영 방안 수립 후 발주처 승인 하에 진행 |

**가. 통합테스트 수행 방안**

| 구분 | 내용 |
|------|------|
| 테스트 범위 | 4개 사이트(국문/영문/관리자/통합정보) 전체 기능 + 외부 연계 |
| 테스트 데이터 | 오류 데이터 포함 실 사용 시나리오 기반 데이터 구성 |
| 테스트 환경 | 실제 운영환경과 동일하게 구성하여 실시, 연계를 포함 |
| 수행 인력 | 개발팀 + QA + 실제사용자 또는 이에 준하는 수준의 인력 |

**나. 통합테스트 유형별 수행**

| 유형 | 검증 내용 | 산출물 |
|------|---------|--------|
| 기능 통합 | 모듈 간 데이터 흐름, 업무 프로세스 정상 동작 | 통합테스트 시나리오/케이스 |
| 연계 통합 | 외부 시스템(Tmydata, SIMS, 스마트트레이드허브) 연계 정상 동작 | 연계테스트 결과서 |
| 보안 통합 | SSO, 권한관리, 암호화 정상 동작 | 보안테스트 결과서 |
| 전환 통합 | 기존 시스템 데이터 마이그레이션 정합성 | 데이터 전환 검증 결과서 |

- 통합테스트 시나리오에 대한 설명과 시연을 통해 진행합니다.
- 기존 시스템과의 병행테스트 방안 수립 후 발주처 승인 하에 진행합니다.
- 해당 테스트서버의 테스트환경은 사업자가 사전에 직접 구성합니다.

**다. 통합테스트 관리**

- **요구사항 추적 매트릭스(RTM)**: 요구사항에 대한 시스템 반영결과 추적을 위한 요구사항 추적 매트릭스를 점검하고 기능의 정상적 수행여부를 검증합니다.
- **결함 추적**: 결함을 파악하고 원인을 추적하여 결함을 제거합니다.
- **테스트 결과 이력관리**: 테스트 결과 및 조치내역의 이력을 관리합니다.
- **시범운영 전환 시나리오**: 시범운영 및 전환가동을 위한 목표치, 검증 상세 시나리오를 제시합니다.

#### 4.3.4 인수테스트 (TER-004)

| 요구사항 | 대응 방안 |
|---------|---------|
| 승인(인수) 검사 및 테스트 시나리오 제시 | 수요기관 관점 인수테스트 시나리오 수립 |
| 검사/테스트 수행방법, 절차, 참여조직 기술 | 인수테스트 계획서 상세 기술 |
| 발주자에게 승인검사 및 테스트 요청 | 최종 산출물 + 테스트 결과물 첨부 제출 |
| 만족한 결과 시까지 보완/테스트 반복 | 결함 발견 시 보완 후 재테스트 수행 |
| 14일 이내 검사/테스트 | 요청일로부터 14일 이내 수행, 기간 연장 가능 |

**가. 인수테스트 수행 계획**

| 항목 | 내용 |
|------|------|
| 수행 방법 | 시나리오 기반 검증 + 자유 탐색 테스트 병행 |
| 수행 절차 | 계획 수립 - 환경 준비 - 테스트 실행 - 결함 보완 - 재테스트 - 승인 |
| 참여 조직 | 중진공(검수 담당), 퀸텟시스템즈(PM), 아스트라비전(기술지원) |
| 점검 사항 | 기능 완전성, 성능 목표 달성, 보안 준수, 데이터 정합성 |
| 최종 검수 기준 | 치명결함 0건, 주요결함 해소율 100% |

**나. 인수테스트 절차**

| 단계 | 활동 | 담당 | 산출물 |
|------|------|------|--------|
| 1. 계획 | 인수테스트 계획 수립, 시나리오 작성 | 제안사 + 중진공 | 인수테스트 계획서 |
| 2. 환경 | 운영환경 동일 테스트 환경 구성 | 제안사 | 테스트 환경 구성서 |
| 3. 실행 | 시나리오 기반 테스트 수행 | 중진공(주도) + 제안사(지원) | 테스트 실행 결과 |
| 4. 보완 | 발견 결함 수정 및 보완 | 제안사 | 결함 조치 보고서 |
| 5. 재테스트 | 보완 사항 확인 테스트 | 중진공 | 재테스트 결과 |
| 6. 승인 | 최종 승인 및 인수 | 중진공 | 인수 확인서 |

**다. 인수테스트 보장 사항**

- 수요기관의 승인(인수) 검사 및 테스트 요청과 관련된 시나리오를 제시합니다.
- 개발 완료 후 최종 산출물 및 테스트 결과물을 첨부하여 수요기관에게 승인 검사 및 테스트를 요청하여야 합니다.
- 승인 검사 및 테스트 과정에서 발견된 하자사항은 만족한 결과를 얻을 때까지 보완/테스트를 반복적으로 실시합니다.
- 본 사업 구축 완료 후 승인 검사 및 테스트를 요청할 수 있으며, 발주자는 검사 및 테스트 요청일로부터 14일 이내에 검사/테스트하여야 합니다. 검사/테스트 과정에서 기일이 추가로 소요되는 경우 상호 합의에 의해서 기간 연장이 가능합니다.
- 수요기관은 현장에 시급하게 적용해야 할 프로그램에 대해서는 사업 중이라도 승인 검사 및 테스트를 실시할 수 있습니다.
