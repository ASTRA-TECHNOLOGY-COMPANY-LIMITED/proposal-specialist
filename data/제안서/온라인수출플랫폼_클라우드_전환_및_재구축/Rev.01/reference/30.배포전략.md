## 1. 배포 전략 개요

### 1.1 무중단 배포의 필요성

온라인수출플랫폼 고비즈코리아(GobizKorea)는 **5만여 개 국내 중소기업**과 **글로벌 바이어**가 실시간으로 거래하는 대한민국 대표 수출지원 플랫폼이다. 서비스 중단은 곧 수출 기회 상실로 직결되며, RFP(SFR-001, SFR-002)에서도 **무중단 배포**를 핵심 요구사항으로 명시하고 있다.

본 사업에서는 레거시 온프레미스 환경에서 클라우드 네이티브 Kubernetes 환경으로 전환하면서, 단순한 무중단 배포를 넘어 **배포 상황에 따라 최적 전략을 선택 적용하는 적응형 배포 체계**를 구축한다.

### 1.2 3대 배포 전략 체계

배포의 규모, 위험도, 롤백 요구 수준에 따라 **세 가지 전략을 모두 지원하는 플랫폼을 구축**하되, 각 배포 건마다 하나의 전략을 선택하여 실행한다.

| 배포 전략 | 적용 비율 | 핵심 가치 | 대상 |
|-----------|----------|----------|------|
| **Rolling Update** | ~85% | 안정성 + 효율성 | 일상적 배포, 패치, 설정 변경 |
| **Blue/Green** | ~10% | 즉시 전환 + 즉시 롤백 | 대규모 릴리스, DB 스키마 변경 |
| **Canary** | ~5% | 리스크 사전 검증 | 신규 기능, 성능 영향 불확실 변경 |

```
배포 유형 판단 기준
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  DB 스키마 변경 없음 + 하위호환 ──→ Rolling Update
  DB 스키마 변경 또는 즉시 롤백 필수 ──→ Blue/Green
  리스크 높음 + 단계적 검증 필요 ──→ Canary
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 2. 무중단 배포 전략 상세

### 2.1 Rolling Update — 기본 전략

Kubernetes의 기본 배포 전략으로, 기존 Pod를 점진적으로 새 버전 Pod로 교체한다. **서비스 중단 없이** 순차적으로 교체가 완료된다.

**동작 원리**

```
v1 Pod [■■■■]
v1 Pod [■■■ ] + v2 Pod [■]       ← 신규 Pod 1개 생성, Readiness 확인
v1 Pod [■■  ] + v2 Pod [■■]      ← 확인 완료 후 다음 Pod 교체
v1 Pod [■   ] + v2 Pod [■■■]     ← 순차 진행
v2 Pod [■■■■]                     ← 전체 교체 완료 (무중단)
```

**무중단 보장 핵심 메커니즘**

| 설정 항목 | 설정값 | 효과 |
|----------|-------|------|
| **maxUnavailable** | 0 | 교체 과정에서 서비스 가능 Pod가 줄어들지 않아 **무중단 보장** |
| **maxSurge** | 1 | 한 번에 1개씩만 추가하여 리소스 급증 없이 안전하게 교체 |
| **readinessProbe** | Health Check | 신규 Pod가 완전히 준비된 후에만 트래픽 라우팅 시작 |

**적용 기준**: DB 스키마 변경 없는 코드 변경, 보안 패치, 버그 수정, 설정 변경 등 하위 호환성이 보장되는 모든 배포

### 2.2 Blue/Green 배포 — 대규모 변경 대응

신규 버전(Green)을 완전히 준비한 상태에서 트래픽을 일괄 전환하는 방식이다. **전환 즉시 100% 트래픽이 신규 버전으로 라우팅**되며, 문제 발생 시 이전 버전(Blue)으로 **수 초 이내 즉시 롤백**이 가능하다.

**동작 원리**

```
┌─ 전환 전 ───────────────────────────────┐
│  [Blue v1 ■■■■] ← Service (현재 트래픽)  │
│  [Green v2 ■■■■]   (대기, 사전 검증 중)  │
└──────────────────────────────────────────┘
                    ↓ 검증 완료, 전환 실행
┌─ 전환 후 ───────────────────────────────┐
│  [Blue v1 ■■■■]   (대기, 롤백 대비)      │
│  [Green v2 ■■■■] ← Service (트래픽 전환) │
└──────────────────────────────────────────┘
```

**구현 방식**: Kubernetes Service의 라벨 셀렉터를 Blue에서 Green으로 변경하는 것만으로 트래픽을 일괄 전환한다. 두 버전이 동시에 존재하므로 전환과 롤백 모두 **서비스 다운타임 제로**로 수행된다.

> **장점**: 코드 배포나 재기동 없이 라우팅 설정 변경만으로 즉시 전환/롤백이 가능하여, 대규모 변경 시에도 사용자 영향을 원천 차단한다.

**적용 기준**
- DB 스키마 변경을 동반하는 대규모 릴리스
- 전자정부 프레임워크 버전 업그레이드
- 외부 연계 인터페이스 변경 (토스페이먼츠, 카페24, 아마존 API 등)
- 즉시 롤백이 반드시 필요한 배포

**롤백**: Service selector를 `version: blue`로 변경 → **수 초 이내 완전 복원**

### 2.3 Canary 배포 — 리스크 사전 검증

전체 트래픽의 일부(10%)만 신규 버전으로 먼저 라우팅하여 실운영 환경에서 검증한 후, 이상이 없을 경우 단계적으로 트래픽 비율을 확대한다. **리스크가 높은 변경의 영향 범위를 최소화**하는 가장 안전한 전략이다.

**단계적 확대 절차**

```
[1단계]  Stable v1 ← 90%    Canary v2 ← 10%    관찰: 30분
            │                    │
            ↓ 에러율 < 0.1%, 응답시간 정상 확인
            │
[2단계]  Stable v1 ← 70%    Canary v2 ← 30%    관찰: 1시간
            │                    │
            ↓ 에러율 유지, 리소스 사용량 정상
            │
[3단계]  Stable v1 ← 50%    Canary v2 ← 50%    관찰: 2시간
            │                    │
            ↓ 전체 지표 안정 확인
            │
[4단계]  Canary v2 ← 100%                       전환 완료
```

| 단계 | Canary 비율 | 관찰 시간 | 승격 판단 기준 |
|------|-----------|---------|-------------|
| 1단계 | 10% | 30분 | HTTP 에러율 < 0.1%, P99 응답시간 정상 범위 |
| 2단계 | 30% | 1시간 | 에러율 유지, CPU/메모리 사용량 정상 |
| 3단계 | 50% | 2시간 | 전체 모니터링 지표 안정 |
| 4단계 | 100% | — | 전체 전환 완료, Stable Deployment 제거 |

**구현 방식**: Nginx Ingress Controller의 Canary 어노테이션으로 트래픽 비율을 제어한다. 별도의 Canary용 Ingress를 생성하고 가중치(weight)를 10% → 30% → 50% → 100%로 단계적으로 조정하여 트래픽을 점진적으로 전환한다.

> **장점**: 전체 사용자의 일부에게만 먼저 노출하여 실운영 환경에서 검증하므로, 문제 발생 시 영향 범위를 최소 10%로 한정할 수 있다.

**적용 기준**
- 검색엔진 고도화 등 성능 영향이 불확실한 변경
- 신규 결제 연동 (토스페이먼츠) 최초 적용
- 대규모 UI/UX 재설계 반영

## 3. 서비스별 배포 전략 매핑

온라인수출플랫폼의 5개 서비스(Namespace)별로 서비스 특성과 사용자 영향도를 고려하여 **권장 기본 전략과 상황별 전략 적용 기준**을 정의한다.

| 서비스 (Namespace) | 사용자 유형 | 기본 전략 | Blue/Green 적용 시나리오 | Canary 적용 시나리오 |
|---|---|---|---|---|
| **국문사이트** (`gobiz-kr`) | 국내 중소기업 | Rolling Update | 프레임워크 업그레이드, 대규모 릴리스 | 검색엔진 변경, UI 전면 개편 |
| **영문사이트** (`gobiz-en`) | 글로벌 바이어 | Rolling Update | 결제 연동 변경, CDN 설정 변경 | 토스페이먼츠 최초 적용, 아마존/카페24 연동 |
| **사업관리서비스** (`gobiz-admin`) | 내부 관리자 | Rolling Update | 권한체계 변경, Nexacro 버전 업 | — (내부 사용자, 불필요) |
| **통합정보시스템** (`gobiz-export`) | 내부 사용자 | Rolling Update | Oracle 연계 변경, SSO 변경 | — (내부 사용자) |
| **외부연계 API GW** | 외부 시스템 | Rolling Update | API 버전 변경, 인증 체계 변경 | 신규 외부 API 연동 최초 적용 |

> **설계 원칙**: 외부 사용자 대면 서비스(국문/영문)는 Canary 전략을 활용하여 리스크를 최소화하고, 내부 사용자 서비스(관리자/통합정보)는 Blue/Green으로 빠른 전환과 롤백을 보장한다.

## 4. 롤백 전략 및 자동 복구

### 4.1 전략별 롤백 메커니즘

배포 후 이상이 감지되면, 배포 전략에 따라 가장 빠른 방법으로 이전 버전으로 복원한다.

| 배포 전략 | 롤백 방식 | 소요 시간 | 장점 |
|---------|---------|---------|------|
| Rolling Update | 이전 버전으로 순차 재배포 | 1~3분 | Kubernetes 내장 기능으로 별도 도구 없이 실행 |
| Blue/Green | Service 라우팅 대상 전환 | **수 초** | 이전 버전이 대기 중이므로 즉시 복원 가능 |
| Canary | Canary 인스턴스 즉시 제거 | **수 초** | 영향 범위가 제한적이어서 전체 서비스에 영향 없음 |

### 4.2 자동 롤백 조건

CI/CD 파이프라인에서 배포 후 모니터링 지표를 자동으로 수집하고, 아래 조건 중 하나라도 충족되면 **사람의 개입 없이 즉시 자동 롤백**을 실행한다.

```
자동 롤백 트리거 조건
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ① HTTP 5xx 에러율이 배포 전 대비 2배 이상 증가
  ② P99 응답시간이 임계값(3초) 초과
  ③ Health Check 연속 실패 3회 이상
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> **핵심 가치**: 장애 발생 시 대응 시간을 사람의 판단 시간(분~시간)에서 시스템의 자동 감지 시간(**초 단위**)으로 단축하여, 사용자 영향을 최소화한다.

## 5. CI/CD 파이프라인 아키텍처

### 5.1 전체 파이프라인 흐름

소스 코드 커밋부터 운영 배포까지 **완전 자동화된 파이프라인**을 구축한다.

```
┌──────────────────────────────────────────────────────────────┐
│                     CI/CD 파이프라인                            │
│                                                              │
│  [소스 커밋]  →  [빌드·검증]  →  [이미지 저장]  →  [GitOps 배포] │
│   GitLab        GitLab CI/CD      Harbor          Argo CD    │
│                 ┌─────────────────────┐                      │
│                 │ · 정적 분석 (SonarQube) │                   │
│                 │ · 단위 테스트           │                   │
│                 │ · 보안 스캔            │                   │
│                 │ · 이미지 빌드          │                   │
│                 └─────────────────────┘                      │
│                                                              │
│  [배포 전략 선택]  →  [K8s 배포]  →  [모니터링]                  │
│  파이프라인 파라미터    Argo CD      Prometheus                  │
│  ┌──────┬──────┬──────┐                                      │
│  │Rolling│B/G  │Canary│                                      │
│  │(기본) │(선택)│(선택) │                                      │
│  └──────┴──────┴──────┘        │                             │
│                         [자동 롤백 판단]                        │
│                          ├─ 정상 → 배포 완료                    │
│                          └─ 이상 → 자동 롤백                    │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 단계별 상세

| 단계 | 도구 | 역할 | 세부 내용 |
|------|------|------|---------|
| 소스 관리 + CI | **GitLab CE** | 버전 관리, 빌드, 테스트 | 브랜치 전략(feature → staging → main), MR 기반 코드 리뷰, `.gitlab-ci.yml` 기반 CI 파이프라인 통합 |
| 코드 품질 검증 | **GitLab CI/CD** | 정적 분석, 보안 스캔 | SonarQube 정적 분석, 단위 테스트, 보안 스캔을 GitLab Pipeline에서 일괄 수행 |
| 이미지 빌드 | **GitLab CI/CD + Docker** | 컨테이너 이미지 생성 | Dockerfile 기반 멀티스테이지 빌드로 최적화된 이미지 생성 |
| 이미지 저장 | **Harbor** | 이미지 관리, 보안 | HTTPS 암호화, 취약점 스캐닝, RBAC 접근 제어, 개발/운영 분리 |
| 배포 | **Argo CD** | GitOps 배포 | Git 매니페스트와 클러스터 상태 동기화, Helm/Kustomize 지원 |

> **GitLab 통합의 장점**: 소스 관리와 CI/CD를 단일 플랫폼에서 운영하여 **별도 CI 서버(Jenkins 등) 운영 부담을 제거**하고, MR(Merge Request) 단위로 빌드·테스트·배포 상태를 통합 추적할 수 있다.

### 5.3 배포 전략 파라미터화

파이프라인에서 배포 전략을 **파라미터로 선택**할 수 있도록 구성하여, 배포 담당자가 상황에 맞는 전략을 직접 지정한다.

| 파라미터 | 선택 항목 | 설명 |
|---------|----------|------|
| **배포 전략** | Rolling(기본) / Blue-Green / Canary | 변경 규모와 리스크에 따라 선택 |
| **대상 서비스** | gobiz-kr / gobiz-en / gobiz-admin / gobiz-export | 배포 대상 네임스페이스 지정 |
| **Canary 비율** | 10%(기본) ~ 100% | Canary 배포 시 초기 트래픽 비율 설정 |

## 6. 브랜치 전략 및 환경 분리

### 6.1 브랜치 구성

브랜치와 배포 환경을 **1:1로 매핑**하는 단순하고 명확한 브랜치 전략을 적용한다.

```
feature/* ──→ staging ──→ main
   │              │          │
  개발 작업     개발팀 PR     사업팀 검증 후
              + Code Review   수동 머지
              + Auto Merge    → 운영 배포
              → staging 배포    (Argo CD 자동 Sync)
                (Argo CD
                 자동 Sync)
```

| 브랜치 | 역할 | 배포 대상 | 병합 방식 |
|--------|------|----------|----------|
| `feature/*` | 기능 개발. PR을 통해 staging에 병합 | — | 개발팀 PR → Code Review → **Auto Merge** |
| `staging` | CI 빌드·테스트 수행, Harbor에 이미지 push, **staging 네임스페이스에 자동 배포** | staging namespace | PR 승인 시 자동 병합 |
| `main` | **운영 네임스페이스에 배포** | production namespace | 사업팀 staging 검증 후 **수동 머지** |

### 6.2 환경 분리 — 단일 클러스터 네임스페이스 방식

하나의 Kubernetes 클러스터에서 **네임스페이스로 스테이징과 운영 환경을 논리적으로 분리**한다. 별도 클러스터 운영 대비 인프라 비용을 절감하면서도 충분한 격리 수준을 확보한다.

```
Kubernetes 클러스터 (1대)
├── namespace: staging      ← 검증용 (staging 브랜치 auto merge 시 자동 배포)
└── namespace: production   ← 운영용 (main 브랜치 수동 머지 시 자동 배포)
```

### 6.3 배포 승인 흐름

```
[개발팀 영역]
  feature/* → staging PR 생성
  → 개발팀 Code Review
  → Review 승인 → Auto Merge
  → Argo CD가 staging 네임스페이스에 자동 배포

[사업팀 영역]
  → 사업팀이 staging 환경에서 기능·품질 검증
  → 검증 완료 확인
  → staging → main 수동 머지 실행
  → Argo CD가 production 네임스페이스에 자동 배포
```

| 단계 | 주체 | 브랜치 | 환경 | 방식 |
|------|------|--------|------|------|
| 1단계 | **개발팀** | feature → `staging` | — | PR 생성 → Code Review → **Auto Merge** |
| 2단계 | **Argo CD** | `staging` | staging namespace | 병합 감지 → **자동 배포** |
| 3단계 | **사업팀** | — | staging 환경 | 기능·품질 테스트 및 검증 |
| 4단계 | **사업팀** | staging → `main` | — | 검증 완료 후 **수동 머지** |
| 5단계 | **Argo CD** | `main` | production namespace | 병합 감지 → **자동 배포** |

### 6.4 GitOps 매니페스트 관리

배포 매니페스트를 별도 Git 저장소에서 관리하여, 배포 이력을 Git 커밋으로 완전히 추적한다.

```
deploy-repo/
├── staging/
│   └── values.yaml       ← Auto Sync (staging 브랜치 병합 시 자동 반영)
└── production/
    └── values.yaml       ← Auto Sync (main 브랜치 병합 시 자동 반영)
```

> staging과 production은 동일한 이미지 태그를 사용하되, 환경별 설정(리소스 제한, 레플리카 수, 외부 연동 엔드포인트)은 `values.yaml`에서 분리 관리한다. **운영 배포의 게이트키핑은 Argo CD Sync 정책이 아닌 staging → main MR 승인 프로세스에서 수행**한다.

## 8. 도구 스택 및 라이선스 전략

### 8.1 전체 도구 체인

```
GitLab CE (SCM + CI)  →  Harbor (Registry)  →  Argo CD (CD/GitOps)
```

### 8.2 구성요소별 역할 및 라이선스

| 구성요소 | 역할 | 라이선스 | 비용 | 비고 |
|---------|------|---------|------|------|
| **GitLab CE** | 소스 관리 + CI/CD 통합 (빌드·테스트·이미지 생성) | MIT License | **무료** | SCM과 CI를 단일 플랫폼으로 통합 |
| **Harbor** | 컨테이너 이미지 저장, 취약점 스캔 | Apache 2.0 | **무료** | CNCF Graduated 프로젝트 |
| **Argo CD** | GitOps 배포, K8s 매니페스트 동기화 | Apache 2.0 | **무료** | CNCF Graduated 프로젝트 |

> **비용 효율성**: GitLab CE가 소스 관리와 CI/CD를 단일 플랫폼으로 통합하여 **별도 CI 서버 없이** 전체 파이프라인을 **100% 오픈소스 무료**로 구성한다. 도구 수를 최소화하여 운영 복잡도를 낮추면서도, CNCF Graduated 프로젝트(Harbor, Argo CD) 채택으로 **엔터프라이즈급 안정성**을 확보한다.

## 종합

본 배포 전략은 다음 세 가지 핵심 가치를 실현한다.

**첫째, 무중단 서비스 보장.** Rolling Update의 `maxUnavailable: 0` 설정과 `readinessProbe` 연동으로 일상 배포의 100%를 서비스 중단 없이 수행하며, Blue/Green과 Canary 전략으로 대규모 변경 시에도 서비스 연속성을 보장한다.

**둘째, 리스크 최소화.** 배포 규모와 위험도에 따라 3대 전략을 선택 적용하고, Prometheus 기반 자동 롤백으로 장애 전파를 **초 단위**로 차단한다. 특히 Canary 배포는 전체 사용자의 10%만 먼저 노출하여 영향 범위를 사전에 통제한다.

**셋째, 비용 효율성.** 100% 오픈소스(GitLab CE + Harbor + Argo CD) 3개 도구만으로 전체 CI/CD를 구성하여 도구 운영 부담과 라이선스 비용을 동시에 제거하면서도, CNCF 표준 도구 채택으로 기술 안정성과 생태계 지원을 확보한다.
