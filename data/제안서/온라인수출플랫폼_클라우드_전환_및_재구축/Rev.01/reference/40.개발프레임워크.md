### 2.4 표준프레임워크 적용

#### 2.4.1 구현 대상 어플리케이션 개요

본 사업의 구현 대상 어플리케이션은 **5개 핵심 영역**으로 구분되며, 각 영역별 구현 대상 내역은 아래와 같다.

| No. | 구현 영역 | 요구사항 번호 | 기술 스택 | 운영 위치 |
|-----|--------|----------|--------|--------|
| 1 | 국문사이트 (고비즈코리아) | SFR-004, SFR-006, SFR-007 | Nginx + Tomcat + MariaDB, 전자정부 표준프레임워크 5.0 | 민간클라우드 |
| 2 | 영문사이트 (고비즈코리아) | SFR-005, SFR-006, SFR-007 | Nginx + Tomcat + MariaDB, CDN 이미지 캐쉬서버 | 민간클라우드 |
| 3 | 행정 및 사업관리서비스 | SFR-008, SFR-009, SFR-010 | Nginx + Nexacro + Tomcat + MariaDB, 전자정부 표준프레임워크 5.0 | 민간클라우드 |
| 4 | 통합정보시스템(수출) | SFR-012, SFR-013 | Nginx + Nexacro + Tomcat + Oracle(기존활용) | 중진공IDC |
| 5 | 외부연계 | SFR-003, SFR-011 | API 게이트웨이, 표준 연계모듈 | 민간클라우드 |

전체 5개 영역의 **백엔드**는 **전자정부 표준프레임워크 5.0(Spring Boot 3.x 기반)**으로 통일하여 개발 일관성과 유지보수성을 확보하고, **프론트엔드**는 영역별 특성에 따라 **이원화 전략**을 적용한다(COR-001, PMR-003).

#### 2.4.2 프론트엔드 이원화 전략

본 사업은 구현 영역의 사용자 특성과 UI 요구사항에 따라 프론트엔드를 **React(Next.js) 기반**과 **Nexacro 기반** 두 가지로 이원화한다. 이를 통해 각 영역에 최적화된 사용자 경험을 제공하면서도, 백엔드의 전자정부 표준프레임워크 5.0 장점을 전 영역에서 동일하게 활용한다.

![전자정부 표준프레임워크 5.0 기반 프론트엔드 이원화 구조도](../images/06_egovframe_50_architecture.png)

##### (1) React(Next.js) 기반 — 국문사이트, 영문사이트, 외부연계

| 구분 | 내용 |
|------|------|
| **적용 영역** | ① 국문사이트(고비즈코리아), ② 영문사이트(고비즈코리아), ⑤ 외부연계 API |
| **백엔드** | 전자정부 표준프레임워크 5.0 기반 **RESTful API** 서버 (Spring Boot 3.x, Spring MVC, MyBatis/JPA) |
| **프론트엔드** | **React 18 + Next.js 14** 기반 SSR/SSG 하이브리드 렌더링 |
| **통신 방식** | Next.js → Nginx Reverse Proxy → Tomcat(전자정부 FW 5.0) REST API 호출 |
| **인증/보안** | Spring Security 기반 JWT/Session 인증, CORS 정책 관리, CSRF 토큰 처리 |

**React(Next.js) 프론트엔드 적용 근거**:

| 장점 항목 | 상세 내용 |
|---------|---------|
| **반응형 웹 대응** | React 컴포넌트 기반 반응형 설계로 PC, 태블릿, 모바일 환경을 단일 코드베이스로 지원하여 '디지털 정부서비스 UI/UX 가이드라인'의 멀티플랫폼 요구사항을 효과적으로 충족한다 |
| **디자인 변경 유연성** | 프론트엔드와 백엔드가 API 기반으로 완전히 분리되어, 비즈니스 로직 수정 없이 UI/UX 디자인을 독립적으로 변경·고도화할 수 있다. 향후 고비즈코리아 디자인 리뉴얼 시 프론트엔드만 수정하면 되므로 유지보수 비용이 대폭 절감된다 |
| **SEO 및 글로벌 성능** | Next.js의 SSR(Server-Side Rendering)과 SSG(Static Site Generation)를 활용하여 영문사이트의 검색엔진 최적화(SEO)를 극대화하고, CDN 이미지 캐쉬서버와 결합하여 미국, 캐나다 등 원거리 해외바이어의 접속 지연을 최소화한다 |
| **개발 생산성** | React 생태계의 풍부한 UI 컴포넌트 라이브러리(Ant Design, Material UI 등)를 활용하여 200M+ 상품, 14개 대분류 카테고리, 6개 지역별 탐색 등 복잡한 UI를 신속하게 구현한다 |
| **전자정부 FW 장점 유지** | 백엔드는 전자정부 표준프레임워크 5.0의 공통컴포넌트(회원관리, 게시판, 보안 등)를 그대로 활용하여 개발 생산성을 확보하면서, 프론트엔드만 현대적 프레임워크로 전환한다 |

**API 통신 아키텍처**:

```
[사용자 브라우저]
    │ HTTPS
    ▼
[Cloud LB + WAF + CDN]
    │
    ▼
[Nginx (WEB Worker Pod)]
    ├── /api/* → Reverse Proxy → [Tomcat (App Worker Pod)]
    │                              전자정부 FW 5.0 REST API
    │                              Spring Boot 3.x + MyBatis/JPA
    │                              MariaDB / Oracle 연동
    │
    └── /* → [Next.js (WEB Worker Pod)]
              React 18 + Next.js 14
              SSR/SSG 하이브리드 렌더링
              fetch('/api/...') → Nginx Proxy → Tomcat API
```

##### (2) Nexacro 기반 — 행정 및 사업관리서비스, 통합정보시스템

| 구분 | 내용 |
|------|------|
| **적용 영역** | ③ 행정 및 사업관리서비스, ④ 통합정보시스템(수출) |
| **백엔드** | 전자정부 표준프레임워크 5.0 기반 비즈니스 로직 (Spring Boot 3.x, Spring MVC) |
| **프론트엔드** | **Nexacro N V24** 최신 버전 기반 JSP 연동, XML/JSON 데이터 바인딩 |
| **통신 방식** | Nexacro N V24 → Nexacro 전용 Controller(JSP) → Spring Boot 서비스 레이어 |
| **인증/보안** | Spring Security 기반 세션 인증, 중진공 업무망 이중방화벽 경유 접근 |

**Nexacro 프론트엔드 적용 근거**:

| 장점 항목 | 상세 내용 |
|---------|---------|
| **대량 데이터 그리드 처리** | 행정시스템의 수출사업 접수/평가/정산, 기업평가, 회원관리 등은 대규모 데이터 테이블 조회·편집이 핵심이며, Nexacro N V24의 고성능 그리드 컴포넌트가 이에 최적화되어 있다 |
| **업무 변경 신속 대응** | 수출지원사업의 정책 변경, 평가기준 수정, 신규 사업 추가 등 업무 변경이 빈번한 행정시스템 특성을 고려하여, Nexacro N V24의 **화면 단위 모듈화 설계**와 **메타데이터 기반 동적 화면 생성** 구조를 적용한다. 이를 통해 업무 변경 시 해당 화면 모듈만 수정·배포하면 되므로 기능 변경 및 추가에 빠르게 대응할 수 있다 |
| **업무 양식 표준화** | 중진공 내부 직원이 사용하는 업무 화면은 Nexacro의 양식 기반 UI가 업무 효율성과 정확성을 보장하며, 공통 UI 템플릿(목록조회, 상세보기, 등록/수정 양식)을 사전 정의하여 신규 업무 화면 개발 시 빠르게 조합·적용할 수 있는 구조로 개발한다 |
| **Nexacro N V24 최신 적용** | RFP 지정 SW인 Nexacro N V24의 최신 기능(HTML5 완전 호환, Web Component 지원, 개선된 데이터 바인딩)을 적극 활용하여 레거시 X-Platform 대비 성능과 호환성을 대폭 향상시킨다 |
| **전자정부 FW 연동** | Nexacro N V24 전용 Controller를 JSP 기반으로 분리 구현하고, XML/JSON 데이터 변환 레이어를 통해 전자정부 표준프레임워크 5.0의 서비스 레이어와 원활하게 연동한다. 백엔드 서비스 레이어를 공통 API 규격으로 설계하여, 업무 로직 변경 시에도 Nexacro 화면과 백엔드 간 영향도를 최소화한다 |

#### 2.4.5 Kubernetes 환경에서의 프론트엔드 이원화 배치

시스템구성도의 K8s 클러스터 구성과 연계하여, React(Next.js)와 Nexacro 프론트엔드를 아래와 같이 배치한다.

| K8s 영역 | Pod 구성 | 프론트엔드 유형 | 백엔드 연동 |
|---------|---------|-----------|---------|
