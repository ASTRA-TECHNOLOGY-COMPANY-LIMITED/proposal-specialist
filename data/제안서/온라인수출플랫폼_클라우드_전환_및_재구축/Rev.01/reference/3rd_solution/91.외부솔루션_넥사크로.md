## 1. 개요

### 1.1 넥사크로란?

넥사크로(Nexacro)는 **투비소프트(TOBESOFT Co., Ltd.)**가 개발한 **기업용 UI/UX 애플리케이션 개발 플랫폼**이다. 하나의 소스코드로 웹, 모바일 앱, 데스크톱 등 다양한 환경에서 동일한 UI/UX를 구현할 수 있는 **One Source Multi Use(OSMU)** 개발 환경을 제공한다.

국내 공공기관 및 대기업 SI 프로젝트에서 가장 널리 사용되는 **RIA(Rich Internet Application) 개발 플랫폼**으로, 전자정부 표준프레임워크와의 호환성을 갖추고 있다.

### 1.2 투비소프트 회사 개요

| 항목 | 내용 |
|------|------|
| 설립 | 2000년 7월 (서울) |
| 대표 제품 | 넥사크로 플랫폼, XPLATFORM, MiPlatform |
| 고객 수 | 2,600+ 기업 고객 |
| 프로젝트 실적 | 7,000+ 디지털 전환 프로젝트 완수 |
| 시장 위치 | 국내 UI/UX 개발 플랫폼 시장 점유율 1위 |
| 상장 여부 | 코스닥 상장 (A079970) |
| 특허 | 국내 44건, 해외 40건 |
| 수상 | IR52 장영실상 (대통령상) |

## 2. 제품 라인업 및 버전 이력

### 2.1 제품 진화 과정

```
MiPlatform → XPLATFORM → 넥사크로 14 → 넥사크로 17 → 넥사크로 N (v21~) → 넥사크로 N v24
```

| 버전 | 출시 시기 | 주요 특징 |
|------|----------|----------|
| MiPlatform | 초기 | X-Internet 기반, 유/무선 통합 |
| XPLATFORM | 중기 | RIA 기반, 풍부한 UI 컴포넌트 |
| Nexacro 14 | 2013년경 | HTML5 지원 시작 |
| Nexacro 17 | 2018년경 | QuickCode(로코드) 도입, 웹 표준 강화 |
| Nexacro N (v21) | 2021.09 | 완전한 웹 표준 기반, 자체 JS 프레임워크 |
| Nexacro N v24 | 2023.11 | 모바일 퍼스트, 가변 그리드 레이아웃 |

### 2.2 최신 버전: 넥사크로 N v24

넥사크로 N v24는 **모바일 퍼스트**를 핵심 콘셉트로 출시되었으며 주요 특징은 다음과 같다.

- **가변 그리드 레이아웃(Fluid Layout)**: 반응형 화면 설계 기능 강화
- **멀티스크린 대응**: 한 번 개발로 스마트폰, 태블릿, 노트북 등 모든 디바이스 대응
- **모바일 퍼스트 컴포넌트**: 모바일 환경에 최적화된 UX 제공
- **하위 호환성**: 기존 프로젝트 설정 변경 없이 다양한 버전과 호환

### 2.3 차세대 플랫폼: Neo-Pangaea

투비소프트는 2026년 중반 글로벌 출시를 목표로 **Neo-Pangaea** 플랫폼을 준비 중이다.
- AI 기반 SaaS형 로코드 DevOps 플랫폼
- 클라우드 네이티브 인프라
- CI 파이프라인 자동화
- 통합 보안 시스템
- 미국, 일본 시장 진출 예정

### 3.4 전자정부 표준프레임워크 연동

넥사크로 플랫폼은 **전자정부 표준프레임워크(eGovFrame)**와 공식 호환성을 갖추고 있어 공공기관 프로젝트에서 별도 인증 절차 없이 사용할 수 있다. 연동 가이드가 공식 제공된다.

### 3.5 서버 연동 구조 (JSP와의 관계)

넥사크로는 **클라이언트 측 UI 플랫폼**으로, 전통적인 JSP의 뷰(View) 역할을 **완전히 대체**한다. 넥사크로를 도입하면 JSP는 사용하지 않으며, 서버는 데이터 제공 역할만 수행한다.

#### 전통적 JSP 방식 vs 넥사크로 방식

```
전통적 JSP 방식:
  브라우저 → HTTP 요청 → [서버: JSP가 HTML 생성] → 브라우저에 HTML 렌더링

넥사크로 방식:
  브라우저 → [넥사크로 엔진이 UI 렌더링] → XML/JSON 트랜잭션 → [서버: 데이터만 제공]
```

| 구분 | JSP 기반 웹 | 넥사크로 기반 |
|------|-----------|------------|
| **화면 렌더링** | 서버에서 HTML 생성 (SSR) | 클라이언트에서 넥사크로 엔진이 렌더링 |
| **서버 역할** | 화면 + 데이터 모두 처리 | **데이터만 제공** (XML/JSON 응답) |
| **통신 방식** | form submit / AJAX | 넥사크로 `transaction()` 함수 |
| **JSP 필요 여부** | 필수 | **불필요** |
| **화면 갱신** | 페이지 전체 리로드 또는 부분 AJAX | Dataset 변경 시 바인딩된 UI 자동 갱신 |

#### 권장 백엔드 조합

넥사크로의 백엔드는 데이터를 XML/JSON으로 응답할 수 있는 모든 서버 환경이 가능하다.

| 백엔드 | 적합도 | 비고 |
|--------|:-----:|------|
| **Spring + 전자정부 표준프레임워크** | 최적 | 공공 SI 표준 조합. 공식 연동 가이드 및 UIAdapter 모듈 제공 |
| **Spring Boot (REST API)** | 최적 | 최신 프로젝트 권장. JSON 기반 통신에 적합 |
| **.NET (ASP.NET Web API)** | 적합 | 공식 지원 |
| **Node.js / Express** | 적합 | REST API로 통신 가능 |
| **Python (Django/Flask)** | 가능 | REST API 제공 시 연동 가능 |

> **참고**: 전자정부 표준프레임워크와 연동 시에도 JSP를 사용하는 것이 아니라, Spring Controller가 **XML/JSON 데이터를 반환**하고 넥사크로의 `transaction()` 함수가 이를 받아 Dataset에 바인딩하는 구조이다. 넥사크로 연동용 `UIAdapter` 모듈이 입출력 데이터 변환을 담당한다.

#### React/Next.js와의 구조적 유사성

넥사크로의 서버 연동 방식은 React/Next.js와 구조적으로 유사하다. 세 가지 모두 **프론트엔드에서 렌더링**하고, **서버는 API 역할만 수행**하는 클라이언트 렌더링 아키텍처를 따른다.

| 구분 | 넥사크로 | React | Next.js |
|------|---------|-------|---------|
| **렌더링 위치** | 클라이언트 (넥사크로 엔진) | 클라이언트 (Virtual DOM) | 서버+클라이언트 (하이브리드) |
| **서버 통신** | `transaction()` → XML/JSON | `fetch()` → REST/GraphQL | Server Actions / `fetch()` |
| **상태 관리** | Dataset 바인딩 | useState / Redux 등 | Server State + Client State |
| **JSP 필요** | 불필요 | 불필요 | 불필요 |

### 4.3 나라장터(조달) 구매

- **공공 SI 프로젝트**에서 빠른 개발이 필요할 때
- **대규모 업무 화면** (수백 개)을 단기간에 구축해야 할 때
- **OSMU** (웹+앱 동시 개발)가 필수 요건일 때
- 기존 넥사크로 시스템의 **마이그레이션/고도화** 프로젝트
- **Low-Code** 개발로 비전문 개발자도 참여해야 할 때

### 5.3 넥사크로 선택이 불리한 경우

- **라이선스 비용**이 프로젝트 예산에 부담이 될 때
- **글로벌 개발자** 투입이 필요한 프로젝트
- **오픈소스 생태계** 활용이 핵심인 프로젝트
- 독자 기술에 대한 **기술 종속(Vendor Lock-in)** 우려
- **SPA/마이크로프론트엔드** 등 최신 웹 아키텍처 적용 시

## 10. React / Next.js 통합 가능성 분석

### 10.1 분석 배경

클라우드 전환 및 재구축 사업에서 넥사크로와 React/Next.js의 통합 사용 가능성은 핵심 아키텍처 의사결정 사항이다. 기존 넥사크로 기반 업무 화면을 유지하면서 신규 화면은 React/Next.js로 개발하는 **하이브리드 전략**이 가능한지, 혹은 두 기술 스택이 하나의 시스템 내에서 공존할 수 있는지를 분석한다.

### 10.2 직접 통합의 기술적 한계

넥사크로와 React/Next.js는 **근본적으로 다른 아키텍처**를 가지고 있어 직접적인 통합은 불가능하다.

| 비교 항목 | 넥사크로 N | React / Next.js |
|----------|-----------|-----------------|
| **렌더링 엔진** | 자체 JavaScript 프레임워크 (넥사크로 엔진) | Virtual DOM 기반 React 렌더링 |
| **화면 정의** | XFDL/XML → JS 변환 (넥사크로 스튜디오) | JSX/TSX 컴포넌트 |
| **상태 관리** | Dataset 기반 데이터 바인딩 | useState/Redux/Zustand 등 |
| **컴포넌트 시스템** | 넥사크로 전용 컴포넌트 (80종+) | React 컴포넌트 생태계 |
| **빌드 체계** | 넥사크로 스튜디오 빌드 | Webpack/Vite/Turbopack |
| **서버 연동** | 전용 트랜잭션 프로토콜 (XML 기반) | REST API / GraphQL |
| **라우팅** | 넥사크로 내부 네비게이션 | React Router / Next.js App Router |

**핵심 제약사항**:
- 넥사크로는 **독자적인 JavaScript 프레임워크** 위에서 동작하며, React의 Virtual DOM과 컴포넌트 라이프사이클을 인식하지 못함
- React 컴포넌트를 넥사크로 화면 내부에 직접 마운트하거나, 넥사크로 컴포넌트를 React 앱 내에서 렌더링하는 것은 **기술적으로 불가능**
- 양 프레임워크가 동일한 DOM 트리를 제어할 경우 충돌 발생

### 10.3 가능한 통합 아키텍처 패턴

직접 통합은 불가능하지만, 다음과 같은 **간접 통합 패턴**을 통해 하이브리드 운영이 가능하다.

#### 패턴 A: WebBrowser 컴포넌트 기반 임베딩 (Iframe 방식)

넥사크로의 **WebBrowser 컴포넌트**를 활용하여 React/Next.js 앱을 넥사크로 화면 내에 임베드하는 방식이다.

```
┌──────────────────────────────────────────────┐
│            넥사크로 애플리케이션 (호스트)          │
│  ┌────────────────────────────────────────┐  │
│  │        넥사크로 메뉴/헤더/사이드바         │  │
│  ├────────────────────────────────────────┤  │
│  │  ┌──────────────────────────────────┐  │  │
│  │  │   WebBrowser 컴포넌트 (iframe)    │  │  │
│  │  │  ┌────────────────────────────┐  │  │  │
│  │  │  │   React / Next.js 앱       │  │  │  │
│  │  │  │   (독립 배포된 SPA/SSR)     │  │  │  │
│  │  │  └────────────────────────────┘  │  │  │
│  │  └──────────────────────────────────┘  │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

**통신 방식**:
- `WebBrowser.callMethod()`: 넥사크로 → React 앱 내 JavaScript 함수 호출
- `execScript()`: 넥사크로에서 iframe 내부 스크립트 실행
- `window.postMessage()`: 양방향 비동기 메시지 전달 (웹 표준)
- URL 파라미터: React 앱 라우팅을 통한 데이터 전달

| 항목 | 평가 |
|------|------|
| **구현 난이도** | 중간 - WebBrowser 컴포넌트는 공식 지원 기능 |
| **장점** | 기존 넥사크로 시스템 변경 최소화, React 앱 독립 배포 가능 |
| **단점** | iframe 특유의 UX 제약 (스크롤, 크기 조절, 성능), 양방향 통신 복잡 |
| **적합 사례** | 대시보드, 리포트, 데이터 시각화 등 독립적 화면 |

#### 패턴 B: API 게이트웨이 기반 분리 운영

넥사크로 앱과 React/Next.js 앱을 **완전히 분리**하되, 동일한 백엔드 API를 공유하는 방식이다.

```
┌─────────────────┐     ┌──────────────────┐
│   넥사크로 앱     │     │  React/Next.js 앱 │
│  (기존 업무 화면)  │     │  (신규 화면)       │
└────────┬────────┘     └────────┬─────────┘
         │                       │
         └───────────┬───────────┘
                     ▼
          ┌─────────────────────┐
          │   API Gateway       │
          │   (Kong / Nginx)    │
          ├─────────────────────┤
          │   ┌─────┐ ┌─────┐  │
          │   │ API │ │ API │  │
          │   │  A  │ │  B  │  │
          │   └─────┘ └─────┘  │
          └─────────────────────┘
                     │
          ┌─────────────────────┐
          │   공유 DB / 서비스    │
          └─────────────────────┘
```

**라우팅 전략**:
- 리버스 프록시(Nginx)로 URL 경로 기반 라우팅: `/legacy/*` → 넥사크로, `/new/*` → Next.js
- SSO/통합 인증으로 사용자 세션 공유

| 항목 | 평가 |
|------|------|
| **구현 난이도** | 낮음~중간 - 각 앱이 독립적이므로 간섭 없음 |
| **장점** | 기술 독립성 최대, 각 팀이 독립 개발/배포 가능, 장애 격리 |
| **단점** | 화면 간 전환 시 페이지 리로드, UX 일관성 유지 어려움 |
| **적합 사례** | 점진적 전환 전략, 대규모 시스템에서 신규 모듈 추가 |

#### 패턴 C: 마이크로 프론트엔드 (Module Federation)

Webpack Module Federation 또는 Single-SPA를 활용하여 넥사크로 화면과 React 화면을 **하나의 셸(Shell) 앱** 아래에서 운영하는 방식이다.

```
┌─────────────────────────────────────────────┐
│           Shell App (React / Next.js)        │
│  ┌────────────────────────────────────────┐  │
│  │         공통 헤더 / 네비게이션            │  │
│  ├──────────────────┬─────────────────────┤  │
│  │  Remote A        │  Remote B           │  │
│  │  (넥사크로 래핑)   │  (React 모듈)       │  │
│  │  - iframe 기반   │  - Module Federation│  │
│  │  - 기존 업무화면  │  - 신규 화면         │  │
│  └──────────────────┴─────────────────────┘  │
└─────────────────────────────────────────────┘
```

| 항목 | 평가 |
|------|------|
| **구현 난이도** | 높음 - 아키텍처 설계 및 인프라 구성 복잡 |
| **장점** | 통합된 UX, 점진적 마이그레이션 가능, 최신 아키텍처 |
| **단점** | 넥사크로 부분은 결국 iframe으로 래핑 필요, 초기 투자 비용 높음 |
| **적합 사례** | 장기적 전면 전환을 계획하되, 단계적으로 진행하는 경우 |

#### 패턴 D: 넥사크로 화면에서 외부 JS 라이브러리 활용

넥사크로 N은 HTML5 웹 표준 기반이므로, **제한적으로** 외부 JavaScript 라이브러리를 로딩하여 사용할 수 있다.

**가능한 범위**:
- Chart.js, D3.js 등 **데이터 시각화 라이브러리** (WebBrowser 컴포넌트 내)
- 유틸리티 라이브러리 (Lodash, Day.js 등)
- 외부 웹 에디터 (CKEditor, Toast UI Editor 등)

**불가능한 범위**:
- React/Next.js **프레임워크 자체**를 넥사크로 내부에서 구동
- React 컴포넌트를 넥사크로 컴포넌트로 래핑하여 사용
- Next.js의 SSR/ISR 기능을 넥사크로와 통합

### 10.4 Next.js 특화 통합 시나리오

Next.js는 React 기반이면서 SSR(Server-Side Rendering), ISR(Incremental Static Regeneration), App Router 등 서버 기능이 강화된 프레임워크로, 넥사크로와의 통합 시 추가적인 고려사항이 있다.

| 시나리오 | 설명 | 실현 가능성 |
|---------|------|-----------|
| Next.js를 넥사크로 WebBrowser에 임베드 | iframe으로 Next.js 페이지 로드 | **가능** - Next.js가 별도 서버에서 동작 |
| Next.js API Routes를 넥사크로 백엔드로 사용 | 넥사크로에서 Next.js API 엔드포인트 호출 | **가능** - REST API로 통신 |
| Next.js App Router 내에서 넥사크로 화면 호출 | Next.js 레이아웃 내에 넥사크로를 iframe으로 임베드 | **가능** - 역방향 임베딩 |
| Next.js RSC와 넥사크로 컴포넌트 혼용 | React Server Components 내에서 넥사크로 직접 사용 | **불가능** - 아키텍처 비호환 |
| Next.js Middleware로 넥사크로 인증 통합 | 통합 인증/세션 관리 | **가능** - 토큰 기반 SSO |

### 10.5 패턴별 비교 종합

| 비교 항목 | A. iframe 임베딩 | B. API 분리 운영 | C. 마이크로 프론트엔드 | D. JS 라이브러리 |
|----------|:---------------:|:---------------:|:-------------------:|:--------------:|
| **UX 통합도** | 중간 | 낮음 | 높음 | 높음 |
| **구현 복잡도** | 중간 | 낮음 | 높음 | 낮음 |
| **유지보수 부담** | 중간 | 낮음 | 높음 | 낮음 |
| **점진적 전환 지원** | 가능 | 매우 적합 | 매우 적합 | 제한적 |
| **성능** | iframe 오버헤드 | 페이지 전환 비용 | 최적화 가능 | 최적 |
| **인력 요구** | 양쪽 기술 필요 | 팀 분리 가능 | 고급 아키텍트 필요 | 넥사크로 중심 |

### 10.6 권장 전략

#### 단기 전략 (사업 수행 기간 내)

**패턴 B(API 분리 운영) + 패턴 A(iframe 임베딩) 혼합**을 권장한다.

1. **기존 업무 화면**: 넥사크로 N으로 유지/업그레이드
2. **신규 사용자 대면 화면**: Next.js(React)로 개발
3. **상호 연동이 필요한 화면**: WebBrowser 컴포넌트를 통한 iframe 임베딩
4. **백엔드**: API Gateway를 통해 두 프론트엔드가 동일한 서비스를 호출
5. **인증**: JWT 기반 SSO로 세션 통합

#### 중장기 전략 (3~5년)

**넥사크로 → React/Next.js 전면 전환**을 목표로 하되, 마이크로 프론트엔드 패턴을 활용하여 점진적으로 마이그레이션한다.

```
[1단계] 신규 화면은 Next.js로 개발, 기존 화면은 넥사크로 유지
         → API 분리 운영 (패턴 B)

[2단계] Shell App(Next.js) 도입, 넥사크로 화면을 iframe으로 통합
         → 마이크로 프론트엔드 전환 (패턴 C)

[3단계] 넥사크로 화면을 순차적으로 React로 재개발
         → 완전한 React/Next.js 전환
```

### 10.7 주요 리스크 및 대응

| 리스크 | 영향도 | 대응 방안 |
|-------|-------|----------|
| iframe 내 UX 불일치 | 중간 | 디자인 토큰 공유, CSS 변수 통일 |
| 이중 기술 스택 운영 부담 | 높음 | 팀 역할 분리, 기술별 담당자 지정 |
| 통합 인증 복잡성 | 중간 | OAuth 2.0 / JWT 기반 SSO 표준 적용 |
| 넥사크로 라이선스 이중 비용 | 높음 | 마이그레이션 로드맵에 따라 단계적 라이선스 축소 |
| 넥사크로-React 간 상태 동기화 | 중간 | postMessage 기반 이벤트 브릿지 구현, 공유 상태 최소화 |
