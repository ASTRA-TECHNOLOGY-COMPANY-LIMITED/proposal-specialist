## 1. 공공 마이데이터 서비스 개요

### 1.1 정의

공공 마이데이터란 **정보주체인 국민의 요구에 따라 행정/공공기관이 보유한 본인 행정정보를 본인 또는 제3자에게 제공하는 서비스**이다. 개인이나 기업이 제출해야 하는 민원서류 중 해당 업무에 필요한 정보만 선별하여 하나의 묶음정보로 만들어 이용기관에 전송하는 방식으로 운영된다.

### 1.2 법적 근거

| 법령 | 조항 | 내용 |
|------|------|------|
| 민원처리에 관한 법률 | 제10조의2 | 본인정보 제공 요구권 |
| 전자정부법 | 제43조의2 | 행정정보의 전자적 제공 |
| 본인에 관한 행정정보의 제공 등에 관한 고시 | 전문 (행정안전부 고시) | 보안저장소 규격, 운영 기준 등 세부사항 |

### 1.3 관할 기관

| 항목 | 내용 |
|------|------|
| 관할 부처 | 행정안전부 (공공 마이데이터), 개인정보보호위원회 (전분야 마이데이터), 금융위원회 (금융 마이데이터) |
| 서비스 종류 | 총 **167종** (2025.12 기준) |
| 서비스 분야 | 행정기관 93종, 공공기관 64종, 금융분야 7종, 통신분야 3종 |
| 이용기관 | 금융사 80곳 + 공공/민간 이용기관 |
| 이용지원기관 | 한국신용정보원 (보안저장소 위탁 운영, 기술 지원) |

## 2. 보안저장소(PDS) 기술 규격

### 2.1 정의 및 목적

보안저장소(PDS, Personal Data Store)란 **이용기관이 보유기관으로부터 본인정보를 제공받아 안전하게 보관하고, 이용 내역을 저장할 수 있는 데이터 저장소**이다.

행정안전부 「본인에 관한 행정정보의 제공 등에 관한 고시」 제27조에 의해, 이용기관이 정보주체의 동의 하에 발급받은 **묶음정보 및 이용내역**은 본인정보 보안저장소에 저장/관리하도록 **의무화**되어 있다.

**보안저장소의 3대 목적:**
1. **본인정보의 안전한 보존**: 보존기간 설정 및 무결성 검증 기능
2. **실시간 쓰기 보호**: 저장된 데이터의 변조/위조 방지
3. **안정성 확보**: 실시간 복제를 통한 이중화 구성 (데이터 유실 방지)

**보안저장소에 저장되는 정보:**

| 구분 | 내용 |
|------|------|
| 묶음정보 | 보유기관으로부터 전송받은 본인 행정정보 (필요한 항목만 선별한 패키지) |
| 이용내역 | 이용기관이 본인정보를 전자적으로 수신하여 업무 처리에 활용한 전자적 기록 |

### 2.2 고시 별표 규격 (핵심 요건)

「본인에 관한 행정정보의 제공 등에 관한 고시」 별표에 규정된 보안저장소의 핵심 기술 규격은 다음과 같다.

#### 2.2.1 보존기간 설정 및 무결성 검증

| 요건 | 상세 |
|------|------|
| 보존기간 설정 | 본인정보별 보존기간을 설정하고 기간 경과 시 자동 삭제 처리 |
| 무결성 검증 | 디지털 지문(SHA-256 등 해시 알고리즘)을 생성하여 데이터 변조 여부를 검증 |
| 검증 주기 | 정기적/수시 무결성 검증 수행 |

#### 2.2.2 실시간 쓰기 보호 (WORM)

| 요건 | 상세 |
|------|------|
| WORM 방식 | Write Once Read Many - 한 번 기록된 데이터는 수정/삭제 불가 |
| Append-Only | 기존 데이터의 덮어쓰기 차단, 추가 기록만 허용 |
| 특수삭제 | Privileged Delete - 보존기간 경과 또는 법적 요건 충족 시에만 관리자 권한으로 삭제 가능 |

#### 2.2.3 이중화 및 안정성

| 요건 | 상세 |
|------|------|
| 실시간 복제 | 블록 단위 실시간 데이터 복제 |
| HA 구성 | 고가용성(High Availability) 기반 이중화 |
| 재해복구 | 데이터 유실 방지를 위한 이중화 백업 체계 |

### 2.3 핵심 기술 요소

| 기술 요소 | 설명 | 적용 목적 |
|-----------|------|-----------|
| WORM Storage | Write Once Read Many 스토리지 | 데이터 변조/위조 원천 차단 |
| SHA-256 해시 | 디지털 지문 생성 알고리즘 | 무결성 검증 |
| Append-Only 로그 | 추가만 가능한 기록 방식 | 이력 위변조 방지 |
| Privileged Delete | 관리자 특수 삭제 기능 | 법적 요건에 따른 안전한 삭제 |
| 블록 단위 복제 | 실시간 블록 레벨 복제 | 데이터 안정성, HA 보장 |
| API 연동 | REST API 기반 업무시스템 연동 | 기존 시스템과의 통합 |

## 3. 보안 요구사항

### 3.1 접근통제

| 항목 | 요구사항 |
|------|----------|
| 접근권한 관리 | 최소 권한 원칙 적용, 역할 기반 접근통제(RBAC) |
| 인증 | 엄격한 본인인증 절차 (정보주체 본인확인) |
| 비인가 접근 차단 | 비인가자의 보안저장소 접근 원천 차단 |
| 관리자 접근 | 특수삭제 등 관리 기능은 별도 관리자 인증 필요 |

### 3.2 암호화

| 구간 | 요구사항 |
|------|----------|
| 전송 구간 | 보유기관 → 이용기관 간 본인정보 전송 시 암호화 |
| 저장 구간 | 보안저장소 내 본인정보 암호화 저장 |
| 무결성 보장 | SHA-256 등 해시 알고리즘을 활용한 디지털 지문 생성 |

### 3.3 감사 및 로그 관리

| 항목 | 요구사항 |
|------|----------|
| 접속기록 보관 | 보안저장소 접근/이용 이력 기록 및 보관 |
| 책임추적성 | 개인정보 오남용/유출 시 책임 추적 가능한 로그 체계 |
| 이용내역 관리 | 본인정보 수신 및 업무 활용 이력의 전자적 기록 |

### 3.4 데이터 생명주기 관리

```

 │        │        │        │        └─ 보존기간 경과 시
 │        │        │        │           Privileged Delete
 │        │        │        │
```

## 4. 보안저장소 구축 방식

### 4.1 구축 옵션

이용기관은 보안저장소를 다음 두 가지 방식 중 선택하여 운영할 수 있다.

| 구축 방식 | 설명 | 장점 | 단점 |
|-----------|------|------|------|
| **자체 구축** | 이용기관이 직접 보안저장소를 구축/운영 | 데이터 직접 통제, 커스터마이징 가능 | 초기 투자 비용, 전문 인력 필요 |
| **위탁 운영** | 이용지원기관(한국신용정보원 등)에 위탁 | 초기 비용 절감, 전문 운영 | 외부 의존도 증가 |

### 4.2 자체 구축 시 절차

```
1. 기관 내규 검토
   └─▶ 시스템 환경 구성 및 보안 현황 검토
       └─▶ 보안저장소 기능요건 확인
           └─▶ 보안저장소 규격 충족 여부 검증
               └─▶ 보안저장소 체크리스트 점검
                   └─▶ 보안저장소 실사 요청 및 승인
                       └─▶ 구축 완료 및 운영 개시
```

### 4.3 자체 구축 시 확인 사항

- **보안저장소 규격**: 고시 별표에 명시된 기술 규격 100% 충족
- **보안저장소 체크리스트**: 기능/보안/운영 요건 전수 점검
- **보안저장소 실사**: 이용지원기관의 현장 실사 및 승인

### 4.4 위탁 시 운영 구조

한국신용정보원이 이용지원기관으로서 보안저장소 위탁 운영을 지원하며, 이용기관은 임시 PDS를 통해 서비스를 개시한 후 자체 PDS로 데이터 마이그레이션이 가능하다.

## 5. 제공업체 분류 체계 및 상세

마이데이터 관련 제공업체는 다음 5개 카테고리로 분류된다.

> **참고**: 위탁 운영 시 초기 구축 비용은 크게 절감되나, 장기적으로 위탁 수수료가 발생하며 데이터 직접 통제가 제한됨.

### 7.5 권고 도입 방안

온라인수출플랫폼의 **클라우드 전환 및 재구축** 사업 특성을 고려할 때:

1. **단기 (서비스 개시)**: 한국신용정보원 **위탁 운영** 방식으로 빠르게 서비스 개시
2. **중기 (안정화 후)**: **클라우드 기반 자체 PDS 구축** (EzPDS 클라우드 버전 권장)
3. **장기 (확대)**: 기업 공공 마이데이터 연동 확대, API 중계 플랫폼(쿠콘 등) 활용

### 7.6 PDS 솔루션 선정 시 체크리스트

- [ ] 행안부 고시 보안저장소 규격 100% 충족 여부
- [ ] GS 인증 (1등급 이상) 보유 여부
- [ ] **클라우드 환경 지원** 여부 (CSP 호환성)
- [ ] 나라장터/디지털서비스몰 등록 여부 (공공 조달 편의성)
- [ ] API 연동 인터페이스 제공 여부
- [ ] 이중화(HA) 구성 지원 여부
- [ ] 임시 PDS → 자체 PDS 마이그레이션 지원 여부
- [ ] 기술지원 및 유지보수 체계

## 9. 요약

공공마이데이터 보안저장소(PDS)는 행정안전부 고시에 의해 이용기관에 **의무적으로 구축**해야 하는 보안 인프라로, **WORM 스토리지 기반의 데이터 변조 방지**, **SHA-256 무결성 검증**, **이중화 백업**, **특수삭제(Privileged Delete)** 등의 핵심 기술 요소를 갖추어야 한다.

시장은 **167종 서비스, 전분야 확대(2025.03), 금융 마이데이터 2.0(2025.06), 기업 마이데이터 확대(2026)** 등으로 급격히 성장 중이며, 관련 제공업체는 PDS 솔루션, API 중계 플랫폼, 서비스 사업자, 보안/인증, SI/컨설팅의 5개 카테고리로 분류된다.

**PDS 솔루션**은 올리브텍(SecuPDS)과 유큐브(EzPDS)가 GS 1등급 인증을 보유한 시장 주도 제품이며, 올리브텍 조달 등록 가격 기준 **SW 5,445만원 + HW 2,350만원(단일 노드)**, 이중화 구성 시 약 **1.56억원**이다. **API 중계 플랫폼**은 쿠콘(300여 API, 종량제)과 코스콤(금융분야 공식 중계기관)이 주도하고 있다.

온라인수출플랫폼 클라우드 전환 사업에서는 **위탁 운영으로 빠른 서비스 개시 후, 클라우드 기반 자체 PDS 구축(EzPDS 권장)으로 전환하는 단계적 접근**이 권장되며, 고시 규격 충족 및 보안 실사 승인까지의 절차를 사업 계획에 반영해야 한다.
