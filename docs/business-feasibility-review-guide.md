## 1. 사업성 검토 프레임워크

### 1.1 Bid/No-Bid 의사결정 프레임워크 개요

제안서 작성 전문가들이 사용하는 사업성 검토(Bid/No-Bid Decision)는 **"제한된 자원을 어디에 집중할 것인가"**를 결정하는 전략적 의사결정 프로세스이다. 제안서 작성에는 상당한 비용(인력, 시간, 기회비용)이 발생하므로, 실현 가능성을 신중히 판단하지 않으면 사업 지연, 위약금, 추가 자원 투입 등 예상치 못한 손실이 발생할 수 있다.

### 1.2 3대 국제 프레임워크

#### (1) Shipley Capture Planning 프레임워크

Shipley Associates가 개발한 체계로, 전 세계적으로 가장 널리 사용되는 제안/수주 관리 방법론이다.

**3단계 의사결정 게이트:**

| 게이트 | 명칭 | 시점 | 판단 내용 |
|--------|------|------|----------|
| Gate 1 | Pursuit Decision | 기회 식별 후 | 추적할 가치가 있는지 판단 |
| Gate 2 | Bid Decision | RFP 발행 후 | 실제 제안서를 작성할지 결정 |
| Gate 3 | Bid Validation | 제안서 완성 후 | 최종 제출 여부 확인 |

**Shipley의 핵심 원칙:**
- 고객 의사결정의 40~60%는 제안서 제출 **이전**에 이루어진다
- PWin(승률)은 추측이 아닌 지식에 기반해야 한다
- 재입찰(Recompete) PWin은 최대 70%를 넘지 않으며, 신규 사업은 40%를 넘기 어렵다
- 보수적 초기 평가가 현실적인 전략 개선을 강제한다

**Color Team Review 체계:**

| 단계 | 명칭 | 목적 |
|------|------|------|
| Blue Team | 전략 검토 | Capture Plan 및 전략 적절성 평가 |
| Black Hat | 경쟁사 분석 | 경쟁사의 예상 솔루션/전략 예측 |
| Pink Team | 준수성 검토 | RFP 요구사항 준수 및 Win 전략 반영 확인 |
| Red Team | 모의 평가 | 평가위원 관점에서 점수 예측 및 개선점 도출 |
| Green Team | 가격 검토 | 가격 적정성 및 수익성 확인 |
| Gold Team | 최종 승인 | Red/Green Team 반영 확인 후 제출 승인 |
| White Hat | 교훈 도출 | 사후 Lessons Learned 기록 |

#### (2) PMBOK 리스크 관리 6단계

1. 리스크 관리 계획수립 (Plan Risk Management)
2. 리스크 식별 (Identify Risks) — 브레인스토밍, 체크리스트, SWOT
3. 정성적 리스크 분석 (Qualitative) — 확률-영향 매트릭스
4. 정량적 리스크 분석 (Quantitative) — Monte Carlo, 민감도 분석
5. 리스크 대응 계획수립 (Plan Responses) — 회피/전가/완화/수용/활용/공유
6. 리스크 감시 및 통제 (Monitor and Control)

#### (3) ISO 31000 프로세스

```
맥락 수립 → 리스크 평가(식별→분석→평가) → 리스크 처리 → 모니터링/검토
```

### 1.3 Bid/No-Bid 의사결정의 5대 핵심 질문 (APMP 기준)

| 질문 영역 | 핵심 질문 |
|-----------|----------|
| Utility (유용성) | 이 기회가 회사 전략 방향과 일치하는가? |
| Desirability (매력도) | 이 기회를 적극 추구하고 싶은가? |
| Capability (역량) | 필요한 자원과 기술이 있는가? |
| Viability (실행가능성) | 수익성이 확보되는가? |
| Winnability (승률) | 현실적으로 이길 수 있는가? |

## 2. 평가 기준 및 체크리스트

### 2.1 7대 Bid/No-Bid 평가 기준 (가중 매트릭스)

| 기준 | 가중치 | 1점 (매우 불리) | 3점 (보통) | 5점 (매우 유리) |
|------|--------|---------------|-----------|---------------|
| 전략적 적합성 | 15% | 핵심 사업과 무관 | 관련은 있으나 부수적 | 핵심 전략 방향과 완전 일치 |
| 고객 관계 | 10% | 접촉 이력 없음 | 과거 수행 이력 있음 | 긴밀한 관계, 핵심 요구 파악 |
| 기술 역량 매칭 | 20% | 요구 기술 30% 미만 보유 | 70% 수준 보유 | 95% 이상 보유 + 차별화 기술 |
| 경쟁 환경 | 15% | 강력한 수임업체 존재 | 2~3개 동급 경쟁사 | 경쟁사 약함, 차별화 명확 |
| 자원 가용성 | 10% | 핵심 인력/장비 미확보 | 일부 충원 필요 | 즉시 투입 가능 |
| 재무 건전성 | 15% | 보증금 부담 과중, 손실 예상 | 적정 수익 예상 | 높은 수익성, 재무 부담 없음 |
| 리스크 수용도 | 15% | 극심/매우높음 리스크 다수 | 높음 리스크 일부 | 보통 이하 리스크만 |

**의사결정 기준:**

| 가중 평균 | 권고 | 설명 |
|----------|------|------|
| 4.0~5.0 | **적극 참여** | 최고 자원 투입, PWin 높음 |
| 3.0~3.9 | **조건부 참여** | 선행 조치(리스크 완화) 후 참여 |
| 2.0~2.9 | **재검토 필요** | 리스크 완화 가능성 재평가 후 판단 |
| 1.0~1.9 | **불참 권고** | 리스크 과다, 자원 낭비 가능성 |

### 2.2 즉시 불참(No-Go) 트리거

다음 중 **하나라도 해당하면** 전체 점수와 무관하게 즉시 No-Go:

- 필수 참가자격(면허, 인증, 실적) 미충족이며 보완 불가
- 납기/일정이 기술적으로 불가능
- 수용 불가한 상업적 조건 또는 리스크 배분
- 법적, 보안, 윤리적 제약 존재
- 핵심 인력 미확보 상태에서 확보 전망 없음
- 최선(Best) 시나리오에서도 손실 발생 (손실 확정)

### 2.3 실무 체크리스트 (15개 항목)

**자격 요건:**
- [ ] 참가자격 요건(자본금, 면허, 인증, 실적) 충족 여부
- [ ] 직접생산확인(중소기업자간 경쟁 시) 보유 여부
- [ ] 입찰보증금(입찰금액 5%) 납부 여력

**기술 역량:**
- [ ] RFP 요구기술 보유율 70% 이상
- [ ] 유사실적 3건 이상 보유
- [ ] 핵심 인력(PM, 기술책임자) 투입 가능

**재무/수익:**
- [ ] 예상 수익률 10% 이상
- [ ] 계약보증금(10%), 이행보증보험 납부 여력
- [ ] 선급금 없이도 초기 운영 가능한 자금 여력

**경쟁 환경:**
- [ ] 경쟁사 3개 이상 파악 완료
- [ ] 수임업체(incumbent) 유무 확인
- [ ] 차별화 포인트 3개 이상 도출

**리스크:**
- [ ] 극심(20~25점) 리스크 없음
- [ ] 지체상금 리스크 관리 가능
- [ ] 하도급/컨소시엄 파트너 확보

## 3. 공공조달 입찰 사업성 분석

### 3.1 예정가격 산정 메커니즘

**산정 절차:**

1. **기초금액** 산출: 공사원가 + 부가세
2. **복수예비가격** 15개 추출: 기초금액의 ±2% 범위에서 무작위 난수 생성
3. **입찰 참가자 추첨**: 15개 중 4개를 추첨으로 선정
4. **예정가격 결정**: 선정된 4개 가격의 산술 평균

**핵심 포인트:**
- 예정가격은 입찰 당일까지 비공개
- 복수예비가격 추첨 결과에 따라 예정가격이 달라짐
- 기초금액 대비 ±2% 범위 내에서 변동

### 3.2 투찰율 산정

**투찰율 = (투찰금액 / 예정가격) × 100**

**입찰 방식별 일반적 투찰율:**

| 입찰 방식 | 일반적 투찰율 | 전략 |
|----------|-------------|------|
| 적격심사 (최저가) | 87.745% 전후 | 낙찰하한가에 최대한 근접 |
| 협상에 의한 계약 | 90~95% | 기술 점수 극대화가 핵심 |
| 2단계 입찰 | 92~97% | 기술 적격 통과 우선 |

**사정률의 중요성:**
- 표면적 투찰율이 아닌 사정률(예정가격 대비 실제 낙찰가 비율)로 패턴 분석
- 특정 발주처, 특정 업체의 입찰 패턴을 사정률로 분석하면 숨겨진 패턴 발견 가능

### 3.3 경쟁사 분석

**분석 항목:**

| 분석 영역 | 확인 사항 |
|----------|----------|
| 수임업체 여부 | 기존 계약 수행사가 재입찰하는지 여부 (가장 강력한 경쟁자) |
| 기술 역량 | 경쟁사 보유 기술, 인증, 특허 |
| 유사 실적 | 경쟁사의 발주기관 납품 이력 |
| 가격 경쟁력 | 과거 투찰 패턴 분석 |
| 인적 네트워크 | 발주기관과의 관계 깊이 |

**경쟁 강도와 전략:**
- 참여 업체 수가 많을수록 낙찰가는 낮아지는 경향
- 경쟁이 치열할수록 더 공격적인 가격 필요
- AI 기반 입찰 분석 도구는 공고 규모, 공종, 발주처 등 다중 변수를 동시 처리하여 예측

### 3.4 낙찰 확률 예측

**핵심 분석 데이터:**

1. **낙찰가 분포**: 예정가격 대비 낙찰가의 범위와 패턴 (단순 평균이 아닌 분포 패턴)
2. **참여업체 수**: 경쟁 강도와 낙찰가의 상관관계
3. **발주기관 패턴**: 동일 기관의 반복 발주 특성
4. **공종별 특성**: 공사/용역/물품별 차이

**전통적 방식의 한계:**
- 평균값 의존: 모든 공고에 동일 비율 적용으로 차별성 부족
- 변동성 미반영: 시기, 경기, 발주처 차이를 반영하지 못함
- 경쟁강도 무시: 참여 업체 수에 따른 전략 조정 불가

## 4. 수익성 분석 방법

### 4.1 SW 개발 대가 산정 (기능점수 기반)

**기본 공식:**

```
SW 개발비 = 기능점수(FP) × FP 단가 × 보정계수 × (1 + 직접경비율) × (1 + 이윤율)
```

**2024년 기준 FP 단가:** 605,784원/FP (전년 대비 9.5% 인상)

**보정계수 체계:**

| 보정 항목 | 범위 | 기준 |
|----------|------|------|
| 규모 보정 | 0.65~1.28 | FP 규모별 (300 미만: 0.65, 5000 이상: 1.28) |
| 연계복잡성 | 0.88~1.12 | 연계 기관 수별 |
| 성능요구 | 0.91~1.09 | 응답시간 요구수준별 |
| 운영환경 호환성 | 0.94~1.19 | OS/브라우저 호환 요구수준별 |
| 보안성 | 별도 | 보안 요구 수준별 |

### 4.2 원가 구조 분석

**투입공수 방식 원가 구성:**

```
총 대가 = 직접인건비 + 제경비 + 기술료 + 직접경비
```

| 항목 | 산정 기준 | 비율/기준 |
|------|----------|----------|
| 직접인건비 | 투입 M/M × 등급별 단가 | SW기술자 평균임금 기준 |
| 제경비 | 직접인건비 × 제경비율 | 110~120% (SW산업 144~154% 적용 사례도 있음) |
| 기술료 | (직접인건비 + 제경비) × 기술료율 | 20~40% |
| 직접경비 | 실비 산정 | 여비, 장비 사용료, 특허 사용료 등 |

**이윤율:**
- 일반적으로 (직접인건비 + 제경비 + 기술료) × 이윤율
- 이윤율: 15~25% (최대 25%)

### 4.3 손익분기점(BEP) 분석

**기본 공식:**

```
BEP = 고정비 / (1 - 변동비율)
BEP = 고정비 / 공헌이익률
```

**SW 사업의 원가 구분:**

| 고정비 (매출과 무관) | 변동비 (매출에 비례) |
|---------------------|---------------------|
| 사무실 임대료 | 직접인건비 (프로젝트 투입) |
| 관리직 인건비 | 외주비 |
| 감가상각비 | 장비 구매/임대비 |
| 보험료 | 여비/교통비 |

### 4.4 수익성 시뮬레이션 체크포인트

| 시나리오 | 확인 사항 |
|---------|----------|
| **최선 (Best)** | 예정 인력으로 예정 기간 내 완수, 추가비용 없음 |
| **기대 (Expected)** | 10~15% 예비비 사용, 소규모 범위 변경 발생 |
| **최악 (Worst)** | 핵심인력 이탈, 대규모 범위 변경, 지체상금 발생 |

## 5. Shipley Bid/No-Bid 상세 방법론

### 5.1 Capture Planning 정의

Shipley는 Capture Planning을 **"기회를 식별하고, 환경을 평가하며, 승리 전략을 고안하고 실행하는 프로세스"**로 정의한다.

### 5.2 PWin (Probability of Win) 산출 모델

**PWin 공식:**

```
PWin = (A × w1) + (B × w2) + (C × w3) + ... + (H × w8)
```

- A~H: 평가 기준별 점수 (1~10점)
- w1~w8: 가중치 (합계 = 1.0)

**평가 기준 및 가중치 (예시):**

| 요소 | 가중치 | 설명 |
|------|--------|------|
| Customer Fit (고객 적합성) | 25~30% | 솔루션이 고객 니즈와 일치하는 정도 |
| Past Performance (유사 실적) | 20% | 유사 프로젝트 수행 이력 |
| Competitive Edge (경쟁 우위) | 15~25% | 경쟁사 대비 차별화 정도 |
| Customer Relationship (고객 관계) | 15~30% | 기존 관계 및 신뢰도 |
| Pricing (가격 경쟁력) | 10% | 가격의 적정성 |

**PWin 해석:**

| PWin 범위 | 판단 | 조치 |
|----------|------|------|
| 70%+ | 강한 포지션 | 적극 투자, 승리 전략 실행 |
| 50~70% | 양호한 기회 | ROI 평가 후 투자 결정 |
| 50% 미만 | 도전적 | 투자 재고, 전략 재수립 |

**Shipley의 보수적 벤치마크:**
- 재입찰(Recompete): PWin은 70%를 **절대 넘지 않음** (자기 기만 경향 존재)
- 신규 사업: PWin은 40%를 **넘기 어려움** (고객에게 선택지 존재)

### 5.3 PWin 산출 실전 예시

사이버보안 기업이 정부 계약을 평가하는 경우:

| 요소 | 가중치 | 점수(1~10) | 가중점수 |
|------|--------|-----------|---------|
| 고객 적합성 | 0.30 | 8 | 2.40 |
| 경쟁 우위 | 0.25 | 6 | 1.50 |
| 유사 실적 | 0.20 | 10 | 2.00 |
| 고객 관계 | 0.15 | 8 | 1.20 |
| 가격 경쟁력 | 0.10 | 5 | 0.50 |
| **합계** | **1.00** | - | **7.60 (76%)** |

→ 결과: 76%로 **강한 포지션** — 적극 참여 권고

### 5.4 9대 PWin 영향 요소 (Shipley)

1. **Incumbency (수임 여부)**: 기존 수행사인지 여부
2. **Relevant Experience (관련 경험)**: 유사 분야 경험 수준
3. **Past Performance (과거 성과)**: 성과 평가 기록
4. **Contract Size (계약 규모)**: 규모에 따른 역량 매칭
5. **Competitor Ratings (경쟁사 평가)**: 경쟁사 대비 상대 위치
6. **Customer Relationship (고객 관계)**: 고객과의 관계 깊이
7. **Technical Solution (기술 솔루션)**: 기술적 해결책 품질
8. **Management Solution (관리 방안)**: 관리 체계 적절성
9. **Cost Competitiveness (가격 경쟁력)**: 가격의 적정성

## 6. 실무 노하우 및 팁

### 6.1 RFP 분석 시 핵심 체크포인트

**1단계: 평가기준 우선 분석**
- 평가기준이 가장 중요한 이유: **평가는 평가표를 기준으로 하기 때문**
- 배점 15점 이상 항목: 반드시 별도 섹션, 최소 2~3페이지 분량
- 배점 5점 이하 항목: 간결하게 처리

**2단계: 숨겨진 요구사항 파악**
- "유사 프로젝트 경험" 배점이 높다면 → 안정적 수행을 원함
- "커뮤니케이션 계획" 항목이 있다면 → 과거 협력사와 소통 문제 경험
- "위험 관리 계획" 요구 → 이전 프로젝트에서 일정 지연/예산 초과 경험

**3단계: 빈 퍼즐(Gap) 식별**
- RFP가 요구하는데 기존 템플릿에 없는 항목 식별
- 빈 퍼즐이 많으면 기존 템플릿과 이번 사업의 갭이 큰 신호
- 일정 계획 시 충분한 시간 배분 필요

### 6.2 Win Theme 도출 방법

**Win Theme 정의:**
"우리가 왜 이 사업을 해야 하는가"에 대한 한 줄짜리 답. 평가위원이 제안서를 덮고 나서 머릿속에 남는 단 한 문장.

**도출 프로세스:**

1. **타이밍**: RFP 분석 직후, 목차 작성 전에 수립
2. **순서**: Win Theme → 스토리라인 → 목차 구성
3. **담당**: 사업 맥락을 이해하는 팀장급 의사결정

**Win Theme 작성 원칙:**
- 기술 스펙이 아닌 **감성적 공감**에서 출발
- "최신 AI 기술 적용" (X) → "지나가는 공간에서 기억이 머무는 공간으로" (O)
- One Liner로 압축하여 킥오프 미팅에서 전 팀원 공유 → 제안서 일관성 유지

**제안서 적용 방법:**

1. Executive Summary에 Win Theme 선언
2. 각 챕터 도입부에 고객 가치와 연결
3. 기술 스펙 전에 "이 기술이 고객의 어떤 가치를 실현하는가" 한 문단 배치
4. 차별화 항목에 더 많은 페이지 할당

### 6.3 정부지원사업/공공입찰 탈락 7대 원인

| 순번 | 탈락 원인 | 핵심 문제 | 해결 방안 |
|------|----------|----------|----------|
| 1 | 요약문이 평범/추상적 | "AI 기술 활용 솔루션" 수준 | "문제-해결-성과-진행상황 응축 1분 피칭"으로 재구성 |
| 2 | 숫자 해석 부재 | 표와 차트는 많지만 의미 설명 없음 | "동종 업계 평균 대비 2배 수준" 등 해석 추가 |
| 3 | 근거 없는 주장 | "기술력 최고 수준" | "기존 대비 정확도 27% 향상(내부 테스트)" 근거 제시 |
| 4 | 성과 데이터 부족 | 계획만 있고 실행 증거 없음 | 고객 피드백, 프로토타입, 유료 베타 결과 제시 |
| 5 | 전문가만 이해하는 기술 설명 | 심사위원 30~50%가 비전문가 | "고객 질문을 실시간 이해해 맞춤 답변 생성하는 AI" 식 설명 |
| 6 | 지원사업 목적과 부합성 미흡 | 공고 요건만 확인 | 정부 정책 키워드(AI 융합, ESG 등) 자연스럽게 포함 |
| 7 | 스토리텔링 부재 | 순서 없이 섞인 내용 | "문제정의→Pain Point→해결책→실행→기대효과" 구조화 |

### 6.4 평가위원 관점에서의 사업성

- 심사위원이 제안서를 읽는 시간은 극히 제한적 (수십 개 제안서를 검토)
- 핵심 메시지가 명확한 제안서가 높은 점수를 받음
- **"중학생도 이해할 수 있도록"** 쉽게 설명하는 전략 필수
- 정부출연기관, 투자자, 산업평가위원 등 다양한 배경의 심사위원 존재

## 7. 정량적 평가 모델

### 7.1 Go/No-Go 6차원 평가 모델

| 차원 | 가중치(예시) | 1점 (Poor) | 3점 (Neutral) | 5점 (Excellent) |
|------|------------|-----------|-------------|----------------|
| 전략적 정합성 | 15% | ICP 불일치 | 부분 일치 | 완벽 일치, 시장 확대 |
| 승률 (Win Probability) | 25% | 강력한 수임업체 존재 | 동급 경쟁 | 수임업체이거나 관계 우수 |
| 역량/준수성 | 20% | 필수요건 미충족 | 대부분 충족 | 모든 요건 초과 충족 |
| 수익성 | 20% | 마진 미미 | 적정 마진 | 높은 수익성 |
| 자원 가용성 | 10% | 인력 부족 | 일부 충원 필요 | 즉시 투입 가능 |
| 리스크 | 10% | 다수 Red Flag | 일부 리스크 | 리스크 최소 |

**의사결정 임계값:**

| 가중 점수 | 결정 | 조치 |
|----------|------|------|
| 4.0~5.0 | **Strong Go** | 48시간 내 킥오프, 자원 투입 |
| 3.0~3.9 | **Conditional Go** | 특정 완화 조치(파트너십, 범위 협상) 후 진행 |
| 3.0 미만 | **No-Go** | 전문적 사양서(Decline Letter) 발송, 미래 관심 표명 |

### 7.2 기대가치(Expected Value) 모델

```
기대가치 = PWin × 계약금액 × 예상 이익률

예: PWin 40% × 10억원 × 이익률 15% = 6천만원

제안 비용이 3천만원이면 → 기대가치(6천만) > 제안비용(3천만) → Go
제안 비용이 8천만원이면 → 기대가치(6천만) < 제안비용(8천만) → No-Go
```

**실무 Rule of Thumb:**
- 제안 비용이 계약 가치의 5%를 초과하고 PWin이 30% 미만이면 기대 ROI 부정적

### 7.3 Go/No-Go 미팅 구조 (45분 이내)

| 시간 | 활동 |
|------|------|
| 5분 | 기회 담당자 요약 및 사전 권고 |
| 20분 | 기준별 점수 검토, 이견 조율, 점수 조정 |
| 5분 | 전체 점수와 무관한 탈락 요인(deal-killer) 확인 |
| 5분 | 최종 의사결정 문서화 |
| 5분 | 후속 조치 및 커뮤니케이션 계획 |

**참석자:** 기회 담당자, 제안 관리자, 경영진 스폰서, SME, 재무 담당, 법무/컴플라이언스

### 7.4 인지적 함정(Cognitive Traps) 회피

| 함정 | 대응 |
|------|------|
| 확증 편향 (Confirmation Bias) | 의도적으로 불참 논거를 만들어 볼 것 |
| 매몰비용 오류 (Sunk Cost Fallacy) | 과거 투자가 약한 제안서를 정당화하지 않음 |
| 정치적 압력 | 경영진 오버라이드 시 자원 트레이드오프 가시화 |
| 딜 열병 (Deal Fever) | 대형 딜의 흥분에 휩쓸리지 말고 기대가치 객관 계산 |

### 8.1 입찰 방식별 비교

#### (1) 적격심사제 (최저가 낙찰)

**적용 대상:** 중소규모 물품/용역 (추정가격 고시금액 미만)

**리스크:** 수익성 악화가 **매우 높은** 리스크. 원가 분석 철저히 하여 적정 마진 확보 필수.

#### (3) 종합심사제

**적용 대상:** 대규모 공사 (추정가격 300억원 이상)

시공능력, 시공실적, 재무상태, 기술능력 등을 종합 평가.

#### (4) 2단계 입찰

**적용 대상:** 기술 사양을 미리 확정하기 어려운 사업

1. 1단계: 기술 제안서 평가 (적격/부적격)
2. 2단계: 적격 업체만 가격 입찰 참여

| 적격심사 (최저가) | 수익성 악화 | 매우 높음 | 원가 분석 철저, 적정 마진 확보 |
| 제한경쟁 | 자격 미충족 탈락 | 치명적 | 사전 자격 검증, 컨소시엄 검토 |
| 2단계 입찰 | 기술 적격 탈락 | 높음 | 기술 적격 요건 완벽 충족 |
| 수의계약 | 가격 협상력 부족 | 보통 | 원가 근거 자료 준비 |

## 9. 사업성 검토 종합 프로세스 플로우

```
[1단계] 기회 식별
    ├── 공고 모니터링 (나라장터, K-스타트업 등)
    ├── 사전 필터링 (업종, 규모, 지역)
    └── 기회 등록

[2단계] 초기 스크리닝 (Gate 1 - Pursuit Decision)
    ├── 필수 자격요건 충족 여부 확인
    ├── 전략적 적합성 판단
    ├── No-Go 트리거 확인
    └── 추적 여부 결정

[3단계] 심층 분석 (Pre-Bid)
    ├── RFP 상세 분석 (평가기준, 배점, 숨겨진 요구사항)
    ├── 경쟁사 분석 (수임업체, 주요 경쟁사)
    ├── 기술 역량 Gap 분석
    ├── 수익성 시뮬레이션 (FP 산정, 원가 분석)
    ├── 리스크 5x5 매트릭스 작성
    └── PWin 산출

[4단계] Bid/No-Bid 결정 (Gate 2 - Bid Decision)
    ├── 7대 기준 가중 평가
    ├── Go/No-Go 미팅 (45분)
    ├── 기대가치(EV) 계산
    └── 최종 의사결정 및 문서화

[5단계] 제안 준비
    ├── Win Theme 수립
    ├── 목차 전략 (RFP 100% 매핑)
    ├── Color Team Review (Blue → Pink → Red → Gold)
    └── 제안서 작성/검토/수정

[6단계] 최종 검증 (Gate 3 - Bid Validation)
    ├── 준수성 최종 확인
    ├── 가격 적정성 확인
    └── 제출 승인

[7단계] 사후 관리
    ├── Win/Loss 분석
    ├── Lessons Learned (White Hat Review)
    └── PWin 모델 보정
```
