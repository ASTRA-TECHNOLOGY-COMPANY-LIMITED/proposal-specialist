# 반복 심층 분석 워크플로우 설계 (v2)

> 기존 단선 워크플로우(검색 → 1건 평가 → 종료)를 **반복 심층 분석 워크플로우**로 개선하는 설계 문서

---

## 1. 현행 문제 분석

### 1.1 현행 워크플로우 (AS-IS)

```
Step 1: 기업 문서 분석
Step 2: 사용자 키워드 확인
Step 3: 공고 검색
Step 4: 공고 1건 선택
Step 5: 선택 공고 평가
Step 6: 최종 보고서 → 종료
```

### 1.2 식별된 문제점

| # | 문제 | 영향 | 심각도 |
|---|------|------|--------|
| P1 | 입찰 마감일 필터링 없음 | 이미 마감된 공고가 결과에 포함되어 사용자가 불필요한 공고를 검토함 | 높음 |
| P2 | 검색 결과에서 자동 선별 로직 부재 | 적합도 기준만 존재하고 실제 정량 스코어링 지침 없음 | 높음 |
| P3 | 다운로드 파일 읽기 단계 누락 | 첨부파일을 다운로드하지만 Read Tool로 읽는 지침이 없음 | 높음 |
| P4 | 단일 공고 평가 후 종료 | 1건만 평가하고 워크플로우가 끝남. 추가 공고 평가 불가 | 중간 |
| P5 | 기업 프로필이 스코어링에 미반영 | bid-searcher가 기업 프로필 없이 키워드 매칭만 수행 | 중간 |

---

## 2. 목표 워크플로우 (TO-BE)

### 2.1 전체 흐름

```
Phase 1: 기업 분석
  Step 1: 기업 문서 분석 → 프로필 + 키워드 도출
  Step 2: 사용자 키워드 확인/보완

Phase 2: 공고 탐색 및 선별
  Step 3: 다중 플랫폼 병렬 검색
  Step 4: 마감일 필터링 (마감 지난 건 제외)
  Step 5: 적합도 스코어링 및 순위화

Phase 3: 심층 분석 (반복)
  Step 6: 사용자 공고 선택 (1건 이상)
  Step 7: 선택 공고 상세 조회
  Step 8: 첨부파일 다운로드
  Step 9: 다운로드 파일 읽기 및 분석
  Step 10: 적합성 평가 보고서 작성
  ──→ "추가 공고를 평가하시겠습니까?" ──→ Yes: Step 6으로
                                         ──→ No: Phase 4로

Phase 4: 종합 보고
  Step 11: 전체 평가 결과 비교 및 종합 보고서
```

### 2.2 AS-IS vs TO-BE 비교

| 구분 | AS-IS | TO-BE |
|------|-------|-------|
| 마감일 처리 | 없음 | 마감 지난 건 자동 제외 + D-day 표시 |
| 적합도 산정 | 기준만 나열 | 정량 스코어링 알고리즘 적용 |
| 공고 선택 | 1건만 | 1건 이상 + 반복 선택 |
| 첨부파일 분석 | 다운로드까지만 | 다운로드 → 읽기 → 분석 완전 연결 |
| 기업 프로필 활용 | 검색에만 사용 | 검색 + 스코어링 + 평가 전체에 활용 |
| 평가 반복 | 불가 (1회 종료) | 반복 심층 분석 루프 |

## 3. 상세 설계

### 3.1 입찰 마감일 필터링

입찰 마감일 검증은 **MCP 서버**와 **에이전트 프롬프트** 두 계층에서 수행한다.

#### 3.1.1 MCP 서버 계층 (g2b_search_bids)

`g2b_search_bids` Tool의 응답에 마감일 관련 정보를 추가한다.

**현행**: `bidClseDt` 필드를 그대로 반환

**개선**: 응답 항목에 다음 필드를 추가 계산하여 반환

| 필드 | 타입 | 설명 |
|------|------|------|
| `bidClseDt` | string | 입찰마감일시 (원본) |
| `isClosed` | boolean | 현재 시점 기준 마감 여부 |
| `daysRemaining` | number | 마감까지 남은 일수 (음수면 이미 마감) |
| `deadlineStatus` | string | `expired` / `urgent` (3일 이내) / `normal` / `upcoming` (2주 이상) |

**마감 판정 기준**:
- `bidClseDt`를 파싱하여 현재 시각(서버 로컬 타임)과 비교
- `bidClseDt`가 현재 시각보다 이전이면 `isClosed: true`
- `daysRemaining`: `Math.ceil((마감일시 - 현재시각) / (24 * 60 * 60 * 1000))`

**마감 건 자동 제외**:
- `g2b_search_bids`의 클라이언트측 필터링 단계에서 `isClosed === true`인 항목을 결과에서 제거
- 제거된 건수를 `expiredCount` 필드로 별도 반환하여 몇 건이 마감으로 제외되었는지 알림

**응답 구조 변경**:

```
{
  "totalCount": 15,          // 필터링 후 유효 건수
  "serverTotalCount": 23,    // API 원본 건수
  "expiredCount": 8,         // 마감으로 제외된 건수
  "currentPage": 1,
  "items": [
    {
      "bidNtceNo": "...",
      "bidNtceNm": "...",
      "bidClseDt": "2026-03-15 18:00",
      "isClosed": false,
      "daysRemaining": 27,
      "deadlineStatus": "upcoming",
      ...
    }
  ]
}
```

#### 3.1.2 지원사업 플랫폼 마감일 처리

각 플랫폼별로 마감일 필드명이 다르므로 플랫폼별로 처리한다.

| 플랫폼 | 마감일 필드 | 처리 방식 |
|--------|------------|-----------|
| 나라장터 입찰 | `bidClseDt` | MCP 서버에서 `isClosed` 계산 후 마감 건 제외 |
| K-스타트업 | `pbanc_rcpt_end_dt` | MCP 서버에서 `isClosed` 계산 후 마감 건 제외. 추가로 기존 `rcrt_prgs_yn === 'Y'` 필터 유지 |
| 중기부 | `applicationEndDate` | MCP 서버에서 `isClosed` 계산 후 마감 건 제외 |
| 과기부 | `pressDt` (게시일만 존재) | 마감일 필드 없음. 에이전트 프롬프트에서 게시일 기준 3개월 이내만 표시하도록 안내 |
| 사전규격 | 마감 개념 없음 (향후 예정) | 제외 불필요. 등록일 기준 최신순 정렬만 적용 |

#### 3.1.3 에이전트 프롬프트 계층 (bid-searcher)

MCP 서버가 마감 건을 제외하더라도, 에이전트 프롬프트에서 다음을 추가로 명시한다.

**bid-searcher.md에 추가할 규칙**:

1. **마감 확인 필수**: 검색 결과의 `isClosed` 필드를 반드시 확인하고, `true`인 공고는 결과 목록에서 제외하라
2. **D-day 표시**: 마감까지 남은 일수를 `D-{n}` 형식으로 표시하라
3. **마감 임박 경고**: `deadlineStatus`가 `urgent` (3일 이내)인 공고는 `[긴급]` 태그를 붙이고 "준비 기간이 부족할 수 있습니다"를 안내하라
4. **최소 준비 기간 기준**: 마감까지 3일 미만인 공고는 적합도를 자동으로 `낮음`으로 하향하라. 단, 사용자가 명시적으로 요청하면 포함 가능

#### 3.1.4 evaluate 커맨드 계층

개별 공고 평가 시에도 마감일을 재확인한다.

**evaluate.md에 추가할 규칙**:

1. `g2b_get_bid_detail`로 상세 조회 후 `bidClseDt`를 확인
2. 이미 마감된 공고인 경우 사용자에게 즉시 경고:
   "이 공고는 {마감일}에 마감되었습니다. 그래도 분석을 진행하시겠습니까?"
3. 사용자가 확인 후에만 분석 진행 (참고용 분석)

### 3.2 적합도 스코어링 알고리즘

현재 `bid-searcher.md`에 가중치 기준표만 존재하고 구체적 산정 방법이 없다. 에이전트가 일관되게 점수를 부여할 수 있도록 정량적 스코어링 지침을 추가한다.

#### 3.2.1 스코어링 입력 요건

스코어링을 수행하려면 다음 두 가지 입력이 필요하다.

| 입력 | 출처 | 필수 여부 |
|------|------|-----------|
| 기업 프로필 | doc-analyzer 에이전트 출력 | 선택 (없으면 키워드 매칭만 수행) |
| 검색 결과 목록 | MCP Tool 응답 | 필수 |

**기업 프로필 전달 방식**: `strategy.md`의 Step 3에서 bid-searcher 에이전트를 호출할 때, Step 1에서 생성된 기업 프로필 요약을 컨텍스트로 전달하도록 프롬프트를 수정한다.

#### 3.2.2 5항목 스코어링 기준

각 항목은 0~100점 스케일로 산정한 후, 가중치를 적용하여 총점을 계산한다.

**항목 1: 키워드 매칭도 (가중치 40%)**

| 조건 | 점수 |
|------|------|
| 공고명에 핵심 키워드 2개 이상 포함 | 100 |
| 공고명에 핵심 키워드 1개 포함 | 70 |
| 공고명에 관련 키워드(확장) 포함 | 40 |
| 키워드 매칭 없음 (분류만 일치) | 10 |

**항목 2: 사업 규모 적합성 (가중치 20%)**

| 조건 | 점수 |
|------|------|
| 추정가격이 기업 평균 수주액의 0.5~2배 | 100 |
| 추정가격이 기업 평균 수주액의 0.3~0.5배 또는 2~3배 | 60 |
| 추정가격 정보 없음 | 50 |
| 추정가격이 기업 규모 대비 과소/과대 (0.3배 미만 또는 3배 초과) | 20 |
| 기업 프로필 없음 (비교 불가) | 50 |

**항목 3: 마감일 여유 (가중치 15%)**

| 조건 | 점수 |
|------|------|
| 마감까지 3주 이상 | 100 |
| 마감까지 2~3주 | 80 |
| 마감까지 1~2주 | 50 |
| 마감까지 3일~1주 | 20 |
| 마감까지 3일 미만 | 0 |
| 마감일 정보 없음 (지원사업 등) | 70 |

**항목 4: 사업 유형 적합성 (가중치 15%)**

| 조건 | 점수 |
|------|------|
| 기업 주력 분야와 사업 분류 완전 일치 | 100 |
| 기업 보유 역량으로 수행 가능 | 60 |
| 일부 역량만 보유 (컨소시엄 필요 가능) | 30 |
| 기업 역량과 무관한 사업 | 0 |
| 기업 프로필 없음 (판단 불가) | 50 |

**항목 5: 발주기관 특성 (가중치 10%)**

| 조건 | 점수 |
|------|------|
| 기존 수행 실적이 있는 기관 | 100 |
| 유사 기관 (동일 부처 산하기관 등) | 60 |
| 실적 없는 기관 | 30 |
| 기업 프로필 없음 (판단 불가) | 30 |

**총점 계산**: `총점 = (항목1 × 0.4) + (항목2 × 0.2) + (항목3 × 0.15) + (항목4 × 0.15) + (항목5 × 0.1)`

**등급 분류**:

| 총점 범위 | 적합도 등급 | 표시 |
|-----------|-----------|------|
| 70점 이상 | 높음 | `[높음]` |
| 40~69점 | 중간 | `[중간]` |
| 40점 미만 | 낮음 | `[낮음]` |

#### 3.2.3 결과 정렬 규칙

1. 적합도 등급 내림차순 (높음 → 중간 → 낮음)
2. 동일 등급 내에서는 총점 내림차순
3. 동일 점수면 마감일 임박순

### 3.3 첨부파일 분석 프로세스

현재 `evaluate.md`에서 "첨부파일을 다운로드한다" 다음에 바로 "rfp-evaluator 에이전트를 호출"으로 넘어가며, 다운로드된 파일을 실제로 읽는 단계가 명시되어 있지 않다.

#### 3.3.1 개선된 평가 절차 (evaluate.md)

```
1. MCP Tool로 공고 상세 정보를 조회한다 (g2b_get_bid_detail 등)
2. 입찰 마감일을 확인한다 → 이미 마감이면 사용자에게 경고
3. 첨부파일 URL을 확인하고 다운로드한다 (download_attachment Tool)
4. 다운로드된 파일 경로를 확인한다 (list_downloads Tool)
5. Read Tool로 다운로드된 각 파일의 내용을 읽는다
   - PDF: Read Tool (pages 파라미터로 분할 읽기, 20페이지 단위)
   - 텍스트/CSV: Read Tool (직접 읽기)
   - HWP/DOCX: Read Tool 시도. 읽기 불가 시 사용자에게 안내
6. rfp-evaluator 에이전트를 호출하여 상세 분석을 위임한다
   - 공고 상세 정보 + 첨부파일 내용을 함께 전달
7. 평가 보고서를 작성한다
```

#### 3.3.2 rfp-evaluator에 파일 읽기 지침 추가

**rfp-evaluator.md의 "2. 첨부파일 분석" 섹션 개선**:

현행:
```
### 2. 첨부파일 분석
다운로드된 제안요청서/규격서에서 추출:
```

개선:
```
### 2. 첨부파일 분석

다운로드된 파일을 분석하라. 파일 경로가 제공되지 않은 경우
list_downloads Tool로 다운로드된 파일 목록을 확인하라.

#### 파일 읽기 절차
1. list_downloads Tool로 다운로드된 파일 경로를 확인한다
2. 각 파일에 대해 Read Tool로 내용을 읽는다
   - PDF 파일: pages 파라미터를 사용하여 20페이지 단위로 읽는다
     (예: pages: "1-20", 이후 필요시 "21-40")
   - 텍스트 파일: 직접 Read Tool로 읽는다
   - HWP 파일: Read Tool로 시도하되, 읽기 불가 시
     "HWP 파일은 직접 분석이 어렵습니다. PDF로 변환 후 다시 시도해주세요"를 안내한다
3. 읽은 내용에서 다음을 추출한다:
   - 상세 요구사항 목록
   - 기술 평가 항목 및 배점
   - 필수/선택 요구 기능
   - 납품/산출물 목록
   - 투입 인력 요건
```

### 3.4 반복 심층 분석 루프

현재 `strategy.md`는 Step 4에서 1건만 선택하고 Step 6에서 종료된다. 이를 다중 선택 + 반복 구조로 개선한다.

#### 3.4.1 strategy.md 개선 구조

```
Phase 1: 기업 분석
  Step 1: 기업 문서 분석 (doc-analyzer)
  Step 2: 사용자 키워드 확인

Phase 2: 공고 탐색
  Step 3: 병렬 검색 (bid-searcher)
    - 기업 프로필을 bid-searcher에 전달
    - 마감된 공고 자동 제외
    - 적합도 스코어링 적용

Phase 3: 반복 심층 분석
  Step 4: 공고 선택
    - 적합도 순으로 정렬된 결과 제시
    - 사용자가 상세 분석할 공고를 1건 이상 선택
    - "상세 분석할 공고 번호를 선택하세요 (쉼표로 구분하여 여러 건 선택 가능)"

  Step 5: 선택된 각 공고에 대해 순차 실행
    5-1. g2b_get_bid_detail로 상세 조회
    5-2. 마감일 재확인 (마감 시 경고)
    5-3. download_attachment로 첨부파일 다운로드
    5-4. list_downloads로 다운로드된 파일 확인
    5-5. Read Tool로 첨부파일 내용 읽기
    5-6. rfp-evaluator로 적합성 평가
    5-7. 평가 보고서 출력

  Step 6: 추가 평가 확인
    "다른 공고도 추가로 평가하시겠습니까?"
    - Yes → Step 4로 복귀 (이미 평가한 공고는 [평가완료] 표시)
    - No → Phase 4로 진행

Phase 4: 종합 보고
  Step 7: 전체 분석 결과 종합
    - 기업 프로필 요약
    - 평가된 모든 공고의 비교표
    - 최종 추천 순위 (평가 점수 기반)
    - 공고별 제안 방향 요약
    - 종합 준비 체크리스트
```

### 3.5 기업 프로필 컨텍스트 전달

#### 3.5.1 문제

현재 bid-searcher 에이전트는 키워드만 받아서 검색하고, 기업 프로필 정보를 참조하지 않는다. 따라서 "사업 규모 적합성", "사업 유형 적합성", "발주기관 특성" 항목의 스코어링이 불가능하다.

#### 3.5.2 해결 방안

`strategy.md`의 Step 3에서 bid-searcher를 호출할 때 기업 프로필 요약을 함께 전달하도록 프롬프트를 개선한다.

```
확정된 키워드로 bid-searcher 에이전트를 통해 병렬 검색하라.

bid-searcher에게 다음 컨텍스트를 반드시 전달하라:
- 검색 키워드 목록 (우선순위 포함)
- 기업 프로필 요약:
  - 업종, 핵심 분야
  - 매출 규모 / 평균 수주 금액
  - 주요 인증/자격

  - 보유 기술 스택
```

**bid-searcher.md에 추가할 지침**:

```
## 스코어링 시 기업 프로필 활용

기업 프로필이 제공된 경우 5항목 전체를 스코어링하라.
기업 프로필이 없는 경우 키워드 매칭도(40%)와 마감일 여유(15%)만으로
간이 스코어링을 수행하고, 나머지 항목은 기본 50점을 부여하라.
```

## 5. MCP 서버 변경 상세

### 5.1 date-utils.ts 추가 함수

마감일 상태를 계산하는 유틸리티 함수를 추가한다.

**추가할 함수 목록**:

| 함수명 | 입력 | 출력 | 용도 |
|--------|------|------|------|
| `calcDeadlineInfo` | 마감일시 문자열 | `{ isClosed, daysRemaining, deadlineStatus }` | 마감 상태 계산 |

**`calcDeadlineInfo` 동작 규칙**:

| 입력 | `isClosed` | `daysRemaining` | `deadlineStatus` |
|------|------------|-----------------|-------------------|
| 마감일시 < 현재 | `true` | 음수 | `expired` |
| 0 ≤ 남은 일수 ≤ 3 | `false` | 0~3 | `urgent` |
| 3 < 남은 일수 < 14 | `false` | 4~13 | `normal` |
| 남은 일수 ≥ 14 | `false` | 14+ | `upcoming` |
| 마감일시 없음/파싱 실패 | `false` | `-1` | `unknown` |

**마감일시 파싱**: 나라장터 `bidClseDt` 필드는 `"2026/02/28 18:00:00"` 형식이다. `Date` 생성자로 파싱하되, 파싱 실패 시 `unknown`을 반환한다.

### 5.2 g2b.ts 변경

**`g2b_search_bids` Tool 변경사항**:

1. 응답 매핑 단계에서 각 항목에 `calcDeadlineInfo(item.bidClseDt)` 호출
2. `isClosed === true`인 항목을 `items` 배열에서 제거
3. 제거된 건수를 `expiredCount`로 반환
4. 기존 `summary` 매핑에 `isClosed`, `daysRemaining`, `deadlineStatus` 필드 추가

### 5.3 kstartup.ts 변경

**`kstartup_search_announcements` Tool 변경사항**:
