## 1. 기능점수란 무엇인가

기능점수(Function Point, FP)는 소프트웨어의 기능적 규모를 **사용자 관점**에서 측정하는 국제 표준 방법론이다.

- **개발자**: 1979년 IBM Allan Albrecht
- **관리기관**: IFPUG (International Function Point Users Group)
- **국제표준**: ISO/IEC 20926
- **한국 법적근거**: 「소프트웨어사업 대가의 기준」 (과학기술정보통신부 고시)

### 핵심 원칙

> "소프트웨어가 **어떻게** 구현되었는지(공급자 관점)"가 아니라, "사용자가 **어떤 기능**을 요구했는지(수요자 관점)"에서 측정한다.

- 개발 언어, 기술 스택, 아키텍처와 **무관**
- 구현 방법이 아닌 **기능의 양**을 측정
- 프로젝트 간 **객관적 비교** 가능

### FP 측정 유형 3가지

| 측정 유형 | 대상 | 설명 |
|---------|-----|------|
| **개발 프로젝트 FP** | 신규 개발 기능 전체 | 프로젝트에서 새로 개발되는 기능의 규모 |
| **개선(향상) 프로젝트 FP** | 기존 시스템 변경/추가 | 추가(Add) + 변경(Change) + 삭제(Delete) 기능 반영 |
| **어플리케이션 FP** | 현행 운영 시스템 전체 | 특정 시점의 현행 애플리케이션 총 기능 규모 |

## 2. 5가지 기능유형 (IFPUG 표준)

### 2.1 데이터 기능 (Data Functions)

| 유형 | 한국어 | 정의 | 예시 |
|------|--------|------|------|
| **ILF** | 내부논리파일 | 측정 대상 애플리케이션 경계 내에서 유지(추가/수정/삭제)되는 논리적 데이터 그룹 | 회원정보 테이블, 주문 테이블, 게시판 테이블 |
| **EIF** | 외부연계파일 | 다른 애플리케이션이 유지하며, 측정 대상에서는 참조(조회)만 하는 외부 데이터 그룹 | 공공 행정코드 DB, 외부 결제사 정보, 우편번호 DB |

**ILF vs EIF 구분 원칙:**

- 해당 애플리케이션에서 데이터를 **추가/수정/삭제** → **ILF**
- 외부 시스템이 관리하고 우리 시스템은 **조회만** → **EIF**
- 같은 데이터라도 **시스템 경계**에 따라 분류가 달라짐

### 2.2 트랜잭션 기능 (Transaction Functions)

| 유형 | 한국어 | 정의 | 예시 |
|------|--------|------|------|
| **EI** | 외부입력 | 데이터를 입력받아 ILF를 생성/수정/삭제하는 기능 | 사용자 등록, 게시글 작성/수정/삭제, 파일 업로드 |
| **EO** | 외부출력 | 데이터를 **가공·계산**하여 외부로 전달하는 기능 (단순 조회 이상의 처리 로직 포함) | 통계 보고서, 합계/평균 계산 결과, 이메일 발송, PDF 생성 |
| **EQ** | 외부조회 | 데이터를 가공 없이 **그대로 조회**하여 출력하는 기능 | 게시판 목록, 글 상세보기, 회원정보 조회, 코드 조회 |

**EO vs EQ 구분 핵심:**

| 판별 기준 | EO (외부출력) | EQ (외부조회) |
|---------|-------------|-------------|
| 계산/집계 | 있음 (합계, 평균, 통계 등) | 없음 |
| 가공/변환 | 있음 (포맷 변환, 데이터 결합) | 없음 (DB에서 읽어 그대로 표시) |
| 파생 데이터 생성 | 있음 | 없음 |
| 예시 | 기간별 매출합계, 대시보드 차트 | 검색 필터 있는 목록, 상세보기 |

### 2.3 측정 요소 정의

| 요소 | 영문명 | 적용 대상 | 정의 |
|------|--------|---------|------|
| **DET** | Data Element Type | ILF/EIF/EI/EO/EQ | 사용자가 식별 가능한 유일하고 비반복적인 데이터 필드 |
| **RET** | Record Element Type | ILF/EIF | ILF/EIF 내 사용자가 식별 가능한 데이터 서브그룹 (논리적 레코드 유형) |
| **FTR** | File Type Referenced | EI/EO/EQ | 트랜잭션이 참조하거나 유지하는 ILF 또는 EIF의 수 |

**DET 계산 예시:** 회원 테이블에 이름, 이메일, 전화번호, 주소, 생년월일 → DET = 5

| **2~5** | 낮음 | 보통 | 높음 |
| **6+** | 보통 | 높음 | 높음 |

**트랜잭션 기능 복잡도 (EI):**

| FTR \ DET | 1~4 | 5~15 | 16+ |
|-----------|-----|------|-----|
| **0~1** | 낮음 | 낮음 | 보통 |
| **2** | 낮음 | 보통 | 높음 |
| **3+** | 보통 | 높음 | 높음 |

**트랜잭션 기능 복잡도 (EO/EQ):**

| FTR \ DET | 1~5 | 6~19 | 20+ |
|-----------|-----|------|-----|
| **0~1** | 낮음 | 낮음 | 보통 |
| **2~3** | 낮음 | 보통 | 높음 |
| **4+** | 보통 | 높음 | 높음 |

## 4. 간이법 vs 정규법

| 구분 | 간이법 (Simplified) | 정규법 (Standard/IFPUG) |
|------|--------------------|-----------------------|
| **적용 시점** | 기획, 발주, ISP 단계 | 설계 완료 이후 |
| **복잡도 적용** | 유형별 평균 가중치 고정 | DET/RET/FTR 실측 후 개별 판별 |
| **VAF 적용** | 미적용 (1.0 고정) | 14개 GSC 평가 후 VAF 계산 |
| **정확도** | ±25% 범위 | ±10% 이내 |
| **소요 시간** | 수 시간 ~ 1일 | 수 일 ~ 수 주 |
| **필요 정보** | 기능 목록 수준 | 상세 설계서, ERD, 요구사항 명세 |
| **주요 용도** | 예산 추정, 발주 규모 산정 | 실계약 대가 산정, 검수 기준 |

**선택 기준:**

- 100 FP 미만 소형 사업 → 간이법으로도 충분
- 1억 원 이상 공공 SW 사업 → 정규법 권장
- RFP 단계 예산 산출 → 간이법 필수
- 제안서 원가 산정 → 간이법 산정 후 주요 고복잡도 기능만 정규법 검증

## 5. FP 산정 절차 (6단계)

```
[1단계] 측정 유형 결정
        → 개발/개선/어플리케이션 중 선택
        → 측정 목적 확인 (예산산정 vs 계약 vs 생산성 측정)

[2단계] 측정 범위 및 경계 식별
        → 애플리케이션 경계(Application Boundary) 정의
        → 어떤 시스템까지 포함할지 결정 (ILF vs EIF 구분의 기준)

[3단계] 데이터 기능 식별 및 측정
        → ILF/EIF 식별
        → 각 파일의 DET, RET 수 계산
        → 복잡도(낮음/보통/높음) 판별 → 가중치 적용

[4단계] 트랜잭션 기능 식별 및 측정
        → EI/EO/EQ 식별 (단위프로세스 단위)
        → 각 트랜잭션의 DET, FTR 수 계산
        → 복잡도 판별 → 가중치 적용

[5단계] UFP(미조정기능점수) 계산
        → UFP = Σ(기능유형별 기능수 × 가중치)

[6단계] VAF 계산 및 AFP 산출 (정규법만 해당)
        → 14개 GSC 평가 → TDI 합산
        → VAF = 0.65 + 0.01 × TDI
        → AFP(조정기능점수) = UFP × VAF
```

## 6. 14가지 일반시스템특성 (GSC)

정규법에서 기술적 복잡도를 반영하기 위해 사용. 각 항목 0~5점 평가.

| 번호 | 특성 | 설명 |
|------|-----|------|
| 1 | 데이터 통신 | 원격 통신을 통한 데이터 전송/수신 정도 |
| 2 | 분산 기능 | 분산된 데이터/처리 기능의 정도 |
| 3 | 성능 | 응답시간/처리율 요구사항 |
| 4 | 중장비 구성 | 특수 하드웨어 운영 환경 |
| 5 | 거래율 | 최고 트랜잭션 처리 비율 |
| 6 | 온라인 데이터 입력 | 인터랙티브 온라인 입력 비율 |
| 7 | 최종 사용자 효율성 | 사용자 효율성 설계 |
| 8 | 온라인 업데이트 | 온라인으로 ILF 직접 갱신 |
| 9 | 복잡한 처리 | 복잡한 내부 처리 로직 |
| 10 | 재사용성 | 재사용 가능한 코드 설계 |
| 11 | 설치 용이성 | 설치 전환 작업의 복잡도 |
| 12 | 운영 용이성 | 운영 백업/회복 절차 |
| 13 | 다중 사이트 | 다중 사이트 설치/운영 |
| 14 | 변경 용이성 | 사용자 변경에 따른 변경 용이성 |

**계산:**
- TDI(총영향도) = Σ(14개 항목 점수), 범위 **0~70**
- VAF(가치조정계수) = 0.65 + (0.01 × TDI), 범위 **0.65~1.35**

## 7. SW 대가산정 (금액 산정)

### 7.1 법적 근거

- **「소프트웨어사업 대가의 기준」** (과학기술정보통신부 고시)
- **「SW사업 대가산정 가이드」** (한국소프트웨어산업협회 공표, 매년 개정)
- 공공기관 SW 발주 시 의무 준용

### 7.2 FP당 단가 변천

| 연도 | FP당 단가(원) | 인상률 |
|------|-------------|-------|
| 2014 | 519,203 | +4.4% |
| 2020 | 553,114 | +6.5% |
| 2022 | 553,114 | - |
| **2024** | **605,784** | **+9.52%** |
| **2025** | **605,784 (적용 중)** | - |

> **2024~2025년 기준 FP당 단가: 605,784원**

### 7.3 개발비 산정 공식 (7단계)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[1단계] 미조정 개발원가
        = 총 기능점수(FP) × FP당 단가(605,784원)

[2단계] 보정 후 개발원가
        = 미조정 개발원가
          × 규모 보정계수
          × 연계복잡성 보정계수
          × 성능요구 보정계수
          × 운영환경 호환성 보정계수
          × 보안성 보정계수

[3단계] 직접경비
        = 전문가 자문료 + SW 라이선스 + 외주비 등
          (증빙 가능한 실비 항목)

[4단계] SW 개발비
        = 보정 후 개발원가 + 직접경비

[5단계] 이윤
        = SW 개발비 × 이윤율 (최대 25% 이내)

[6단계] 개발 대가
        = SW 개발비 + 이윤

[7단계] 최종 대가
        = 개발 대가 × 1.1 (부가가치세 10%)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 7.4 계산 예시 ① — 소규모 사업 (간이법)

**FP 산정:**

| 기능유형 | 기능수 | 가중치 | FP |
|---------|-------|-------|-----|
| ILF | 5 | 7.5 | 37.5 |
| EIF | 2 | 5.4 | 10.8 |
| EI | 12 | 4.0 | 48.0 |
| EO | 5 | 5.2 | 26.0 |
| EQ | 15 | 3.9 | 58.5 |
| **합계** | **39** | | **180.8 FP** |

**금액 산정:**

```
[1] 미조정 개발원가: 180.8 FP × 605,784원 = 109,525,747원

[2] 보정계수 적용:
    - 규모 보정 (500FP 미만): 0.65
    - 연계복잡성 (1개 기관): 1.00
    - 성능요구 (일반): 1.00
    - 운영환경 (표준 웹): 1.00
    - 보안성 (기본): 1.00

    보정 후 개발원가: 109,525,747 × 0.65 = 71,191,736원

[3] 직접경비: 5,000,000원 (서버 라이선스 등)

[4] SW 개발비: 71,191,736 + 5,000,000 = 76,191,736원

[5] 이윤 (15%): 76,191,736 × 0.15 = 11,428,760원

[6] 개발 대가: 76,191,736 + 11,428,760 = 87,620,496원

[7] 최종 대가 (VAT 포함): 87,620,496 × 1.1 = 96,382,546원
                          ≒ 약 9,640만 원
```

| 기능유형 | 기능수 | 가중치 | FP |
|---------|-------|-------|-----|
| ILF | 10 | 7.5 | 75.0 |
| EIF | 4 | 5.4 | 21.6 |
| EI | 25 | 4.0 | 100.0 |
| EO | 12 | 5.2 | 62.4 |
| EQ | 35 | 3.9 | 136.5 |
| **합계** | **86** | | **395.5 FP** |

**금액 산정:**

```
[1] 미조정 개발원가: 395.5 × 605,784 = 239,587,492원

[2] 보정계수 적용:
    - 규모 보정 (300~499FP): 0.97
    - 연계복잡성 (3개 기관 연계): 1.03
    - 성능요구 (보통): 1.00
    - 운영환경 (표준 웹): 1.00
    - 보안성 (기본): 1.00

    보정 후 개발원가: 239,587,492 × 0.97 × 1.03 ≒ 239,394,000원

[3] 직접경비: 15,000,000원

[4] SW 개발비: 239,394,000 + 15,000,000 = 254,394,000원

[5] 이윤 (10%): 254,394,000 × 0.10 = 25,439,400원

[6] 개발 대가: 254,394,000 + 25,439,400 = 279,833,400원

[7] 최종 대가 (VAT 포함): 279,833,400 × 1.1 = 307,816,740원
                          ≒ 약 3.08억 원
```

### 7.6 개선(유지보수) 사업 대가산정

기존 시스템 개선 사업은 변경 유형별로 차등 적용한다.

```
개선 프로젝트 FP = 추가 기능 FP + (변경 기능 FP × 변경률) + 삭제 기능 FP × 0.4

개선 개발원가 = 개선 FP × FP당 단가 × 보정계수
```

| 변경 유형 | 산정 방법 |
|---------|---------|
| **추가(Add)** | 신규 기능과 동일하게 풀 FP 산정 |
| **변경(Change)** | 변경 전 FP + 변경 후 FP 중 큰 값에 변경률 적용 |
| **삭제(Delete)** | 삭제 기능 FP × 0.4 (40% 반영) |

## 9. 흔한 실수와 주의사항

### 9.1 기능 과소 산정 패턴

| 실수 유형 | 설명 | 올바른 방법 |
|---------|------|----------|
| **업무모듈 묶음** | 등록/수정/삭제를 하나로 봄 | CUD 각각이 독립 단위프로세스면 별개 EI |
| **EQ 누락** | 단순 조회를 기능으로 인식 안 함 | 목록, 상세, 검색 모두 EQ로 산정 |
| **ILF 누락** | 로그, 이력, 설정 테이블 누락 | 관리성 데이터 파일도 ILF |
| **EIF 미식별** | 외부 연계 데이터 EIF 미계상 | 행정코드, 우편번호 등 EIF |
| **보조 기능 누락** | 로그인/로그아웃, 권한관리 누락 | 공통 기능도 반드시 포함 |
| **엑셀/파일 출력 누락** | 목록의 엑셀 다운로드 미계상 | EO로 별도 산정 |

### 9.2 기능 과대 산정 패턴

| 실수 유형 | 설명 | 올바른 방법 |
|---------|------|----------|
| **화면 단위 계산** | 화면별 EI/EO/EQ 별개 산정 | 단위프로세스(비즈니스 논리) 기준 |
| **동일 EQ 중복** | 같은 조회를 탭/메뉴별 별개 산정 | 동일 데이터 조회는 1개 EQ |
| **배치 과대 계상** | 내부 처리를 EO로 산정 | 사용자에게 직접 제공 안 되면 제외 |
| **기술적 기능 포함** | API 에러, DB 연결을 FP로 계상 | 사용자 기능이 아닌 기술 요소 제외 |

### 9.3 경계(Boundary) 설정 오류

- 경계를 **너무 좁게** → EIF가 ILF로 잘못 분류
- 경계를 **너무 넓게** → ILF가 EIF로 잘못 분류
- 연계 시스템을 경계 내로 포함 → 양쪽 기능이 **중복 산정**

### 9.4 단위프로세스 식별 오류

> 단위프로세스는 "사용자에게 의미 있는 **최소 완결 활동**"이어야 한다.

| 기준 | 결과 |
|------|-----|
| 너무 크게 (업무모듈 단위) | → 과소 산정 |
| 너무 작게 (화면 컴포넌트 단위) | → 과대 산정 |
| **적정 기준** | 해당 기능만으로도 비즈니스 가치가 완성되는가? |

### 9.5 금액 산정 시 주의사항

| 주의사항 | 설명 |
|---------|------|
| **FP당 단가 연도 확인** | 반드시 해당 연도 고시 단가 적용 (2024~2025: 605,784원) |

## 10. 전문가 실무 노하우

### 10.1 FP 산정 전략 (제안사 입장)

1. **RFP 기능목록 기준 식별**: RFP의 화면정의서, 기능요구사항 목록에서 EI/EO/EQ를 도출하면 가장 일관성 있음
2. **유사 시스템 벤치마크**: 비슷한 규모의 기완료 사업 FP와 비교하여 합리성 검증
3. **간이법 → 정규법 검증**: 간이법으로 전체 FP 산정 후, 고복잡도 기능만 정규법으로 크로스체크
4. **FP-MM 연계**: 1FP ≈ 0.3~0.5 MM 기준을 내부적으로 갖추면 인력 투입 계획과 정합성 확인 가능
5. **화면수 × 평균 FP 방식**: 급할 때는 유사 프로젝트의 화면당 평균 FP(보통 5~15 FP/화면)로 Ballpark Estimate

### 10.2 발주기관 대응 전략

- FP 산정 근거를 **기능목록 형태**로 상세 제시 (기능ID, 기능명, 유형, 가중치, FP)
- 발주기관이 RFP에 제시한 FP와 차이가 있으면 **근거를 들어 협의**
- 검수 단계에서 완성 기능점수를 **재산정**하여 변경사항 반영

### 10.3 금액 협상 노하우

| 전략 | 상세 |
|------|-----|
| **기능 누락 식별** | RFP에 명시되지 않았지만 필수적인 기능(인증, 로그, 관리자 등) 추가 제안 |
| **복잡도 상향** | 연계 기관이 많거나 보안 요건이 높으면 보정계수로 금액 상향 가능 |
| **직접경비 활용** | 클라우드 비용, 라이선스비 등을 직접경비로 별도 산정 |
| **이윤율 적정화** | 사업 규모/난이도에 맞는 이윤율 설정 (일반 10%, 난이도 높으면 15~20%) |
| **유지보수 연계** | 개발 후 유지보수 사업 연계 시 전체 사업 규모 확대 |

### 10.4 재사용 소프트웨어 처리

| 유형 | FP 적용 |
|------|--------|
| 신규개발 | 풀 가중치 100% 적용 |
| 기능 변경 (수정 후 재사용) | 변경률에 영향계수를 곱하여 부분 적용 |
| 수정 없이 재사용 | **0 FP** (산정 대상 제외) |

### 10.5 Sanity Check 기법

FP 산정 후 반드시 아래 기준으로 합리성을 검증한다.

| 검증 항목 | 기준값 | 이상 판단 |
|---------|-------|---------|
| 화면당 평균 FP | 5~15 FP | 3 미만 또는 20 초과 시 재검토 |
| ILF : EI 비율 | 1 : 2~4 | ILF 대비 EI가 너무 적거나 많으면 재검토 |
| EQ 비율 | 전체의 30~50% | EQ가 없거나 극단적이면 재검토 |
| FP당 금액 | 60만 원 내외 | 보정 후 FP당 금액이 30만 원 미만이면 과소 산정 의심 |
| 유사사업 대비 | ±30% 이내 | 유사 규모 사업과 크게 차이나면 재검토 |

- **AI 도입사업 대가산정 가이드 신설**: AI/ML 솔루션 도입은 FP 대신 구독료+전문인력비 별도 산정
- **2025년 적용 엑셀 템플릿**: 한국소프트웨어산업협회 배포

### 12.2 AI/클라우드 환경 이슈

| 기술 영역 | 이슈 | 현행 처리 방안 |
|---------|-----|------------|
| **AI/ML 기능** | 모델 학습/추론의 FP 기준 불명확 | 별도 전문인력비로 산정 |
| **클라우드 네이티브** | 마이크로서비스별 경계 설정 | 서비스별 개별 FP 산정 |
| **API 연동** | REST API 제공 자체는 FP 대상 아님 | 연동 결과의 데이터 기능으로 EIF 산정 |
| **자동화 FP** | 생성형 AI 활용 FP 자동 산정 연구 | 도입 전 단계 (2024 ACM 논문 발표) |
| **DevOps/CI·CD** | 빌드/배포 파이프라인은 FP 대상 아님 | 직접경비 처리 |

### 12.3 국제 표준 동향

|------|-----|
| **IFPUG CPM 4.3.1** | 현행 최신 버전 유지 |
| **SFP 2.2** (Simple FP) | 더 간단한 대안 방법론 |
| **AFP** (Automated FP) | OMG 표준, 요구사항 문서에서 자동 산정 |
